Le [object Object] qui a cassé un rapport client

C'était un lundi matin, en réunion d'équipe. Un développeur d'un projet client avait passé le week-end à construire une fonctionnalité d'export de données. Le chef de produit a ouvert le fichier Excel, fait défiler jusqu'à la colonne adresse, et chaque cellule affichait [object Object]. La réunion a été interrompue. L'export a été retiré. Trois heures ont été nécessaires pour comprendre ce qui s'était passé.

Ce qui s'était passé était simple : la réponse de l'API était un tableau JSON où chaque enregistrement contenait un objet adresse imbriqué avec les sous-champs ville, code_postal et pays. Le script d'export avait fait ligne.adresse – et JavaScript avait silencieusement converti l'objet en sa représentation textuelle au lieu d'étendre les champs en colonnes séparées.

C'est le problème du JSON imbriqué. Il se produit chaque fois qu'un format de table plate (CSV, Excel, SQL) rencontre un format de données hiérarchique (JSON). Savoir aplatir correctement – et quand ne pas le faire – vous évitera cette conversation à chaque fois qu'une API modifie la forme de sa réponse.

Passez directement à l'outil : Si vous avez juste besoin d'aplatir du JSON maintenant, ouvrez le Convertisseur JSON vers CSV de DevToolbox, collez votre JSON, cochez Aplatir les objets et téléchargez. Revenez ici quand vous voudrez comprendre pourquoi un cas particulier se comporte de manière inattendue.

Ce que fait réellement l'aplatissement

L'aplatissement convertit une structure JSON hiérarchique en une seule table plate. Chaque objet imbriqué devient un ensemble de colonnes supplémentaires, avec les clés reliées par un séparateur (généralement un point). La profondeur de l'imbrication devient la longueur du nom de colonne, pas un niveau de structure supplémentaire.

❌ Avant – imbriqué (inutilisable dans Excel)
[{
  "nom": "Alice",
  "adresse": {
    "ville": "Paris",
    "code_postal": "75001"
  }
}]
✅ Après – plat (prêt pour Excel)
nom,adresse.ville,adresse.code_postal
Alice,Paris,75001

Le sous-objet adresse disparaît complètement. Ses champs deviennent de nouvelles colonnes avec des en-têtes en notation pointée. N'importe quel outil de tableur peut désormais filtrer, trier et croiser les données sans analyser le JSON.

L'algorithme en Python et JavaScript

L'aplatissement est une fonction récursive. Pour chaque clé de l'objet : si la valeur est un autre objet (pas un tableau), on plonge récursivement et on construit une clé composée en joignant la clé parente et la clé enfant avec le séparateur. Si la valeur est un type primitif (chaîne, nombre, booléen, null), on la stocke directement.

PythonImplémentation de base
Prêt à l'emploi 
def aplatir(obj, prefix='', sep='.'):
    elements = {}
    for cle, val in obj.items():
        nouvelle_cle = f"{prefix}{sep}{cle}" if prefix else cle
        if isinstance(val, dict):
            elements.update(aplatir(val, nouvelle_cle, sep))
        else:
            elements[nouvelle_cle] = val
    return elements

# Exemple
donnees = {"nom": "Alice", "adresse": {"ville": "Paris", "code_postal": "75001"}}
print(aplatir(donnees))
# → {'nom': 'Alice', 'adresse.ville': 'Paris', 'adresse.code_postal': '75001'}

# Appliquer à un tableau d'enregistrements
import csv, io
enregistrements = [{"nom": "Alice", "adresse": {"ville": "Paris", "code_postal": "75001"}},
           {"nom": "Bob",   "adresse": {"ville": "Lyon",  "code_postal": "69001"}}]

plats = [aplatir(r) for r in enregistrements]
cles = sorted({k for ligne in plats for k in ligne})

sortie = io.StringIO()
writer = csv.DictWriter(sortie, fieldnames=cles)
writer.writeheader()
writer.writerows(plats)
print(sortie.getvalue())
JavaScriptNavigateur et Node.js
Prêt à l'emploi 
function aplatir(obj, prefix = '', sep = '.') {
  return Object.keys(obj).reduce((acc, cle) => {
    const nouvelleCle = prefix ? `${prefix}${sep}${cle}` : cle;
    if (
      typeof obj[cle] === 'object' &&
      !Array.isArray(obj[cle]) &&
      obj[cle] !== null
    ) {
      Object.assign(acc, aplatir(obj[cle], nouvelleCle, sep));
    } else {
      acc[nouvelleCle] = obj[cle];
    }
    return acc;
  }, {});
}

// Convertir un tableau d'enregistrements en CSV
function versCSV(enregistrements, sep = '.') {
  const plats = enregistrements.map(r => aplatir(r, '', sep));
  const cles = [...new Set(plats.flatMap(Object.keys))].sort();
  const enTete = cles.join(',');
  const lignes = plats.map(r => cles.map(k => JSON.stringify(r[k] ?? '')).join(','));
  return [enTete, ...lignes].join('\n');
}

console.log(versCSV([
  { nom: 'Alice', adresse: { ville: 'Paris', code_postal: '75001' } },
  { nom: 'Bob',   adresse: { ville: 'Lyon',  code_postal: '69001' } }

Cas particuliers qui surprennent les développeurs

1Tableaux d'objets à l'intérieur d'un enregistrement
Attention

Lorsqu'un champ contient un tableau d'objets – "commandes": [{"id": 1, "total": 50}, {"id": 2, "total": 30}] – la simple fonction d'aplatissement n'a pas de colonne unique où les placer. Le convertisseur DevToolbox développe chaque élément du tableau sur sa propre ligne, en dupliquant tous les champs parents. C'est généralement ce que vous voulez pour une analyse Excel, mais cela produit des lignes répétées qui peuvent perturber un import en base de données.

Résultat de l'expansion des lignes
nom,commandes.id,commandes.total
Alice,1,50
Alice,2,30    ← Alice apparaît deux fois

Si vous préférez conserver le tableau dans une seule cellule, réglez l'option de traitement des tableaux sur Sérialiser en chaîne JSON. La cellule contiendra "[{""id"":1,""total"":50},{""id"":2,""total"":30}]".

2Points dans les noms de clés originaux
Attention

Si votre JSON a une clé comme "prenom.nom", l'aplatissement avec un séparateur point crée un nom de colonne ambigu, vous ne pouvez pas dire si prenom.nom était une clé originale ou un chemin aplati. Le convertisseur DevToolbox détecte cela et affiche un avertissement. Passez au séparateur __ ou nettoyez les clés avant la conversion.

Résultat ambigu – à éviter
// La clé originale contient un point
{ "prenom.nom": "Alice", "adresse": { "ville": "Paris" } }
// Aplati avec séparateur point → ambigu :
// prenom.nom  ← est-ce "prenom" → "nom" ou la clé originale ?
// adresse.ville
Utilisez le séparateur double underscore
// Sûr : aucune ambiguïté avec le séparateur __
aplatir(obj, '', '__')
// → { 'prenom.nom': 'Alice', 'adresse__ville': 'Paris' }
3Valeurs nulles et clés manquantes
Bon à savoir

Lorsque les enregistrements d'un tableau ont des clés différentes, un utilisateur a un champ telephone et un autre non, le CSV aplati doit avoir une colonne telephone qui est vide pour le deuxième utilisateur. Le convertisseur DevToolbox gère cela automatiquement, en créant une union de toutes les clés sur tous les enregistrements et en remplissant les cellules vides pour les valeurs manquantes.

[
  { "nom": "Alice", "telephone": "0612345678" },
  { "nom": "Bob" }   ← pas de champ telephone
]
// Sortie aplatie :
// nom,telephone
// Alice,0612345678
// Bob,              ← cellule vide, pas une erreur
Null vs absent : Un champ défini à null et un champ totalement absent deviennent tous les deux des cellules vides dans la sortie CSV. Si la distinction est importante pour votre pipeline, traitez-la avant l'aplatissement en remplaçant null par une valeur sentinelle.
4Imbrication profonde produisant des centaines de colonnes
Bon à savoir

Une réponse avec 8 niveaux d'imbrication peut produire des noms de colonnes comme donnees.utilisateurs.profil.parametres.notifications.email.resume.frequence. Le convertisseur vous permet de définir une profondeur d'aplatissement maximale. Tout ce qui est en dessous est sérialisé sous forme de chaîne JSON dans une seule cellule, ce qui garde la table exploitable.

aplatir(obj, '', '.', profondeurMax=3)
// Les champs à plus de 3 niveaux → sérialisés en chaîne JSON dans leur colonne parente

Étape 1Collez votre JSON

Ouvrez le Convertisseur JSON / CSV et collez votre JSON brut dans le panneau de gauche. L'outil accepte aussi bien un tableau racine [{...}, {...}] qu'une réponse d'API encapsulée {"data": [{...}]}.

La plupart des réponses d'API enveloppent les données réelles dans une clé :

{
  "statut": "ok",
  "data": {
    "utilisateurs": [ {...}, {...} ]
  }

Tapez data.utilisateurs dans le champ Chemin du tableau. Le convertisseur ignore l'objet enveloppe et traite uniquement les enregistrements dont vous avez besoin. Sans cela, il essaierait d'aplatir l'enveloppe elle-même et produirait une seule ligne avec un tableau sérialisé.

Erreur la plus fréquente : oublier de définir le chemin du tableau. Si votre sortie est une seule ligne avec une valeur du style [object Object], c'est la raison.
Étape 3Activez l'aplatissement des objets et prévisualisez

Cochez Aplatir les objets. L'aperçu du tableau en direct se met à jour en temps réel. Vous pouvez voir immédiatement comment les champs imbriqués correspondent aux noms de colonnes et repérer les éventuels cas particuliers avant de télécharger.

Étape 4Téléchargez avec BOM UTF-8 pour Excel

Cliquez sur Convertir en CSV → puis choisissez Télécharger CSV (UTF-8 BOM). Le BOM est un marqueur d'ordre des octets invisible au début du fichier qui indique à Excel de décoder le fichier en UTF-8. Sans lui, les accents français, les trémas allemands, les caractères arabes et tout texte non ASCII s'afficheront sous forme de symboles illisibles lorsqu'ils sont ouverts directement dans Excel sous Windows.

Pour les gros fichiers : pandas et jq

L'outil en ligne est conçu pour des conversions interactives rapides jusqu'à quelques mégaoctets. Pour les exports volumineux depuis un entrepôt de données, un dump complet de base de données ou un pipeline CI/CD, utilisez l'une de ces approches.

pandasPython, idéal pour les pipelines de données
Recommandé

pd.json_normalize est la fonction standard pour cela. Elle gère les objets imbriqués, les tableaux d'enregistrements, les séparateurs personnalisés et produit un DataFrame que vous pouvez transformer avant l'export.

import pandas as pd
import json

with open('reponse_api.json', encoding='utf-8') as f:
    donnees = json.load(f)

# Cas simple : la racine est un tableau
df = pd.json_normalize(donnees, sep='.')
df.to_csv('sortie.csv', index=False, encoding='utf-8-sig')

# Tableau encapsulé : les données sont dans data.utilisateurs
df = pd.json_normalize(donnees['data']['utilisateurs'], sep='.')
df.to_csv('sortie.csv', index=False, encoding='utf-8-sig')

# Tableau imbriqué dans chaque enregistrement (commandes par utilisateur)
# record_path : le tableau imbriqué à développer en lignes
# meta : les champs parents à conserver sur chaque ligne développée
df = pd.json_normalize(
    donnees['utilisateurs'],
    record_path='commandes',
    meta=['id', 'nom', 'email'],
    sep='.'
)
df.to_csv('utilisateurs_commandes.csv', index=False, encoding='utf-8-sig')
jqTerminal, idéal pour les pipelines shell
Rapide & scriptable

jq traite le JSON comme un flux, ce qui signifie qu'il peut gérer des fichiers beaucoup plus gros que la RAM disponible. C'est idéal pour les pipelines CI/CD, les tâches cron et tout flux de travail où installer Python n'est pas pratique.

# Extraire des champs spécifiques en CSV
jq -r '
  .data.utilisateurs[] |
  [.nom, .adresse.ville, .adresse.code_postal, .email] |
  @csv
' reponse_api.json > sortie.csv

# Ajouter une ligne d'en-tête
(
  echo '"nom","ville","code_postal","email"'
  jq -r '.data.utilisateurs[] | [.nom, .adresse.ville, .adresse.code_postal, .email] | @csv' entree.json
) > sortie.csv

# Gérer les valeurs nulles (remplacer par une chaîne vide)
jq -r '.data.utilisateurs[] | [(.nom // ""), (.adresse.ville // ""), (.email // "")] | @csv' entree.json
Quel outil utiliser : Utilisez le convertisseur en ligne pour des exports ponctuels et lorsque vous voulez un retour visuel immédiat. Utilisez pandas quand vous devez enchaîner des transformations dans un notebook Jupyter ou un script planifié. Utilisez jq dans les scripts shell et les pipelines CI/CD où la vitesse et le peu de dépendances comptent.

Foire aux questions

Pourquoi mon CSV aplati a-t-il des lignes répétées pour le même enregistrement ?

Votre JSON contient un champ de type tableau, par exemple, un utilisateur avec plusieurs commandes. L'algorithme d'aplatissement développe chaque élément du tableau sur sa propre ligne et répète tous les champs parents (nom, email, etc.) sur chaque ligne. C'est intentionnel : c'est la seule façon de représenter une relation un-à-plusieurs dans une table plate. Si vous voulez éviter la duplication, normalisez les données en deux fichiers CSV distincts, ou configurez le convertisseur pour sérialiser le tableau sous forme de chaîne JSON dans une seule cellule.

Aplatir ou normaliser, lequel choisir ?

L'aplatissement donne un seul CSV large, rapide, simple, idéal pour Excel et les outils de reporting. La normalisation donne plusieurs fichiers CSV liés par des clés étrangères – correct pour importer dans une base de données relationnelle sans redondance. Si la sortie va dans un tableur, un outil BI ou à un collègue non technique, aplatissez. Si la sortie va dans PostgreSQL, MySQL ou toute base SQL, normalisez.

Mes clés JSON contiennent des points. Que faire ?

Passez le séparateur de point à double underscore dans les paramètres du convertisseur. Avec le point comme séparateur, une clé nommée prenom.nom dans le JSON original produit le même nom de colonne qu'un chemin aplati prenom → nom. Cette ambiguïté casse tout outil en aval qui essaie d'interpréter les noms de colonnes comme des chemins. Le double underscore est sûr car il apparaît rarement dans les clés JSON originales.

Que deviennent les valeurs null dans la sortie ?

Les valeurs null deviennent des cellules vides dans le CSV, la représentation la plus propre pour les outils de tableur. Un champ défini à null et un champ totalement absent produisent tous les deux une cellule vide. Si votre système en aval exige la chaîne littérale "null" ou un zéro numérique, nettoyez les données avant la conversion ou ajustez le comportement des cellules vides dans les paramètres du convertisseur.

Comment gérer de très gros fichiers JSON ?

Pour les fichiers de plus de quelques mégaoctets, utilisez pd.json_normalize en Python pour des pipelines scriptables, ou jq pour le streaming depuis la ligne de commande. Les deux gèrent efficacement les gros fichiers et couvrent les mêmes cas d'usage que le convertisseur en ligne. Le code prêt à copier pour les deux se trouve dans l'article ci-dessus.

Mes données sont-elles envoyées à un serveur lorsque j'utilise le convertisseur ?

Non. Le convertisseur JSON vers CSV de DevToolbox s'exécute entièrement côté client en JavaScript. Vos données JSON sont traitées dans la mémoire de votre navigateur et ne sont jamais transmises à un serveur. Aucun compte n'est requis, aucun fichier n'est stocké et aucune analyse n'est collectée sur vos données d'entrée.
