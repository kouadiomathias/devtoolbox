Pourquoi deux outils de diff affichent des résultats différents pour le même changement

J'ai remarqué cela pour la première fois lors d'une revue de code. Un collègue m'a envoyé un diff de notre fichier de configuration. J'ai ouvert les deux mêmes fichiers dans mon outil de diff. Nous regardions le même changement, mais les diffs étaient différents — le mien montrait trois lignes modifiées, le sien cinq. Nous avions tous les deux raison.

Cela se produit parce que la comparaison de textes n'est pas une opération déterministe unique. Lorsque vous demandez « qu'est-ce qui a changé entre ces deux fichiers ? », la réponse dépend de l'algorithme utilisé et, lorsque plusieurs réponses valides existent, de la manière dont les égalités sont départagées. Deux fichiers peuvent avoir plusieurs sous-séquences communes les plus longues valides différentes — et en choisir différentes conduit à des diffs différents.

Comprendre pourquoi cela se produit fait de vous un bien meilleur utilisateur des outils de diff. Cela explique également certains comportements qui ressemblent à des bugs mais qui sont en réalité corrects : un changement attribué à une ligne différente de celle attendue, des lignes de contexte qui apparaissent ou disparaissent, ou deux diffs équivalents qui semblent complètement différents en tant que texte.

L'algorithme LCS — comment fonctionne réellement le diff

Le problème que résout le diff est de trouver la plus longue sous-séquence commune (LCS, pour Longest Common Subsequence) entre deux séquences de lignes. La LCS est la plus longue séquence de lignes qui apparaît dans les deux textes dans le même ordre relatif — pas nécessairement contiguës, juste dans l'ordre. Chaque ligne de la LCS est « inchangée ». Tout ce qui n'est pas dans la LCS est soit un ajout (uniquement dans la nouvelle version), soit une suppression (uniquement dans l'ancienne version).

Voici un exemple minimal. Supposons que le fichier original comporte quatre lignes :

A
B
C
D

Et que le nouveau fichier comporte quatre lignes :

A
X
C
D

La LCS est A, C, D. Ces trois lignes sont inchangées. B est supprimée (elle était dans l'original, pas dans la nouvelle version). X est ajoutée (elle est dans la nouvelle version, pas dans l'original).

Pour trouver la LCS efficacement, l'algorithme construit une matrice. Chaque cellule (i,j) contient la longueur de la LCS entre les i premières lignes de l'original et les j premières lignes de la nouvelle version.

Cellules violettes = correspondances (même ligne dans les deux textes). La valeur en bas à droite (3) est la longueur de la LCS — trois lignes sont inchangées. En remontant depuis cette cellule, on obtient la LCS : A, C, D.

Pourquoi cela explique le problème des « diffs différents »

Considérez un exemple légèrement différent où le mot « erreur » apparaît trois fois dans les deux fichiers, et où une occurrence passe de la ligne 12 à la ligne 15. La LCS pourrait faire correspondre la première et la troisième occurrence (montrant celle du milieu comme supprimée et réajoutée), ou la deuxième et la troisième (montrant la première comme supprimée et réajoutée). Les deux sont des solutions LCS valides de même longueur. Des algorithmes différents, ou des règles de départage différentes dans le même algorithme, choisiront différemment — et le diff résultant aura l'air complètement différent alors que les deux sont corrects.

L'algorithme de Myers, publié en 1986 et utilisé par Git, GNU diff et la plupart des outils modernes, minimise le nombre d'opérations d'édition (insertions + suppressions). Il est rapide et produit des diffs d'aspect naturel pour le code. Mais même Myers peut avoir des égalités dans certains cas limites.

Vue fractionnée vs vue unifiée — quand utiliser chacune

Tout outil de diff sérieux propose deux modes d'affichage. Comprendre quand chacun est approprié fait gagner du temps lors de la revue de code et du débogage.

Vue fractionnée (côte à côte)

Les deux versions apparaissent côte à côte. Les lignes modifiées sont placées sur la même ligne avec une surbrillance visuelle. Les lignes qui existent dans une version mais pas dans l'autre affichent un emplacement vide du côté opposé. Cela permet de voir facilement exactement à quoi ressemblait une ligne avant et après un changement, sans la charge mentale de suivre les préfixes + et -.

La vue fractionnée est idéale pour : les petits changements dans de longs fichiers, les changements où la comparaison avant/après importe (renommage, réécriture, correction), et les relecteurs non techniques qui n'ont pas internalisé la syntaxe du diff unifié.

Vue unifiée (style git)

Un flux unique de lignes. Les lignes de contexte inchangées n'ont pas de préfixe. Les lignes ajoutées sont préfixées par +. Les lignes supprimées sont préfixées par -. Une ligne d'en-tête (@@ -1,7 +1,8 @@) marque chaque « hunk » — un bloc contigu de changements avec quelques lignes de contexte autour.

La vue unifiée est idéale pour : les fichiers de patch (le format produit par diff -u et git diff), la revue de code par email, la copie de modifications à appliquer ailleurs, et l'affichage compact de nombreux petits changements dans de nombreux fichiers.

Diff mot à mot — précision à l'intérieur d'une ligne

Le diff au niveau des lignes marque toute une ligne comme modifiée même si un seul mot a changé. Sur de longues lignes — une requête SQL, une phrase dans la documentation, une valeur de configuration — cela n'est pas utile. Vous savez que quelque chose a changé sur cette ligne, mais vous devez quand même lire toute la ligne attentivement pour trouver le changement.

Le diff mot à mot va plus loin : il exécute un second algorithme de diff à l'intérieur de la ligne modifiée, trouvant exactement quels mots ont été ajoutés et lesquels ont été supprimés.

Avant (diff mot à mot appliqué à une ligne SQL modifiée) :

WHERE created_atupdated_at > '2024-01-01' AND status = 'en_attente''actif'

C'est l'équivalent de git diff --word-diff, qui affiche des marqueurs {-ancien-} et {+nouveau+} autour des mots exacts qui ont changé. C'est particulièrement utile pour :

  • Fichiers de configuration — une valeur est passée de true à false dans une ligne de 60 caractères
  • Documentation — un seul mot corrigé dans un long paragraphe
  • Migrations SQL — un nom de colonne changé dans une instruction ALTER TABLE
  • Valeurs JSON/YAML — une valeur de champ modifiée mais la clé et la structure sont restées les mêmes

La commande diff — référence complète

La commande Unix diff est disponible sur tous les systèmes Linux et macOS, et sur Windows via Git Bash ou WSL. C'est le moyen le plus fiable de comparer de gros fichiers, d'intégrer des diffs dans des scripts et de produire des fichiers de patch pour la distribution.

# Diff de base — format de sortie par défaut (difficile à lire)
diff fichier1.txt fichier2.txt

# Format unifié — lisible, compatible git, standard pour les patches
diff -u fichier1.txt fichier2.txt

# Unifié avec nombre de lignes de contexte personnalisé (3 par défaut)
diff -U5 fichier1.txt fichier2.txt     # 5 lignes de contexte autour de chaque changement

# Ignorer les différences de casse
diff -i fichier1.txt fichier2.txt

# Ignorer les différences d'espaces
diff -w fichier1.txt fichier2.txt      # Tous les espaces
diff -b fichier1.txt fichier2.txt      # Uniquement les espaces de début/fin

# Ignorer les lignes vides
diff -B fichier1.txt fichier2.txt

# Récursif — comparer des arborescences entières de répertoires
diff -r rep1/ rep2/

# Récursif + unifié + liste brève des fichiers modifiés d'abord
diff -rq rep1/ rep2/             # Liste simplement les fichiers qui diffèrent

# Produire un fichier de patch
diff -u original.txt modifie.txt > changements.patch

# Appliquer un patch
patch original.txt < changements.patch

# Appliquer en sens inverse (annuler le patch)
patch -R original.txt < changements.patch

# Appliquer un patch au style git (supprimer un composant de chemin)
patch -p1 < changements.patch

# Vérifier si le patch s'appliquerait proprement (simulation)
patch --dry-run -p1 < changements.patch

Combinaison avec d'autres outils

# Comparer deux fichiers JSON après formatage pour plus de lisibilité
diff <(python3 -m json.tool fichier1.json) <(python3 -m json.tool fichier2.json)

# Comparer deux fichiers YAML après normalisation
diff <(python3 -c "import yaml,sys; print(yaml.dump(yaml.safe_load(open('f1.yaml'))))") \
     <(python3 -c "import yaml,sys; print(yaml.dump(yaml.safe_load(open('f2.yaml'))))")

# Afficher uniquement les fichiers qui diffèrent dans une arborescence
diff -rq rep1/ rep2/ | grep "diffèrent"

# Compter les lignes modifiées
diff -u ancien.sql nouveau.sql | grep -c "^[+-]"

# Sortie en couleur dans le terminal
diff --color=always -u fichier1.txt fichier2.txt | less -R

git diff — les commandes que les développeurs utilisent quotidiennement

git diff est l'outil de diff le plus utilisé au monde. Il s'intègre profondément avec la zone d'index, les branches et l'historique de Git. Voici les commandes qui couvrent 95 % de l'utilisation quotidienne.

Utilisation de base

# Changements dans le répertoire de travail non encore indexés
git diff

# Changements indexés pour le prochain commit (ce qui sera commité)
git diff --staged
# ou de manière équivalente :
git diff --cached

# Tous les changements depuis le dernier commit (indexés + non indexés)
git diff HEAD

# Changements dans un fichier spécifique
git diff -- chemin/vers/fichier.js

# Changements dans un répertoire spécifique
git diff -- src/

Comparaison de branches et de commits

# Comparer deux branches (ce qui a changé de main vers branche-fonctionnalite)
git diff main..branche-fonctionnalite

# Équivalent : quels commits sont dans branche-fonctionnalite mais pas dans main
git diff main...branche-fonctionnalite   # trois points = point de divergence

# Comparer deux commits spécifiques
git diff abc1234..def5678

# Comparer un commit au répertoire de travail actuel
git diff abc1234

# Ce qui a changé dans un fichier spécifique entre deux branches
git diff main..branche-fonctionnalite -- src/api/users.js

# Comparer une branche à un tag
git diff v2.0.0..main

Options de format de sortie

# Diff mot à mot — surligne les mots exacts modifiés dans une ligne
git diff --word-diff

# Diff mot à mot en couleur (plus visuel)
git diff --word-diff=color

# Résumé : juste la liste des fichiers modifiés avec des statistiques
git diff --stat
git diff --stat main..branche-fonctionnalite

# Noms seulement — juste les noms de fichiers, pas le contenu
git diff --name-only main..branche-fonctionnalite

# Ignorer les changements d'espaces
git diff --ignore-all-space
git diff -w

# Afficher chaque changement comme un commit séparé dans la sortie
git diff -p

# Condensé : afficher uniquement les noms des fichiers qui diffèrent
git diff --diff-filter=M --name-only   # M = fichiers modifiés uniquement
# Filtres : A=ajouté, D=supprimé, M=modifié, R=renommé, C=copié

Flux de travail utiles

# Voir ce qui a changé dans le dernier commit
git diff HEAD~1 HEAD

# Voir ce qui a changé sur les N derniers commits
git diff HEAD~5 HEAD

# Trouver quand une ligne spécifique a été introduite
git log -S "chaîne à rechercher" --source --all

# Générer un patch à partir du dernier commit pour l'envoyer à un collègue
git format-patch -1 HEAD

# Appliquer un patch d'un collègue
git am < 0001-leur-changement.patch

# Indexation interactive — n'indexer que des modifications spécifiques dans un fichier
git add -p fichier.js   # Parcourir chaque hunk et choisir oui/non/diviser

Diff par type de contenu — la bonne approche pour chaque format

Fichiers JSON

Le diff texte brut de JSON est souvent trompeur. Deux fichiers JSON peuvent représenter exactement les mêmes données mais avoir l'air complètement différents en tant que texte : ordre des clés différent, indentation différente, guillemets différents. Un diff au niveau des lignes montrerait des centaines de changements alors que rien n'a changé structurellement.

Le flux de travail correct : formatez d'abord les deux fichiers de manière cohérente, puis diff le résultat formaté. Utilisez le Formateur JSON DevToolbox ou la ligne de commande :

# Formater les deux fichiers, puis diff
diff <(python3 -m json.tool --sort-keys fichier1.json) \
     <(python3 -m json.tool --sort-keys fichier2.json)

# Alternative Node.js
diff <(node -e "console.log(JSON.stringify(require('./fichier1.json'),null,2))") \
     <(node -e "console.log(JSON.stringify(require('./fichier2.json'),null,2))")

Pour les vérifications d'égalité sémantique (en ignorant totalement le formatage), utilisez le score de similarité du Comparateur de texte DevToolbox ou comparez les sorties formatées — si le diff est vide, les données sont identiques.

Fichiers SQL (migrations)

Lors de la revue de migrations SQL avant déploiement, formatez d'abord les deux scripts. Le Formateur SQL DevToolbox normalise les espaces, la casse des mots-clés et l'indentation. Après formatage, un diff ne montre que les changements structurels — pas le bruit de formatage.

# Comparer deux scripts de migration dans le terminal
diff -u --label "original" --label "modifié" \
    migration_v1.sql migration_v2.sql

# Dans Git, toujours visualiser les diffs SQL avec le diff mot à mot
git diff --word-diff -- migrations/

Fichiers de configuration YAML

Les ancres YAML et les chaînes multi-lignes peuvent rendre les diffs texte brut confus. Le même principe s'applique : normalisez d'abord. Analysez et ré-affichez les deux fichiers avec un formatage cohérent avant de diff.

# Normaliser et diff des fichiers YAML
diff <(python3 -c "import yaml; print(yaml.dump(yaml.safe_load(open('config-v1.yaml')), sort_keys=True))") \
     <(python3 -c "import yaml; print(yaml.dump(yaml.safe_load(open('config-v2.yaml')), sort_keys=True))")

Prose et documentation

Pour les documents textuels, le diff mot à mot est presque toujours plus utile que le diff ligne par ligne. Une seule phrase de 80 caractères contient de nombreux mots — marquer toute la ligne comme modifiée vous en dit moins que de montrer exactement quel mot a été corrigé.

Dans le terminal, git diff --word-diff fonctionne bien pour Markdown. Pour les fichiers texte brut, le mode diff mot à mot du Comparateur de texte DevToolbox est l'option la plus visuelle.

Lire un diff unifié — annoté ligne par ligne

Le format de diff unifié est la sortie standard de diff -u et git diff. C'est le format utilisé dans les fichiers de patch, les diffs de pull request et les outils de revue de code. Une fois que vous savez le lire, cela devient une seconde nature.

--- a/src/config.js    2025-06-07 09:00:00
← Fichier original (préfixe a/ ajouté par git)
+++ b/src/config.js    2025-06-08 14:23:00
← Fichier modifié

@@ -12,7 +12,8 @@ function loadConfig() {
← En-tête de hunk : lignes originales 12-18 → nouvelles lignes 12-19. Nom de la fonction affiché comme contexte.

const host = process.env.HOST || 'localhost';
const port = process.env.PORT || 3000;
← Lignes de contexte (espace en tête) : inchangées, affichées pour orientation

- const timeout = 5000;
← Ligne supprimée (préfixe -) : était dans l'original, pas dans la nouvelle version
+ const timeout = parseInt(process.env.TIMEOUT) || 5000;
+ const maxRetries = parseInt(process.env.MAX_RETRIES) || 3;
← Lignes ajoutées (préfixe +) : dans la nouvelle version, pas dans l'original

return { host, port, timeout };
← Ligne de contexte : inchangée. Note : maxRetries devrait probablement être ajouté ici aussi.
}

L'en-tête de hunk expliqué

L'en-tête @@ -12,7 +12,8 @@ est la partie la plus cryptique pour les nouveaux venus. Voici ce que chaque nombre signifie :

PartieSignificationDans notre exemple
-12,7Dans le fichier original : commence à la ligne 12, s'étend sur 7 lignesLignes 12–18 de l'original
+12,8Dans le nouveau fichier : commence à la ligne 12, s'étend sur 8 lignesLignes 12–19 de la nouvelle version (une ligne ajoutée)
Texte après @@La fonction ou la section la plus proche (contexte optionnel)function loadConfig() {

La différence de nombre de lignes (7 vs 8 dans notre exemple) vous indique immédiatement combien de lignes ont été ajoutées ou supprimées : nouveau_compte - ancien_compte = lignes nettes ajoutées. Un nombre négatif signifie que plus de lignes ont été supprimées qu'ajoutées dans ce hunk.

Appliquer et inverser un patch

# Enregistrer un diff dans un fichier de patch
git diff main..branche-fonctionnalite > fonctionnalite.patch
# ou
diff -u original.conf nouveau.conf > changements.patch

# Appliquer le patch
patch original.conf < changements.patch

# Vérifier que le patch s'appliquerait proprement avant de le faire réellement

# Inverser le patch (annuler les modifications)
patch -R original.conf < changements.patch

# Appliquer une sortie de git format-patch (format différent, inclut le message de commit)
git am < 0001-ajout-configuration-timeout.patch

Foire aux questions

Quelle est la différence entre la vue fractionnée et la vue unifiée ?

La vue fractionnée montre les deux textes côte à côte avec les différences alignées sur la même ligne. La vue unifiée montre un flux unique avec les lignes ajoutées précédées de + et les lignes supprimées par -, ce qui est le format produit par diff -u et git diff. La vue fractionnée est plus facile à lire pour les petits changements ciblés ; la vue unifiée est plus compacte, est le format standard pour les fichiers de patch et est nécessaire pour appliquer des patches avec la commande patch.

Pourquoi deux outils de diff affichent-ils des résultats différents pour le même changement ?

La comparaison de textes trouve la plus longue sous-séquence commune (LCS) entre deux fichiers — la plus longue séquence de lignes qui apparaît dans les deux dans le même ordre. Lorsque plusieurs solutions LCS de longueur égale existent, différents algorithmes les départagent différemment. L'algorithme de Myers (utilisé par Git, GNU diff et la plupart des outils modernes) minimise le nombre d'opérations d'édition, mais certains textes ont plusieurs solutions minimales équivalentes qui semblent complètement différentes en sortie. Les deux sont correctes ; aucune n'est « plus juste » que l'autre.

Qu'est-ce que le diff mot à mot et quand dois-je l'utiliser ?

Le diff mot à mot (aussi appelé « word diff ») exécute une seconde comparaison à l'intérieur de chaque ligne modifiée pour mettre en évidence exactement quels mots ont été ajoutés ou supprimés, au lieu de marquer toute la ligne comme modifiée. Utilisez-le pour les fichiers de configuration avec de longues lignes, la documentation, les migrations SQL et tout contenu où une ligne a changé mais où la majeure partie est restée la même. Dans git : git diff --word-diff. Dans le Comparateur de texte DevToolbox, cela se fait automatiquement dans les lignes modifiées.

Comment fonctionne la fonctionnalité de partage d'URL ?

Le bouton Partager du Comparateur de texte DevToolbox encode les deux textes et toutes les options (ignorer la casse, ignorer les espaces, mode d'affichage, langage) dans le fragment d'URL en utilisant un encodage Base64. Rien n'est stocké sur un serveur — l'état complet est dans l'URL elle-même. Envoyez le lien à un collègue et il verra exactement le même diff, décodé et rendu localement dans son navigateur. Cela fonctionne également pour mettre en signet une comparaison spécifique.

Qu'est-ce que l'algorithme LCS ?

LCS signifie Longest Common Subsequence (plus longue sous-séquence commune). Il trouve la plus longue séquence d'éléments (des lignes, dans le contexte du diff texte) qui apparaît dans les deux entrées dans le même ordre relatif, mais pas nécessairement de manière contiguë. L'algorithme de diff utilise la LCS pour identifier les lignes inchangées — elles forment la LCS. Tout ce qui n'est pas dans la LCS est soit un ajout, soit une suppression. L'algorithme de Myers, publié en 1986 et utilisé par Git, calcule un script d'édition minimal efficacement en utilisant un chemin diagonal à travers le graphe d'édition.

Comment appliquer un fichier patch à partir d'un diff unifié ?

Utilisez la commande patch : patch original.txt < changements.patch. Pour un patch au style git qui inclut un préfixe de chemin : patch -p1 < changements.patch (le -p1 supprime le préfixe a/ que git ajoute). Pour tester sans modifier les fichiers : patch --dry-run -p1 < changements.patch. Pour inverser un patch : patch -R original.txt < changements.patch. Pour les patches créés avec git format-patch (qui incluent les messages de commit), utilisez git am < fichier_patch à la place.
