Le script qui a tourné pendant deux mois sur le mauvais serveur

Une équipe que je connais migrait une application Django de son ancien VPS vers un nouveau serveur cloud. Ils ont déplacé le code, migré la base de données, mis à jour le DNS. La liste de contrôle de déploiement était minutieuse. Une chose qu'ils n'ont pas vérifiée : le crontab de l'ancien serveur.

Pendant deux mois après la migration, un script de sauvegarde nocturne a continué à s'exécuter sur le serveur mis hors service — récupérant des données en direct via un jeton d'API qui n'avait jamais été révoqué, stockant des archives compressées dans un répertoire que personne ne surveillait, accumulant des frais de stockage sur une machine qu'ils pensaient inactive. Lorsqu'ils ont finalement remarqué (un pic de facturation), ils ont trouvé 61 gigaoctets de sauvegardes sur un serveur qui n'était plus censé exister.

La tâche cron elle-même fonctionnait parfaitement. L'expression était correcte. Le script tournait impeccablement. Ce qui a échoué, c'est le processus de déploiement qui l'entourait : aucune documentation des tâches cron existantes, aucune alerte lors de l'exécution, aucun journal que quelqu'un lisait.

Ce guide porte sur le cycle de vie complet d'une tâche cron — pas seulement l'écriture de l'expression, mais l'écriture d'un script solide, son test en toute sécurité avant sa mise en production, et la mise en place d'une surveillance pour savoir quand il fonctionne et quand il ne fonctionne pas.

Ce guide traite de la construction et du déploiement de vraies tâches. Si vous souhaitez comprendre la syntaxe des expressions cron en profondeur — opérateurs, piège du OU, écueils des fuseaux horaires et de l'heure d'été — lisez notre article complémentaire : Expressions Cron : Le guide complet. Les deux articles couvrent délibérément des terrains différents.

Étape 1 Écrire un script qui fonctionne de manière isolée

La première règle des scripts cron : n'écrivez jamais de logique directement dans la ligne cron. Les commandes en ligne sont impossibles à tester, impossibles à versionner et impossibles à lire six mois plus tard. Chaque tâche cron doit appeler un fichier de script.

Avant même de penser à l'expression cron, écrivez votre script pour qu'il puisse s'exécuter de manière autonome — depuis le terminal, à la main — et produire un journal clair de ce qu'il a fait. Voici le modèle de script shell que j'utilise pour chaque nouvelle tâche cron :

🐚Modèle de script shell prêt pour la production
Copiez et adaptez 
#!/bin/bash
# ─────────────────────────────────────────────────────────────────
# Script : /opt/scripts/backup.sh
# Objectif : Sauvegarde nocturne PostgreSQL vers S3
# Cron    : 0 2 * * * /opt/scripts/backup.sh
# Responsable : deploy@monserveur.com
# ─────────────────────────────────────────────────────────────────
set -euo pipefail   # Quitter en cas d'erreur, variable non définie, échec de pipeline

LOG=/var/log/backup.log
DATE=$(date '+%Y-%m-%d_%H-%M-%S')
BACKUP_DIR=/tmp/backups
S3_BUCKET=s3://sauvegardes-entreprise/postgres

mkdir -p "$BACKUP_DIR"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG"
}

log "=== Sauvegarde démarrée ==="

# 1. Vider la base de données
log "Vidage de la base de données..."
pg_dump -U postgres mabase > "$BACKUP_DIR/mabase_$DATE.sql"

# 2. Compresser
log "Compression..."
gzip "$BACKUP_DIR/mabase_$DATE.sql"

# 3. Téléverser vers S3
log "Téléversement vers S3..."
aws s3 cp "$BACKUP_DIR/mabase_$DATE.sql.gz" "$S3_BUCKET/"

# 4. Nettoyer le fichier temporaire local
rm -f "$BACKUP_DIR/mabase_$DATE.sql.gz"

log "=== Sauvegarde terminée avec succès ==="
exit 0

Trois choses que ce modèle fait et qui comptent dans un contexte cron :

  • set -euo pipefail le script s'arrête immédiatement si une commande échoue. Sans cela, un pg_dump en échec continue silencieusement jusqu'à l'étape de téléversement, envoyant un fichier vide ou corrompu.
  • Journalisation explicite avec horodatage chaque action significative est journalisée avec un horodatage. Quand vous lisez le journal à 9h et que vous voyez que la sauvegarde s'est terminée à 02:03:17, vous savez qu'elle a fonctionné. Quand vous voyez que le journal s'arrête à "Vidage de la base de données...", vous savez exactement où elle a échoué.
  • Bloc de commentaires d'en-tête la ligne cron, l'objectif et le responsable sont documentés à l'intérieur du script lui-même. Quand quelqu'un trouve ce script six mois plus tard, il sait ce qu'il fait et où il s'exécute.
Rendez toujours le script exécutable avant de l'ajouter au crontab : 
chmod +x /opt/scripts/backup.sh
# Vérifier
ls -la /opt/scripts/backup.sh
# → -rwxr-xr-x 1 deploy deploy 1234 Jun 20 08:00 /opt/scripts/backup.sh

Étape 2 Construire et valider l'expression cron

Une fois que votre script fonctionne de manière autonome, vous avez besoin d'une expression qui se déclenche exactement au bon moment. Les deux erreurs qui coûtent le plus de temps ici sont les expressions qui semblent correctes mais se déclenchent à des intervalles inattendus, et les expressions avec des erreurs de syntaxe qui sont ignorées silencieusement.

Ouvrez le Générateur Cron DevToolbox. Tapez votre planification prévue en langage clair — ou construisez-la avec les curseurs visuels — et l'outil vous montre les 10 prochaines dates d'exécution exactes. Cela prend 30 secondes et attrape les erreurs de planification avant qu'elles n'atteignent la production.

Étape 3. Ajouter au crontab en toute sécurité

Ne modifiez jamais directement /etc/crontab pour les tâches utilisateur. Utilisez crontab -e — cela ouvre le crontab dans un fichier temporaire, valide la syntaxe lors de l'enregistrement et n'installe la nouvelle version que si elle est valide. Une erreur de syntaxe avec une modification directe du fichier casse silencieusement l'ensemble du crontab.

⚙La structure complète du crontab de production
Modèle de production 
# Ouvrir pour modification (utilisez 'crontab -e' en tant qu'utilisateur propriétaire des scripts)
crontab -e
# ══════════════════════════════════════════════════════
# Crontab de l'utilisateur : deploy
# Serveur : prod-web-01 — Dernière mise à jour : 2026-06-20
# ══════════════════════════════════════════════════════

# Environnement
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
MAILTO=ops@example.com
CRON_TZ=UTC
HOME=/home/deploy

# ── Sauvegardes ──────────────────────────────────────────
# Sauvegarde quotidienne PostgreSQL à 2h UTC
0 2 * * * /opt/scripts/backup.sh >> /var/log/backup.log 2>&1

# ── Maintenance ──────────────────────────────────────────
# Nettoyer les fichiers journaux de plus de 30 jours — tous les dimanches à 3h
0 3 * * 0 /opt/scripts/cleanup_logs.sh >> /var/log/cleanup.log 2>&1

# ── Synchronisation ─────────────────────────────────────
# Synchroniser le catalogue produits depuis l'API — toutes les 15 minutes pendant les heures ouvrées
*/15 8-18 * * 1-5 /opt/scripts/sync_catalog.py >> /var/log/sync.log 2>&1

Quelques conventions qui rendent un crontab lisible six mois plus tard :

  • Un commentaire par tâche dites ce que fait la tâche en langage clair, pas seulement la planification.
  • Regroupez les tâches par catégorie sauvegardes, maintenance, synchronisation, rapports. Chaque groupe reçoit un commentaire d'en-tête.
  • Redirigez toujours stdout et stderr >> /var/log/job.log 2>&1 sur chaque ligne. La sortie non redirigée est envoyée à MAILTO si configuré, ou est silencieusement ignorée.
  • Documentez le fichier en haut nom du serveur, utilisateur, dernière mise à jour. Lorsque vous vous connectez en SSH sur un serveur inconnu à 2h du matin, ces métadonnées sont précieuses.
Commandes crontab utiles
crontab -l              # Lister le crontab de l'utilisateur courant
crontab -e              # Modifier le crontab de l'utilisateur courant
sudo crontab -l         # Lister le crontab de root (fichier séparé !)
crontab -l > ~/crontab.sauv  # Exporter avant de faire des modifications
crontab ~/crontab.sauv       # Restaurer à partir de la sauvegarde

Étape 4 Tester avant la première exécution planifiée

Le mode de défaillance le plus courant des tâches cron est quelque chose qui fonctionne parfaitement dans le terminal et échoue silencieusement dans le contexte cron. La raison est presque toujours l'environnement : cron s'exécute avec un PATH minimal, aucun profil shell chargé et un répertoire de travail différent de celui de votre session terminal.

Vous pouvez reproduire cet environnement exact avant que la tâche ne se déclenche pour la première fois :

🧪Simuler l'environnement cron
Tester avant de déployer
Méthode 1 env -i (le plus proche du vrai cron)
# Cela supprime votre environnement actuel et exécute le script
# avec uniquement les variables que cron aurait
env -i HOME=/home/deploy \
       USER=deploy \
       LOGNAME=deploy \
       SHELL=/bin/bash \
       PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin \
       /bin/bash -c '/opt/scripts/backup.sh'

# Lisez attentivement la sortie — toute erreur "commande introuvable"
# signifie que vous devez utiliser un chemin absolu ou étendre le PATH dans le crontab
Méthode 2 — ligne cron temporaire avec un délai de 2 minutes
# Trouver l'heure actuelle
date '+%M %H'   # ex. "32 14" → 14:32

# Ajouter une ligne cron qui se déclenche dans 2 minutes
# Si l'heure actuelle est 14:32, mettez la minute à 34
34 14 * * * /opt/scripts/backup.sh >> /tmp/test_backup.log 2>&1

# Attendez 3 minutes, puis lisez le journal
cat /tmp/test_backup.log

# Supprimez la ligne de test du crontab une fois terminé
crontab -e
Méthode 3 — vérifier que cron a bien distribué la tâche
# Après l'heure d'exécution prévue, vérifiez le journal système cron
grep CRON /var/log/syslog | grep backup | tail -5
# Vous devriez voir une ligne comme :
# Jun 20 02:00:01 serveur CRON[12345]: (deploy) CMD (/opt/scripts/backup.sh)
# Si cette ligne est absente → l'expression n'a pas correspondu, ou le démon cron ne tourne pas
# Si cette ligne est présente mais que la tâche a échoué → vérifiez /var/log/backup.log
La règle des deux journaux : Vérifiez toujours deux journaux lors du débogage d'une tâche cron. D'abord le journal système cron (/var/log/syslog) pour confirmer que la tâche a été distribuée. Ensuite votre journal applicatif (/var/log/backup.log) pour voir ce que le script a fait. Si le premier journal a la ligne de distribution mais que le second est vide, le script a planté avant d'écrire quoi que ce soit — vérifiez les permissions et le comportement de set -e.

Étape 5 — Surveiller et recevoir des alertes

Une tâche cron qui s'exécute silencieusement chaque nuit est invisible jusqu'à ce qu'elle tombe en panne. L'objectif de la surveillance est de rendre le succès visible et l'échec bruyant. Vous voulez savoir immédiatement quand une tâche échoue — pas quand la sauvegarde est manquante à 3h du matin un vendredi soir ou quand une synchronisation de catalogue produits a échoué pendant six semaines.

🔔Trois couches de surveillance — implémentez les trois
Ne sautez pas cette étape
Couche 1 Fichiers journaux (minimum viable)
# Dans votre crontab : redirigez tout vers un journal avec un préfixe horodaté
0 2 * * * /opt/scripts/backup.sh >> /var/log/backup.log 2>&1

# Dans votre script : préfixez chaque ligne de journal par un horodatage
log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" >> /var/log/backup.log; }

# Rotation des journaux pour éviter le remplissage du disque (configuration logrotate)
# /etc/logrotate.d/taches-cron
/var/log/backup.log {
    daily
    rotate 30
    compress
    missingok
    notifempty
}
Couche 2 Email en cas d'échec (simple mais efficace)
#!/bin/bash
# Dans votre script : envoyez un email si quelque chose tourne mal
MAILTO=ops@example.com
SCRIPT_NAME=$(basename "$0")

on_error() {
    local line=$1
    echo "ERREUR dans $SCRIPT_NAME à la ligne $line $(date)" | \
        mail -s "ÉCHEC CRON : $SCRIPT_NAME" "$MAILTO"
}
trap 'on_error $LINENO' ERR

# Le reste de votre script ci-dessous
# Si une commande échoue, le trap se déclenche et envoie un email
Couche 3. Ping de healthcheck (le plus fiable)
#!/bin/bash
# Pinger une URL de healthcheck quand la tâche RÉUSSIT
# Des services comme healthchecks.io ou BetterUptime vous alertent
# si le ping n'arrive pas dans la fenêtre attendue

HEALTHCHECK_URL="https://hc-ping.com/votre-uuid-unique"

# Votre logique métier
pg_dump -U postgres mabase | gzip > /tmp/backup.sql.gz
aws s3 cp /tmp/backup.sql.gz s3://mon-bucket/

# Signaler le succès — si cette ligne n'est jamais atteinte, le healthcheck
# vous alertera une fois la fenêtre d'exécution attendue passée
curl -fsS --retry 3 "$HEALTHCHECK_URL" > /dev/null 2>&1

# Pour signaler un échec explicitement :
# curl -fsS "$HEALTHCHECK_URL/fail" > /dev/null 2>&1
Les services de healthcheck comme healthchecks.io (version gratuite disponible) fonctionnent à l'inverse de la surveillance traditionnelle : au lieu de vérifier si un service tourne, ils vous alertent quand un ping attendu n'arrive pas. Cela attrape les tâches qui cessent de s'exécuter silencieusement — aucun changement de code, aucune erreur, juste une tâche qui a été supprimée du crontab lors d'une reconstruction de serveur.

4 vraies tâches cron avec le code complet

1Sauvegarde nocturne de base de données vers S3
Shell + AWS CLI
Ligne cron
0 2 * * * /opt/scripts/db_backup.sh >> /var/log/db_backup.log 2>&1
Script
#!/bin/bash
set -euo pipefail
DATE=$(date '+%Y-%m-%d')
DB_NAME=myapp_production
BUCKET=s3://sauvegardes-entreprise

log() { echo "[$(date '+%H:%M:%S')] $*"; }

log "Démarrage de la sauvegarde de $DB_NAME"
pg_dump -U postgres "$DB_NAME" | gzip > /tmp/"$DB_NAME"_"$DATE".sql.gz
log "Vidage terminé. Téléversement..."
aws s3 cp /tmp/"$DB_NAME"_"$DATE".sql.gz "$BUCKET"/postgres/
rm /tmp/"$DB_NAME"_"$DATE".sql.gz

# Supprimer les sauvegardes de plus de 30 jours de S3
aws s3 ls "$BUCKET"/postgres/ | awk '{print $4}' | \
  while read f; do
    age=$(( ($(date +%s) - $(date -d "${f:8:10}" +%s)) / 86400 ))
    [ "$age" -gt 30 ] && aws s3 rm "$BUCKET/postgres/$f"
  done

log "Sauvegarde terminée et anciens fichiers nettoyés"
curl -fsS "https://hc-ping.com/VOTRE-UUID" > /dev/null 2>&1
2Synchronisation d'API Python — toutes les 15 minutes en semaine
Python 3 + venv
Ligne cron
*/15 8-18 * * 1-5 /opt/venv/bin/python /opt/scripts/sync_products.py >> /var/log/sync.log 2>&1
Pourquoi /opt/venv/bin/python et pas python3 ?

Utiliser le binaire Python du venv signifie que tous les paquets installés (requests, psycopg2, etc.) sont disponibles sans activer l'environnement virtuel. Pas de source activate dans la ligne cron — cela nécessite un shell avec chargement de profil, ce que cron ne fait pas.

Script
#!/usr/bin/env python3
"""
Synchronise le catalogue produits depuis l'API du fournisseur.
Cron : */15 8-18 * * 1-5
"""
import requests, psycopg2, logging, sys
from datetime import datetime

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s %(levelname)s %(message)s',
    stream=sys.stdout
)
log = logging.getLogger(__name__)

API_URL = "https://api.fournisseur.com/v2/produits"
API_KEY = "sk_live_xxxxx"   # Utilisez une variable d'env en production : os.environ['SUPPLIER_API_KEY']
DB_DSN  = "postgresql://app:secret@localhost/myapp"

def sync():
    log.info("Récupération des produits depuis l'API...")
    r = requests.get(API_URL, headers={"Authorization": f"Bearer {API_KEY}"}, timeout=30)
    r.raise_for_status()
    produits = r.json()["data"]
    log.info(f"Reçu {len(produits)} produits")

    conn = psycopg2.connect(DB_DSN)
    cur = conn.cursor()

    updated = 0
    for p in produits:
        cur.execute("""
            INSERT INTO produits (id, nom, prix, stock, updated_at)
            VALUES (%(id)s, %(nom)s, %(prix)s, %(stock)s, NOW())
            ON CONFLICT (id) DO UPDATE
            SET nom=EXCLUDED.nom, prix=EXCLUDED.prix,
                stock=EXCLUDED.stock, updated_at=NOW()
        """, p)
        updated += 1

    conn.commit()
    cur.close()
    conn.close()
    log.info(f"Synchronisation terminée — {updated} produits upsertés")

if __name__ == "__main__":
    sync()

Ligne cron

#!/bin/bash
set -euo pipefail

LOG_DIRS=("/var/log/app" "/var/log/nginx" "/tmp/uploads")
MAX_AGE_DAYS=30

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*"; }
log "=== Nettoyage des journaux démarré ==="

total_freed=0
for dir in "${LOG_DIRS[@]}"; do
    if [ ! -d "$dir" ]; then
        log "Répertoire $dir introuvable — ignoré"
        continue
    fi

    # D'ABORD UN ESSAI À BLANC — lister les fichiers qui seraient supprimés
    log "Fichiers de plus de $MAX_AGE_DAYS jours dans $dir :"
    find "$dir" -name "*.log*" -mtime +"$MAX_AGE_DAYS" -type f | \
        while read f; do
            size=$(du -sh "$f" | cut -f1)
            log "  → $f ($size)"
        done

    # Supprimer réellement
    count=$(find "$dir" -name "*.log*" -mtime +"$MAX_AGE_DAYS" -type f | wc -l)
    find "$dir" -name "*.log*" -mtime +"$MAX_AGE_DAYS" -type f -delete
    log "Supprimé $count fichiers de $dir"
done

log "=== Nettoyage terminé ==="
Journalisez toujours la liste des fichiers avant de les supprimer. Le script de nettoyage ci-dessus écrit le nom et la taille de chaque fichier qu'il supprime. Si quelque chose est nettoyé par erreur, vous savez au moins ce que c'était et quand cela s'est produit.

Foire aux questions

Comment exécuter un script Python avec cron ?

Utilisez le chemin absolu vers l'interpréteur Python de votre environnement virtuel : /opt/venv/bin/python /opt/scripts/monscript.py. Le Python du venv connaît déjà tous les paquets installés, vous n'avez donc pas besoin d'activer l'environnement virtuel. Si vous utilisez le Python système sans venv, utilisez /usr/bin/python3. N'utilisez jamais un simple python ou python3 dans une ligne cron — le PATH minimal de cron pourrait ne pas le trouver, ou trouver la mauvaise version.

Comment tester une tâche cron sans attendre l'heure planifiée ?

Exécutez le script dans un environnement minimal qui imite cron : env -i HOME=/home/votreatilisateur PATH=/usr/bin:/bin /bin/bash -c '/opt/scripts/monscript.sh'. Cela supprime votre environnement shell actuel et exécute le script avec uniquement les variables que cron fournirait. Toute erreur "commande introuvable" que vous obtenez ici est une erreur que vous obtiendriez également de cron. Vous pouvez également ajouter une ligne cron temporaire qui se déclenche dans 2 minutes, surveiller le fichier journal, puis supprimer la ligne.

Comment recevoir une alerte email lorsqu'une tâche cron échoue ?

Ajoutez MAILTO=vous@example.com en haut de votre crontab. Toute sortie non redirigée de vos lignes cron sera envoyée par email à cette adresse — y compris les messages d'erreur. Cela nécessite qu'un service de messagerie local (postfix ou sendmail) soit en cours d'exécution. Pour une alerte plus fiable, ajoutez un trap 'envoyer_alerte' ERR à l'intérieur de votre script qui envoie une notification via curl vers Mailgun, SendGrid, un webhook Slack ou un service de healthcheck. Ainsi, vous contrôlez exactement ce qui déclenche une alerte et ce que le message contient.

Quelle est la manière la plus sûre de supprimer d'anciens fichiers avec une tâche cron ?

Exécutez toujours la commande find sans -delete d'abord et redirigez la sortie vers un fichier journal. Passez soigneusement en revue la liste avant d'ajouter le drapeau de suppression. Dans votre script de production, journalisez le nom et la taille de chaque fichier avant de le supprimer, afin d'avoir une trace si quelque chose est nettoyé par erreur. N'exécutez jamais find -delete en tant que root sur un chemin large comme /var sans un filtre bien testé. Le script de nettoyage de ce guide montre un modèle sûr avec journalisation et traitement par répertoire.

Comment faire en sorte qu'une tâche cron s'exécute une fois après un redémarrage du serveur ?

Utilisez la chaîne spéciale @reboot dans votre crontab : @reboot /opt/scripts/start_service.sh >> /var/log/start.log 2>&1. Cela se déclenche une fois, peu après le démarrage du démon cron suite à un démarrage du système. Notez que @reboot n'est pas disponible sur toutes les implémentations et n'est pas pris en charge sur AWS CloudWatch ou Kubernetes CronJob. Pour les tâches de démarrage critiques en production sur un serveur Linux moderne, un service systemd avec WantedBy=multi-user.target est plus fiable et plus facile à surveiller.

Comment voir la sortie d'une tâche cron qui s'est déjà exécutée ?

Si vous avez redirigé la sortie vers un fichier journal (ce que vous devriez toujours faire), lisez-le avec cat /var/log/monjob.log ou tail -50 /var/log/monjob.log. Pour vérifier si cron a bien distribué la tâche, cherchez dans le journal système cron : grep CRON /var/log/syslog | grep monjob sur Debian/Ubuntu, ou grep CRON /var/log/cron | grep monjob sur RHEL/CentOS. Si le journal système a la ligne de distribution mais que votre journal applicatif est vide, le script a planté avant d'écrire quoi que ce soit — vérifiez le code de sortie du script et le comportement de set -e.

Puis-je exécuter plusieurs commandes sur une seule ligne cron ?

Techniquement oui, en utilisant des points-virgules (exécuter la commande suivante quel que soit le code de sortie précédent) ou && (s'arrêter au premier échec). Mais pour tout ce qui dépasse une seule commande simple, écrivez plutôt un script shell. Les scripts sont plus faciles à tester, plus faciles à lire, prennent en charge une gestion des erreurs correcte avec set -euo pipefail, peuvent être versionnés avec git et produisent des journaux structurés. Une ligne cron devrait idéalement ne contenir que : la planification, le chemin du script, la redirection de journal. Tout le reste appartient au script.
