Comprendre les expressions Cron en profondeur — Guide complet 2026

Publié le 17 mai 2026 par l'équipe DevToolbox — 15 min de lecture

Une expression cron semble simple : cinq champs et quelques étoiles. Pourtant, derrière cette simplicité se cachent des subtilités qui transforment une tâche planifiée en cauchemar silencieux. Ce guide vous donne toutes les clés pour maîtriser le format cron, que vous soyez sur Linux, Quartz, AWS ou Kubernetes. Vous y trouverez des tableaux d’opérateurs, les pièges du jour du mois, la gestion avancée des fuseaux horaires, une checklist de débugging et bien plus.

Pour tester chaque exemple en direct, utilisez notre générateur d’expressions cron gratuit, qui valide la syntaxe et affiche les prochaines exécutions.

1. Structure de base et opérateurs par champ

La syntaxe standard (Vixie cron, cronie) comporte cinq champs : minute (0-59), heure (0-23), jour du mois (1-31), mois (1-12), jour de la semaine (0-7, 0 ou 7 = dimanche). Les implémentations étendues (Quartz, AWS CloudWatch) ajoutent un sixième champ (secondes) en première position. Kubernetes CronJob utilise 5 champs, mais supporte un fuseau horaire explicite depuis la version 1.25.

Chaque champ accepte des opérateurs. Voici le tableau complet, avec leur disponibilité selon l’environnement :

2. Le piège du comportement OR (jour du mois + jour de la semaine)

C’est l’un des pièges les moins documentés. Quand les deux champs jour du mois et jour de la semaine sont spécifiés (aucun n’est *), cron exécute la tâche quand l’un OU l’autre correspond — pas quand les deux sont vrais. Exemple concret : 

0 0 1 * 1

Cette expression s’exécute le 1er de chaque mois ET chaque lundi, pas uniquement le premier lundi du mois. Si vous vouliez le premier lundi du mois uniquement, il faut utiliser l’opérateur # ou L de Quartz, ou combiner avec un test dans le script. Pour éviter ce comportement surprenant, laissez toujours l’un des deux champs à * ou utilisez ? en environnement Quartz.

3. Noms textuels des mois et jours

Le cron Linux accepte des abréviations : MON, TUE, WED pour les jours, et JAN, FEB pour les mois. Cela rend les expressions plus lisibles : 

0 9 * JAN-MAR MON-FRI

Attention, les noms sont en anglais et ne fonctionnent qu’avec le cron traditionnel (Vixie, cronie). Ni Quartz ni Kubernetes ne les supportent. Si vous générez une expression pour plusieurs environnements, privilégiez les valeurs numériques.

5. Gestion des fuseaux horaires — en détail

Une tâche planifiée à 0 2 * * * s’exécute à 2h du matin, mais dans quel fuseau ? La réponse dépend de l’environnement.

  • Linux classique : cron utilise l’heure système. Vérifiez avec timedatectl ou date +%Z. On peut définir CRON_TZ=Europe/Paris en haut de la crontab (cronie uniquement).
  • AWS CloudWatch : toujours en UTC. Aucune variable possible. Si vous voulez une exécution à 8h heure de Paris en été, écrivez 0 6 * * ? * (UTC+2).
  • Kubernetes CronJob (≥1.25) : spécifiez .spec.timeZone: "Europe/Paris". La syntaxe cron reste en 5 champs, l’heure est interprétée dans ce fuseau.
  • Piège du changement d’heure : une tâche à 0 2 * * * peut s’exécuter deux fois (en octobre) ou jamais (en mars) si le fuseau passe de CET à CEST. Privilégiez des heures comme 3h du matin (3:00) ou utilisez CRON_TZ avec UTC pour éviter ce problème.

6. Le comportement de chevauchement (overlap)

Si une tâche cron prend 10 minutes et est planifiée toutes les 5 minutes, le cron standard lance une nouvelle instance sans attendre la fin de la précédente. Cela peut entraîner des dizaines de processus concurrents. Solutions :

  • Linux : utilisez flock pour verrouiller l’exécution : flock -n /tmp/monjob.lock /chemin/script.sh
  • Ubuntu : la commande run-one (paquet run-one) garantit qu’une seule instance tourne.
  • Kubernetes : positionnez .spec.concurrencyPolicy: Forbid dans le CronJob.

7. Variables d’environnement dans la crontab

La crontab accepte des variables en entête. C’est essentiel car le PATH par défaut dans cron est différent du shell interactif, source de 90% des "mon script marche en manuel mais pas en cron". 

SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin
MAILTO=admin@example.com
CRON_TZ=Europe/Paris

*/10 * * * * /opt/scripts/mon_job.sh

Pour désactiver les emails système (sortie standard), laissez MAILTO="". Redirigez toujours les sorties vers un fichier : >> /var/log/mon_job.log 2>&1.

8. Cron vs systemd timers — quand switcher

Les timers systemd offrent une alternative moderne avec journalisation centralisée, dépendances entre services et rattrapage après redémarrage. Comparaison directe : 

# Cron : toutes les 5 minutes
*/5 * * * *

# systemd timer équivalent
OnCalendar=*:0/5

Avantages des timers systemd :

  • RandomizedDelaySec=30 — ajoute un délai aléatoire pour éviter les pics de charge.
  • Persistent=true — exécute la tâche manquée après un redémarrage.
  • Logs intégrés dans journalctl -u mon-timer.service.

Si vous gérez des machines modernes (systemd), envisagez de migrer les tâches complexes vers des timers. Pour des tâches simples, cron reste parfaitement adapté.

9. Déboguer un cron job silencieux — checklist ultime

Quand une tâche cron ne s’exécute pas, aucune erreur n’apparaît par défaut. Voici les étapes de diagnostic :

  1. Vérifier les logs : grep CRON /var/log/syslog ou /var/log/cron. Cherchez la ligne correspondant à votre commande.
  2. Distinguer crontab utilisateur vs root : crontab -l (utilisateur courant) et sudo crontab -l (root) sont deux fichiers distincts.
  3. Rediriger les sorties : ajoutez >> /tmp/mon_job.log 2>&1 à la fin de la ligne pour capturer stdout et stderr.
  4. Vérifier les permissions : le script doit être exécutable (chmod +x script.sh) et accessible dans le PATH défini dans la crontab.
  5. Tester le PATH : lancez votre script dans un environnement minimal : env -i /bin/bash -l -c "/chemin/script.sh". Cela simule l’environnement cron.
  6. Activer MAILTO : MAILTO=vous@email.com en haut de la crontab. Toute sortie d’erreur sera envoyée par mail (si le service mail est configuré).
  7. Utiliser le générateur cron : notre outil de validation vérifie la syntaxe et affiche les prochaines exécutions, ce qui aide à repérer une erreur de planification.

FAQ

Comment écrire une expression cron toutes les 5 minutes ?

*/5 * * * * en cron standard. Chaque champ signifie : minute (toutes les 5), heure (toutes), jour du mois (tous), mois (tous), jour de la semaine (tous).

Quelle est la différence entre cron Linux et cron AWS ?

Linux utilise 5 champs, AWS CloudWatch (et Quartz) peut utiliser 6 champs avec les secondes. Notre générateur gère les deux modes.

Pourquoi ma tâche cron ne s'exécute pas ?

Vérifiez le PATH, les permissions, redirigez la sortie vers un fichier et consultez /var/log/syslog. Utilisez MAILTO pour recevoir les erreurs.

Comment éviter qu’une tâche cron se lance en double ?

Utilisez flock, run-one ou la propriété concurrencyPolicy: Forbid dans Kubernetes.
