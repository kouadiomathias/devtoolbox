Encodage et décodage d'URL : le guide complet pour développeurs (2026)

8 juin 2026 · 14 min de lecture · Par Mathias Kouadio

En 2021, une simple accolade non encodée dans un paramètre redirect_uri a permis une attaque open redirect sur un service OAuth très utilisé. Le jeton était valide, la vérification de domaine passait — mais l'URL n'avait jamais été correctement décodée avant validation. Un seul caractère, contournant silencieusement la sécurité. L'encodage d'URL fait partie de ces sujets qui semblent triviaux jusqu'à ce qu'un incident de production prouve le contraire.

Ce guide couvre tout ce que vous devez savoir : la différence exacte entre les trois fonctions d'encodage JavaScript, le bug de double-encodage qui casse silencieusement les APIs, + vs %20, des exemples de code dans cinq langages et les vulnérabilités de sécurité que permet un encodage incorrect. L'outil gratuit en ligne d'encodage/décodage d'URL fonctionne entièrement dans votre navigateur — aucune donnée ne quitte votre machine.

Table de décision — choisissez la bonne fonction selon votre situation

Passez directement ici si vous connaissez l'encodage d'URL et cherchez simplement la réponse pour votre situation actuelle.

Les trois fonctions d'encodage JavaScript — différences exactes

La plupart des articles les expliquent vaguement. Voici le comportement exact avec la même chaîne d'entrée pour rendre la comparaison sans ambiguïté :

const input = "bonjour le monde/chemin?clé=val&autre=a+b#ancre";

// encodeURI — préserve la structure de l'URL
encodeURI(input);
// "bonjour%20le%20monde/chemin?cl%C3%A9=val&autre=a+b#ancre"
// ✅ / ? & # = + préservés intentionnellement
// ⚠️  espace → %20, mais & et = ne sont PAS encodés

// encodeURIComponent — encode TOUT sauf les caractères alphanumériques
encodeURIComponent(input);
// "bonjour%20le%20monde%2Fchemin%3Fcl%C3%A9%3Dval%26autre%3Da%2Bb%23ancre"
// ✅ / ? & # = + TOUS encodés — correct pour les valeurs de paramètres

// escape() — NE PAS UTILISER, déprécié depuis ES3
escape(input);
// "bonjour%20le%20monde/chemin%3Fcl%E9%3Dval%26autre%3Da+b%23ancre"
// ❌ encode différemment, incompatible avec les normes modernes

N'utilisez jamais escape() dans du code moderne. Elle encode les caractères Unicode en Latin-1 (%E9 pour é) au lieu d'UTF-8 (%C3%A9), ce qui casse tout contenu non-ASCII. Elle échoue également complètement sur les emojis et les caractères au-dessus de U+00FF. Elle a été dépréciée en ES3 et n'existe aujourd'hui que pour la compatibilité des scripts hérités.

Le piège du double-encodage — le bug silencieux le plus courant

C'est le bug qui génère le plus de questions confuses sur Stack Overflow. Vous obtenez %2520 dans votre URL alors que vous attendiez %20. Voici exactement ce qui s'est passé et comment le corriger.

// Le scénario : vous encodez une chaîne déjà encodée
const original = "bonjour le monde";
const encoded = encodeURIComponent(original);
// "bonjour%20le%20monde"

// Bug : l'encoder une deuxième fois par erreur
const doubleEncoded = encodeURIComponent(encoded);
// "bonjour%2520le%2520monde"
// Le caractère % a lui-même été encodé en %25
// Donc %20 est devenu %2520

// Quand le serveur décode : il reçoit "bonjour%20le%20monde"
// pas "bonjour le monde" comme attendu
decodeURIComponent("bonjour%2520le%2520monde"); // "bonjour%20le%20monde" ← toujours encodé !

Détecter et corriger le double-encodage par programme

// Détecter si une chaîne est déjà encodée
function estDejaEncodee(str) {
  try {
    // Si le décodage change la chaîne, elle était encodée
    return str !== decodeURIComponent(str);
  } catch {
    // URI malformée — a des séquences % invalides
    return false;
  }
}

// Encoder de façon idempotente — encoder seulement si pas déjà encodé
function encodageSécurisé(str) {
  return estDejaEncodee(str) ? str : encodeURIComponent(str);
}

// Détecter le double-encodage dans une URL
// Le motif %25XX est un signe quasi-certain de double-encodage
const regexDoubleEncodage = /%25[0-9A-Fa-f]{2}/gi;

function aDoubleEncodage(url) {
  return regexDoubleEncodage.test(url);
}

console.log(aDoubleEncodage("bonjour%2520le%2520monde")); // true — double encodé
console.log(aDoubleEncodage("bonjour%20le%20monde"));   // false — correct
Cause racine : le double-encodage se produit généralement aux frontières des frameworks — un middleware encode une valeur, puis votre code l'encode à nouveau avant de l'insérer dans une URL. La solution consiste à identifier quelle couche est responsable de l'encodage et à supprimer le doublon.

La différence + vs %20 qui casse les intégrations d'API

Les deux représentent un espace dans une URL — mais ils proviennent de normes différentes et ne sont pas interchangeables dans tous les contextes. Les mélanger provoque des échecs silencieux dans les appels d'API.

// Norme application/x-www-form-urlencoded (formulaires HTML)
// Les espaces sont encodés en +
const formData = new URLSearchParams({ q: "bonjour le monde", filtre: "outils dev" });
console.log(formData.toString()); // "q=bonjour+le+monde&filtre=outils+dev"

// Norme RFC 3986 (encodeURIComponent)
// Les espaces sont encodés en %20
encodeURIComponent("bonjour le monde"); // "bonjour%20le%20monde"

// Le problème : mélanger les deux
const url1 = `https://api.exemple.com/recherche?q=${encodeURIComponent("bonjour le monde")}`;
// ✅ "https://api.exemple.com/recherche?q=bonjour%20le%20monde"
// La plupart des APIs acceptent cela — norme RFC 3986

const url2 = `https://api.exemple.com/recherche?q=${"bonjour le monde".replace(/ /g, "+")}`;
// ⚠️ "https://api.exemple.com/recherche?q=bonjour+le+monde"
// Certaines APIs exigent cela, d'autres ne décodent pas + comme un espace

// Règle : utilisez encodeURIComponent par défaut (%20)
// N'utilisez + que si la spécification de l'API l'exige explicitement
// ou lors de l'utilisation de URLSearchParams pour la soumission de formulaires

La manière sécurisée de construire des URLs en JavaScript

// ❌ Ne jamais construire d'URLs par concaténation de chaînes
const mauvaiseUrl = "https://api.exemple.com/recherche?q=" + entreeUtilisateur;
// Si entreeUtilisateur = "a&b" → casse la chaîne de requête
// Si entreeUtilisateur = "<script>" → XSS potentiel

// ✅ Toujours utiliser la classe URL
const url = new URL("https://api.exemple.com/recherche");
url.searchParams.set("q", entreeUtilisateur);          // Encodé automatiquement
url.searchParams.set("filtre", "prix<100");         // < → %3C
url.searchParams.set("tags", "a&b");                // & → %26
console.log(url.toString());
// "https://api.exemple.com/recherche?q=bonjour%20le%20monde&filtre=prix%3C100&tags=a%26b"

Encodage d'URL en 5 langages — prêt à copier-coller

Chaque langage a ses propres fonctions d'encodage et ses propres pièges. Voici la référence complète pour les tâches les plus courantes dans chaque écosystème.

Python

from urllib.parse import (
    quote, unquote,
    quote_plus, unquote_plus,
    urlencode, urlparse, urlunparse, parse_qs
)

# Encoder une valeur de paramètre (équivalent de encodeURIComponent)
quote("bonjour le monde/chemin", safe='')
# "bonjour%20le%20monde%2Fchemin"
# safe='' signifie tout encoder, y compris /

# Encoder pour les formulaires HTML (espace → +)
quote_plus("bonjour le monde")
# "bonjour+le+monde"

# Décoder
unquote("bonjour%20le%20monde")       # "bonjour le monde"
unquote_plus("bonjour+le+monde")      # "bonjour le monde"

# Encoder un dictionnaire de paramètres
params = {"q": "bonjour le monde", "page": 2, "filtre": "prix<100"}
urlencode(params)
# "q=bonjour+le+monde&page=2&filtre=prix%3C100"

# Construire une URL complète de façon sécurisée
from urllib.parse import urlencode, urlunparse

params = urlencode({"q": "bonjour le monde", "lang": "fr"})
url = urlunparse(("https", "api.exemple.com", "/recherche", "", params, ""))
# "https://api.exemple.com/recherche?q=bonjour+le+monde&lang=fr"

# Analyser une URL existante
parsed = urlparse("https://exemple.com/chemin?q=bonjour%20le%20monde#ancre")
print(parsed.scheme)    # "https"
print(parsed.netloc)    # "exemple.com"
print(parsed.path)      # "/chemin"
print(parsed.query)     # "q=bonjour%20le%20monde"
print(parsed.fragment)  # "ancre"

# Analyser les paramètres de requête
parse_qs("q=bonjour+le+monde&page=2")
# {'q': ['bonjour le monde'], 'page': ['2']}
Piège Python : quote() a safe='/' par défaut, ce qui signifie que les barres obliques ne sont PAS encodées. Passez toujours safe='' lors de l'encodage des valeurs de paramètres. Sinon, quote("chemin/vers/fichier") retourne "chemin/vers/fichier" inchangé.

Encodage d'URL et sécurité — les vulnérabilités à connaître

Un mauvais encodage d'URL n'est pas qu'un problème de formatage. Il permet trois catégories d'attaques réelles. Les comprendre changera votre façon d'écrire le code de validation.

Open Redirect via encodage incomplet

// L'attaque : encoder seulement une partie d'une URL malveillante
// La validation d'URL vérifie le domaine — mais sur la version encodée

// Validation côté serveur (INCORRECT)
function estRedirectionValide(url) {
  return url.startsWith("https://confiance.com");
}

// URL d'attaque 1 : deux-points encodé
const urlAttaque1 = "https%3A//malveillant.com/phishing";
estRedirectionValide(urlAttaque1); // false ← détecté

// URL d'attaque 2 : la vérification de domaine passe, puis le serveur décode
const urlAttaque2 = "https://confiance.com.malveillant.com/phishing";
estRedirectionValide(urlAttaque2); // true ← CONTOURNÉ

// URL d'attaque 3 : barre oblique doublement encodée
const urlAttaque3 = "https://confiance.com%2F@malveillant.com";
// Après décodage : "https://confiance.com/@malveillant.com" → redirige vers malveillant.com

// Approche CORRECTE : toujours décoder PUIS valider
function redirectionSecurisee(urlEncodee) {
  let decodee;
  try {
    decodee = decodeURIComponent(urlEncodee);
  } catch {
    return null; // URL malformée — rejeter
  }
  
  try {
    const parsed = new URL(decodee);
    const domainesAutorises = ["confiance.com", "api.confiance.com"];
    
    if (!domainesAutorises.includes(parsed.hostname)) {
      return null; // Domaine non autorisé
    }
    return decodee;
  } catch {
    return null; // Pas une URL valide
  }
}

XSS via fragment d'URL non encodé

// Bug XSS classique : utiliser le fragment (#) dans du HTML sans encodage
// Le fragment n'est jamais envoyé au serveur — le navigateur le gère

// ❌ Code vulnérable
const fragment = location.hash.slice(1); // Pas d'encodage côté serveur
document.getElementById('titre').innerHTML = fragment; // ← XSS

// URL d'attaque
// https://app.exemple.com/page#<img src=x onerror=alert(document.cookie)>

// ✅ Code sécurisé
const fragment = decodeURIComponent(location.hash.slice(1));
document.getElementById('titre').textContent = fragment;
//                                ^^^^^^^^^^^^^
//                         textContent au lieu de innerHTML
//                         — traite le contenu comme du texte, pas du HTML

Injection de paramètres via chaînes de requête non validées

// ❌ Construire une URL backend avec une entrée utilisateur non validée
app.get('/proxy', (req, res) => {
  const cible = req.query.url;
  // Si url = "https://api.exemple.com/data?admin=true"
  // L'attaquant ajoute ses propres paramètres

  const urlBackend = `https://api-interne.com/recuperer?url=${cible}`;
  // ← Risque SSRF (Server-Side Request Forgery)
});

// ✅ Toujours valider la valeur décodée contre une liste d'autorisation
app.get('/proxy', (req, res) => {
  const urlBrute = req.query.url;
  
  let decodee;
  try {
    decodee = decodeURIComponent(urlBrute);
  } catch {
    return res.status(400).json({ erreur: "Encodage d'URL invalide" });
  }
  
  const ORIGINES_AUTORISEES = [
    "https://api.confiance.com",
    "https://donnees.confiance.com"
  ];
  
  const estAutorise = ORIGINES_AUTORISEES.some(origine => decodee.startsWith(origine));
  if (!estAutorise) {
    return res.status(403).json({ erreur: "URL non autorisée" });
  }
  
  // Sûr de continuer
});

Checklist de sécurité — audit d'encodage d'URL

  • Décoder AVANT de valider — ne jamais valider une URL encodée
  • Ne jamais injecter de paramètres d'URL non encodés dans du HTML (XSS)
  • Utiliser des requêtes paramétrées même avec des valeurs décodées (SQLi)
  • Valider les URLs de redirection contre une liste d'autorisation après décodage complet (Open Redirect)
  • Vérifier la présence de %25XX dans les logs — signe probable de double-encodage
  • Utiliser la classe native URL — ne jamais concaténer des URLs manuellement
  • Tester toutes les entrées avec : & = + ? # / : @ % [ ]
  • Gérer URIError de decodeURIComponent() — les séquences invalides ne doivent pas planter l'application

Cas limites que notre outils gère parfaitement

Unicode, emojis et caractères accentués

// Emoji → séquences UTF-8 multi-octets
encodeURIComponent("🔧");
// "%F0%9F%94%A7"  — 4 octets, chacun → %XX

encodeURIComponent("🔧 répare-le");
// "%F0%9F%94%A7%20r%C3%A9pare-le"

// Caractères accentués — important pour les marchés FR/ES/DE
encodeURIComponent("déjà vu");
// "d%C3%A9j%C3%A0%20vu"
// é → %C3%A9 (UTF-8, correct)
// PAS %E9 (Latin-1, produit par escape() déprécié)

// Décoder à la fois UTF-8 et Latin-1 pour la compatibilité héritée
function decodageRobuste(str) {
  try {
    return decodeURIComponent(str); // Gère UTF-8 %C3%A9
  } catch {
    try {
      // Fallback : unescape gère Latin-1 %E9 (déprécié mais utile ici)
      return unescape(str);
    } catch {
      return str; // Retourne inchangé si tout le décodage échoue
    }
  }
}

Séquences de pourcentage malformées

// decodeURIComponent lève une exception sur les séquences invalides
try {
  decodeURIComponent("bonjour%XYle%20monde"); // %XY n'est pas un hexadécimal valide
} catch (e) {
  console.error(e.name);    // "URIError"
  console.error(e.message); // "URI malformed"
}

// Décodage sécurisé qui ne lève jamais d'exception
function decodageSecurise(str) {
  try {
    return decodeURIComponent(str);
  } catch {
    // Retourne la chaîne originale si elle ne peut pas être décodée
    // C'est mieux que de planter l'application
    return str;
  }
}

// Notre outil en ligne affiche l'erreur clairement sans planter
decodageSecurise("bonjour%XYle%20monde"); // "bonjour%XYle%20monde" (inchangé)

URLs déjà partiellement encodées

// Entrée mixte — partiellement encodée
const mixte = "bonjour%20le%20monde & café";

// Correct — décoder d'abord les parties encodées
decodeURIComponent("bonjour%20le%20monde & caf%C3%A9");
// "bonjour le monde & café" ✅

// Puis ré-encoder proprement si nécessaire
encodeURIComponent("bonjour le monde & café");
// "bonjour%20le%20monde%20%26%20caf%C3%A9"

// Détecter l'encodage partiel avant traitement
function statutEncodage(str) {
  const aPourcent = /%[0-9A-Fa-f]{2}/.test(str);
  const aNonEncode = /[&=+?#\[\] ]/.test(str);
  
  if (aPourcent && aNonEncode) return 'partiel';
  if (aPourcent) return 'encode';
  if (aNonEncode) return 'brut';
  return 'propre';
}

Décoder et encoder des URLs dans le terminal

Ces commandes finissent dans les dotfiles, les scripts shell et les Gists d'équipe. Ce sont celles que vous utilisez pendant le débogage sans vouloir ouvrir un navigateur.

# Python — disponible sur tous les systèmes, gère correctement l'UTF-8
python3 -c "import urllib.parse; print(urllib.parse.unquote('bonjour%20le%20monde%2Fchemin'))"
# bonjour le monde/chemin

# Encoder
python3 -c "import urllib.parse; print(urllib.parse.quote('bonjour le monde/chemin', safe=''))"
# bonjour%20le%20monde%2Fchemin

# Node.js
node -e "console.log(decodeURIComponent('bonjour%20le%20monde%2Fchemin'))"
# bonjour le monde/chemin

node -e "console.log(encodeURIComponent('bonjour le monde/chemin'))"
# bonjour%20le%20monde%2Fchemin

# curl — encode les paramètres automatiquement avec --data-urlencode
curl -G "https://api.exemple.com/recherche" \
  --data-urlencode "q=bonjour le monde" \
  --data-urlencode "filtre=prix<100"
# → GET https://api.exemple.com/recherche?q=bonjour+le+monde&filtre=prix%3C100

# Perl (disponible sur la plupart des Linux/macOS)
perl -MURI::Escape -e 'print uri_unescape("bonjour%20le%20monde%2Fchemin"), "\n"'
# bonjour le monde/chemin

# Pure bash (pour les cas simples)
printf '%b\n' "${1//%/\\x}"
# Remplace %XX par le caractère réel (basique, pas d'UTF-8)

Foire Aux Questions

Quelle est la différence entre encodeURI et encodeURIComponent ?

encodeURI est conçu pour encoder une URL complète et préserve tous les caractères structurellement significatifs dans une URL : /, ?, &, #, =, :, @. Utilisez-le lorsque vous avez une URL déjà formée et que vous voulez simplement garantir sa validité. encodeURIComponent encode tout sauf les caractères alphanumériques et l'ensemble sécurisé - _ . ! ~ * ' ( ). Utilisez-le pour encoder des valeurs de paramètres individuelles avant de les insérer dans une URL.

Pourquoi mon URL contient-elle %2520 au lieu de %20 ?

C'est un bug de double-encodage. Le caractère % a lui-même été encodé en %25, transformant %20 en %2520. Cela se produit lorsqu'une valeur déjà encodée (comme bonjour%20le%20monde) est encodée à nouveau. La solution : trouvez quelle couche encode la valeur deux fois et supprimez le doublon. Utilisez le modèle encodageSécurisé() de cet article pour encoder de façon idempotente, ou le détecteur aDoubleEncodage() pour trouver les requêtes affectées dans vos logs.

Quelle est la différence entre + et %20 dans une URL ?

Les deux représentent un espace mais proviennent de normes différentes. L'encodage + vient de la norme application/x-www-form-urlencoded utilisée dans les formulaires HTML — c'est ce que produisent URLSearchParams et la plupart des processeurs de formulaires. L'encodage %20 suit la RFC 3986, la norme générale des URL — c'est ce que produit encodeURIComponent(). La plupart des serveurs acceptent les deux, mais certaines implémentations strictes n'en acceptent qu'un seul. En cas de doute, utilisez %20 (encodeURIComponent). N'utilisez + que si une API le spécifie explicitement ou lors de la soumission de formulaires HTML.

Dois-je valider une URL avant ou après décodage ?

Toujours décoder d'abord, puis valider. Valider une URL encodée est une erreur de sécurité critique. Un attaquant peut encoder des caractères pour contourner vos règles de validation, puis le serveur les décode en quelque chose de malveillant après que la vérification a réussi. Par exemple, https://confiance.com%2F@malveillant.com pourrait passer une vérification startsWith("https://confiance.com"), mais après décodage, cela devient une redirection vers malveillant.com. Utilisez toujours new URL(decodee) pour analyser l'URL, puis vérifiez parsed.hostname contre une liste d'autorisation.

Comment décoder une URL dans un terminal ?

Python : python3 -c "import urllib.parse; print(urllib.parse.unquote('bonjour%20le%20monde'))". Node.js : node -e "console.log(decodeURIComponent('bonjour%20le%20monde'))". Pour encoder : python3 -c "import urllib.parse; print(urllib.parse.quote('bonjour le monde', safe=''))".

L'encodage d'URL est-il sensible à la casse ?

Le pourcentage d'encodage lui-même est insensible à la casse : %2F et %2f sont équivalents selon la RFC 3986, et les navigateurs modernes normalisent en majuscules. Cependant, les valeurs décodées (ce que représentent les séquences de pourcentage) peuvent être sensibles à la casse — cela dépend de l'interprétation du chemin ou du paramètre par le serveur. Restez cohérent en utilisant l'encodage en majuscules (encodeURIComponent produit des majuscules par défaut).
