L'incident de production qui a tout déclenché

En 2015, des chercheurs ont découvert une vulnérabilité affectant une application bancaire majeure au Royaume-Uni. L'attaque était simple : un réseau publicitaire tiers avait été compromis et diffusait une page qui chargeait l'écran de confirmation de virement de la banque à l'intérieur d'un <iframe> transparent. Lorsqu'un utilisateur visitait la page malveillante, il pensait cliquer sur un bouton « Gagnez un iPhone ». Il cliquait en réalité sur « Confirmer le virement » sur une page bancaire qu'il ne pouvait pas voir.

La correction aurait tenu en deux lignes de configuration sur le serveur de la banque. Deux lignes qui auraient ajouté X-Frame-Options: DENY à chaque réponse. Elles n'étaient pas là. Ce schéma d'attaque — le clickjacking — est documenté par la fondation OWASP depuis 2008, et il fonctionne encore sur les sites qui n'ont pas ajouté cet en-tête.

Ce n'est pas un cas isolé. Les chercheurs en sécurité de SecurityHeaders.com effectuent des scans automatisés du top million de sites Alexa. Dans leur dernière analyse, environ 3 % seulement des sites web ont obtenu une note A ou A+. Les 97 % restants ne possédaient pas au moins un en-tête de sécurité critique. Sur la plupart d'entre eux, la mauvaise configuration n'était pas complexe — elle était simplement absente.

Les en-têtes de réponse HTTP sont invisibles en navigation normale. Ils n'apparaissent pas dans le code source de la page. Ils vivent dans l'onglet Réseau des DevTools, dans la sortie de curl -I, dans les fichiers de configuration Nginx. La plupart des développeurs ne les regardent jamais jusqu'à ce que quelque chose casse. Cet article change cela.

Vérifiez vos en-têtes maintenant : Collez vos en-têtes de réponse HTTP dans l'Analyseur d'en-têtes HTTP de DevToolbox et obtenez une note de sécurité instantanée avec des explications en langage clair. Tout fonctionne dans votre navigateur — rien n'est envoyé à un serveur.

Les 8 en-têtes de sécurité dont chaque serveur de production a besoin

1. Strict-Transport-Security (HSTS)

Strict-Transport-Security Critique

Indique au navigateur de toujours utiliser HTTPS pour ce domaine, même si l'utilisateur tape http:// ou clique sur un lien HTTP.

max-age=31536000; includeSubDomains; preload

Sans HSTS, voici ce qui se passe sur un Wi-Fi public : un utilisateur tape mabanque.com dans la barre d'adresse. Le navigateur envoie la première requête en HTTP — il n'a aucun moyen de savoir qu'il devrait utiliser HTTPS avant d'établir le contact. Un attaquant réseau sur le même réseau intercepte cette requête HTTP et ne la transmet jamais au vrai serveur. Il sert sa propre page à la place. C'est une attaque de rétrogradation de protocole, et HSTS l'empêche en faisant en sorte que le navigateur refuse d'établir des connexions HTTP vers le domaine après la première visite HTTPS réussie.

La valeur max-age est en secondes. 31536000 correspond à un an — la recommandation standard. includeSubDomains étend la protection à tous les sous-domaines. preload soumet votre domaine à la liste de préchargement HSTS maintenue par Google et incluse dans tous les principaux navigateurs, ce qui signifie que HTTPS est appliqué avant même la première visite — aucune première requête non protégée.

Une mise en garde importante : n'ajoutez pas preload à moins d'être certain que chaque sous-domaine sert du HTTPS valide et le fera toujours. Retirer un domaine de la liste de préchargement prend des semaines et nécessite une révision manuelle.

2. Content-Security-Policy (CSP)

Content-Security-Policy Critique

Définit une liste blanche de sources de confiance pour les scripts, styles, images et autres ressources. Le navigateur l'applique — tout ce qui n'est pas dans la liste est bloqué.

default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'self'

Le CSP est votre défense principale contre le Cross-Site Scripting (XSS). Si un attaquant parvient à injecter un script dans votre page — via un champ de commentaire, une réponse d'API rendue sans assainissement, ou une bibliothèque tierce compromise — un CSP bien configuré empêchera ce script de s'exécuter.

L'erreur la plus fréquente que je vois dans les projets réels est l'utilisation de unsafe-inline pour autoriser les scripts et styles en ligne. Cette seule directive désactive complètement la protection XSS pour les scripts. Si vous avez des scripts en ligne, l'approche moderne consiste à utiliser des nonces : une valeur aléatoire générée par requête, ajoutée à l'en-tête CSP et à chaque balise de script légitime.

# Avec CSP basé sur les nonces (recommandé pour les applis avec scripts en ligne)
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-{aleatoire_par_requete}'; object-src 'none'

Si vous ne savez pas par où commencer, utilisez d'abord Content-Security-Policy-Report-Only. Cela envoie des rapports de violation à une URL que vous spécifiez sans rien bloquer réellement. Cela vous permet de comprendre ce que votre politique casserait avant de l'appliquer.

# Tester sans appliquer — ajoutez report-uri pour collecter les violations
Content-Security-Policy-Report-Only: default-src 'self'; report-uri https://votreapp.com/csp-report

3. X-Frame-Options

X-Frame-Options Élevée

Empêche votre page d'être chargée dans un <iframe> sur une autre origine. La défense principale contre le clickjacking.

DENY

Le scénario de clickjacking est simple : un attaquant construit une page qui superpose un iframe transparent chargeant votre application. L'utilisateur interagit avec ce qu'il croit être la page de l'attaquant mais envoie en réalité des clics à votre application en dessous. Des actions sensibles — confirmer des transactions, changer des mots de passe, accorder des permissions — peuvent être déclenchées à l'insu de l'utilisateur.

DENY bloque complètement le cadrage, sur tous les domaines, y compris le vôtre. SAMEORIGIN autorise le cadrage uniquement depuis la même origine. Le paramètre obsolète ALLOW-FROM origin est ignoré par les navigateurs modernes — si vous devez autoriser une origine spécifique, utilisez plutôt la directive CSP frame-ancestors.

CSP vs X-Frame-Options : La directive CSP frame-ancestors est plus flexible et remplace X-Frame-Options dans les navigateurs qui la supportent. Mais X-Frame-Options couvre encore les navigateurs plus anciens. Conserver les deux coûte une ligne de configuration et ne coûte rien en performance.

4. X-Content-Type-Options

X-Content-Type-Options Élevée

Empêche le navigateur de deviner le type de contenu d'une réponse. La seule valeur valide est nosniff.

nosniff

Sans cet en-tête, les navigateurs essaient d'être utiles en « reniflant » le type de contenu lorsque le type déclaré semble incorrect. Un attaquant peut téléverser un fichier avec une extension .txt qui contient du JavaScript. Si une balise script pointe vers ce fichier, un navigateur plus ancien pourrait l'exécuter quand même, indépendamment de ce que le serveur déclare comme Content-Type. nosniff dit au navigateur : « Faites confiance au Content-Type que je déclare. Ne devinez pas. »

C'est l'un des en-têtes les plus simples à ajouter — une ligne de configuration, zéro effet secondaire si vos en-têtes Content-Type sont corrects. Il n'y a aucune raison valable de ne pas l'avoir en production.

5. Referrer-Policy

Referrer-Policy Moyenne

Contrôle la quantité d'informations d'URL incluse dans l'en-tête Referer lors de la navigation vers d'autres sites.

strict-origin-when-cross-origin

Considérez une URL comme https://votreapp.com/reinitialiser-motdepasse?jeton=abc123. Sans Referrer-Policy, si un utilisateur clique sur un lien de cette page vers un autre site, les logs du serveur de destination reçoivent l'URL complète — y compris le jeton de réinitialisation. Ce jeton est maintenant dans les logs d'un serveur tiers, potentiellement accessible à leurs ingénieurs, et peut-être collecté par des services d'analyse.

La même chose s'applique aux paramètres de recherche, aux identifiants internes, aux jetons de session passés dans les URL, et à toute autre donnée dans votre structure d'URL. strict-origin-when-cross-origin envoie l'URL complète aux destinations de même origine mais seulement l'origine (pas de chemin, pas de paramètres) aux destinations cross-origin. C'est le paramètre par défaut du navigateur depuis Chrome 85, mais le définir explicitement garantit un comportement cohérent sur tous les navigateurs.

6. Permissions-Policy

Permissions-Policy Moyenne

Restreint l'accès aux API du navigateur pour votre page — caméra, microphone, géolocalisation, paiement, et plus.

camera=(), microphone=(), geolocalisation=(), payment=()

Si votre application n'utilise pas la caméra, il n'y a aucune raison pour qu'un script sur votre page puisse y accéder — y compris une bibliothèque tierce compromise ou une charge XSS réussie. Permissions-Policy applique cela au niveau du navigateur. Même si un attaquant parvient à exécuter du code sur votre page, le navigateur refusera d'accorder l'accès à la caméra si votre politique dit camera=().

Cet en-tête a remplacé l'ancien en-tête Feature-Policy en 2020. Le successeur est supporté par tous les navigateurs modernes. Listez uniquement les API que votre application utilise réellement, et refusez explicitement tout le reste.

7. Cross-Origin Opener Policy (COOP)

Cross-Origin-Opener-Policy Moyenne

Isole votre contexte de navigation des fenêtres cross-origin, requis pour les minuteurs haute résolution et SharedArrayBuffer.

same-origin

Après la découverte de la vulnérabilité CPU Spectre en 2018, les navigateurs ont restreint l'accès aux API de chronométrage précis qui pouvaient être utilisées pour des attaques par canal auxiliaire. Pour réactiver des fonctionnalités comme SharedArrayBuffer et performance.now() haute résolution, votre site doit être dans un contexte isolé cross-origin — ce qui nécessite à la fois COOP et COEP (Cross-Origin-Embedder-Policy: require-corp). Si vous n'utilisez pas SharedArrayBuffer, COOP same-origin reste un bon en-tête de défense en profondeur qui empêche votre page d'être référencée par des fenêtres cross-origin.

8. Cross-Origin Resource Policy (CORP)

Cross-Origin-Resource-Policy Faible

Contrôle quelles origines peuvent charger cette ressource, protégeant contre les attaques de type Spectre via des lectures cross-origin.

same-origin (pour les ressources sensibles)

Cet en-tête est particulièrement pertinent pour les API et les ressources qui renvoient des données sensibles. same-origin signifie que seuls les documents de même origine peuvent charger la ressource. Utilisez cross-origin pour les ressources intentionnellement partagées (comme un CDN public). Définissez-le sur les API qui ne devraient jamais être intégrées ou chargées en cross-origin.

Cache-Control : chaque directive expliquée

Cache-Control est l'en-tête que la plupart des développeurs comprennent à moitié. Ils savent que no-cache signifie « ne pas mettre en cache » — sauf que ce n'est pas le cas. Se tromper sur ce point a des conséquences réelles : les utilisateurs voient des données périmées, des pages sensibles sont mises en cache par les CDN, ou votre application fait des requêtes inutiles au serveur d'origine à chaque chargement. Permettez-moi de détailler chaque directive correctement.

La confusion la plus courante : no-cache ≠ « pas de mise en cache ». Cela signifie « mettre en cache, mais toujours vérifier auprès du serveur d'abord ». Si vous ne voulez vraiment aucune mise en cache, utilisez no-store. La confusion vient du nom, qui était sans doute un mauvais choix des auteurs de la spécification HTTP.

Une stratégie pratique pour les applications web modernes : utilisez Cache-Control: public, max-age=31536000, immutable pour les assets statiques avec hash de contenu (votre bundler ajoute un hash au nom de fichier à chaque build), et Cache-Control: no-cache pour vos fichiers HTML (qui sont courts et peu coûteux à revalider, et doivent référencer les derniers assets).

# Nginx — différentes stratégies de cache par type de ressource

# Fichiers HTML : toujours revalider (ils référencent les derniers assets)
location ~* \.html$ {
    add_header Cache-Control "no-cache";
}

# Assets avec hash de contenu : mettre en cache indéfiniment, ne jamais revalider
location ~* \.(js|css|woff2)$ {
    add_header Cache-Control "public, max-age=31536000, immutable";
}

# Réponses d'API : privé, 60 secondes max
location /api/ {
    add_header Cache-Control "private, max-age=60";
}

# Pages sensibles : ne jamais stocker
location /compte/ {
    add_header Cache-Control "no-store";
}

Les en-têtes qui révèlent trop d'informations

Certains en-têtes ne devraient pas être là du tout. Ils sont ajoutés par défaut par les serveurs web et les frameworks, et ils fournissent gratuitement aux attaquants des renseignements sur votre infrastructure.

Server: Apache/2.4.51 (Ubuntu)

Cet en-tête de réponse indique à un attaquant exactement quoi chercher sur la NVD (National Vulnerability Database) ou sur Exploit-DB. Apache 2.4.49 avait une traversée de chemin CVE (CVE-2021-41773) qui a été activement exploitée dans les jours suivant sa divulgation. Un serveur qui affiche sa version dit essentiellement : « Voici une liste de mes vulnérabilités connues. »

Supprimer la version est trivial. Supprimer entièrement le nom du serveur est légèrement plus complexe mais possible sur tous les serveurs majeurs.

# Nginx — masquer le numéro de version
server_tokens off;
# Résultat : Server: nginx (version supprimée)

# Apache — httpd.conf ou .htaccess
ServerTokens Prod
ServerSignature Off
# Résultat : Server: Apache (seulement le nom du produit, pas de version)

X-Powered-By: PHP/8.1.0 ou X-Powered-By: Express

Ajouté par défaut par PHP et Express.js. Même problème : vous annoncez votre framework et sa version. Pour le supprimer :

# Express.js — désactiver complètement l'en-tête
app.disable('x-powered-by');

# Ou utiliser Helmet, qui gère cela et d'autres en-têtes de sécurité automatiquement
const helmet = require('helmet');
app.use(helmet());

# PHP — php.ini
expose_php = Off

X-AspNet-Version et X-AspNetMvc-Version

Ajoutés par défaut dans les anciennes configurations ASP.NET. Aucun de ces en-têtes n'a de fonction légitime en production.


<system.web>
  <httpRuntime enableVersionHeader="false" />
</system.web>

CORS : le mécanisme de sécurité le plus mal compris

Le Cross-Origin Resource Sharing (CORS) est le mécanisme qui contrôle quelles origines peuvent faire des requêtes cross-origin vers votre serveur. Il est appliqué par les navigateurs — pas par votre serveur lui-même. Comprendre cette distinction est crucial, car CORS ne protège pas votre serveur contre les requêtes serveur-à-serveur ou contre les clients non-navigateur comme curl.

L'erreur classique : wildcard + credentials

Un motif que j'ai vu en production à plusieurs reprises :

# ❌ Cette combinaison est rejetée par tous les navigateurs modernes
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true

La spécification CORS interdit explicitement cette combinaison. Lorsque Access-Control-Allow-Credentials: true, le navigateur n'inclura les cookies et les en-têtes d'autorisation dans la requête cross-origin que si le serveur spécifie une origine concrète — pas un wildcard. La logique : un wildcard signifie « n'importe quelle origine », ce qui est acceptable pour des ressources vraiment publiques, mais vous ne voudriez jamais envoyer vos cookies d'authentification à une requête provenant d'une origine inconnue.

Si vous avez besoin de credentials, vous devez renvoyer l'origine spécifique de la requête (après l'avoir validée par rapport à votre liste d'autorisation) :

# ✅ Correct pour les requêtes cross-origin avec credentials
# (Définir dynamiquement dans votre code serveur après validation de l'en-tête Origin)
Access-Control-Allow-Origin: https://app.votresociete.com
Access-Control-Allow-Credentials: true
Vary: Origin  # ← Indique aux caches que les réponses diffèrent selon l'Origine
Ajoutez toujours Vary: Origin quand vous définissez Access-Control-Allow-Origin dynamiquement. Sans cela, un CDN pourrait mettre en cache la réponse avec l'en-tête d'une origine et la servir à une origine différente — soit en exposant votre politique de credentials, soit en faisant échouer des requêtes légitimes.

Requêtes simples vs requêtes pré-vérifiées

Toutes les requêtes cross-origin ne déclenchent pas une requête OPTIONS preflight. Les requêtes « simples » — GET et POST avec des types de contenu standards comme application/x-www-form-urlencoded — passent directement. Les requêtes complexes — celles avec des en-têtes personnalisés, des types de contenu non standards (comme application/json), ou des méthodes autres que GET/POST — déclenchent d'abord un preflight. Le navigateur envoie une requête OPTIONS pour vérifier les permissions, et votre serveur doit répondre avec les en-têtes appropriés avant que la vraie requête ne soit envoyée.

Utilisez Access-Control-Max-Age pour mettre en cache les résultats preflight et éviter l'aller-retour supplémentaire à chaque requête. Une valeur de 86400 (un jour) est raisonnable pour la plupart des API.

Configurations serveur prêtes à copier

Ces configurations vous donnent une base de sécurité solide. Elles ne sont pas exhaustives — votre CSP en particulier doit être adapté aux sources réelles de votre application — mais elles couvrent les en-têtes les plus critiques et vous feront passer immédiatement d'un F à un B ou mieux.

Nginx — bloc complet d'en-têtes de sécurité

server {
    listen 443 ssl http2;
    server_name exemple.com;

    # ── En-têtes de sécurité ──────────────────────────────────

    # Forcer HTTPS pendant 1 an, y compris les sous-domaines
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # Empêcher le clickjacking
    add_header X-Frame-Options "DENY" always;

    # Empêcher le MIME sniffing
    add_header X-Content-Type-Options "nosniff" always;

    # Contrôler les informations de référent
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # Restreindre l'accès aux API du navigateur
    add_header Permissions-Policy "camera=(), microphone=(), geolocalisation=(), payment=()" always;

    # Content Security Policy — à adapter pour votre application
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; object-src 'none'; base-uri 'self';" always;

    # ── Masquer les informations du serveur ──────────────────
    server_tokens off;

    # ── Vos blocs SSL, location, etc. ────────────────────────
}

Apache — .htaccess ou VirtualHost

# Nécessite que mod_headers soit activé : a2enmod headers

<IfModule mod_headers.c>
    Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
    Header always set X-Frame-Options "DENY"
    Header always set X-Content-Type-Options "nosniff"
    Header always set Referrer-Policy "strict-origin-when-cross-origin"
    Header always set Permissions-Policy "camera=(), microphone=(), geolocalisation=()"
    Header always set Content-Security-Policy "default-src 'self'; object-src 'none';"
</IfModule>

# Masquer la version d'Apache
ServerTokens Prod
ServerSignature Off

Express.js (Node.js) — avec Helmet

const express = require('express');
const helmet = require('helmet');

const app = express();

// Helmet configure automatiquement tous les en-têtes de sécurité importants
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],
      objectSrc: ["'none'"],
      baseUri: ["'self'"],
    },
  },
  hsts: {
    maxAge: 31536000,
    includeSubDomains: true,
    preload: true,
  },
  referrerPolicy: { policy: 'strict-origin-when-cross-origin' },
  permissionsPolicy: {
    camera: [],
    microphone: [],
    geolocalisation: [],
  },
}));

// Supprimer X-Powered-By (Helmet le fait aussi, mais explicite c'est mieux)
app.disable('x-powered-by');

Foire aux questions

Dois-je mettre tous les en-têtes de sécurité sur chaque page ?

HSTS, X-Content-Type-Options, Referrer-Policy et Permissions-Policy doivent être définis globalement sur chaque réponse. CSP et X-Frame-Options sont fortement recommandés partout mais peuvent nécessiter une personnalisation par page si vous intégrez des iframes légitimes ou chargez des scripts tiers. Les en-têtes Server et X-Powered-By doivent être supprimés ou neutralisés globalement — ils n'ont aucune utilité pour l'utilisateur en production.

Un CSP strict va-t-il casser mon site web ?

Un CSP mal configuré peut bloquer des ressources légitimes — vos propres scripts, Google Analytics, les polices intégrées. La bonne approche consiste à utiliser d'abord Content-Security-Policy-Report-Only. Ce mode envoie des rapports de violation à une URL que vous spécifiez sans rien appliquer. Surveillez les rapports pendant une semaine, ajoutez les sources nécessaires à votre politique, puis passez à l'application. Évitez également unsafe-inline pour les scripts — utilisez des nonces à la place, qui autorisent des scripts en ligne légitimes spécifiques tout en bloquant ceux qui sont injectés.

X-Frame-Options est-il encore nécessaire si j'utilise CSP frame-ancestors ?

Strictement parlant, non — CSP frame-ancestors remplace X-Frame-Options dans les navigateurs qui supportent CSP Level 2 (tous les navigateurs modernes). Mais conserver les deux en-têtes offre une défense en profondeur pour les navigateurs plus anciens et les extensions de navigateur qui pourraient ne pas implémenter complètement CSP. Le coût est d'une ligne de configuration supplémentaire. Le bénéfice est une couverture légèrement plus large. À moins d'avoir une raison spécifique de le supprimer, gardez les deux.

Quelle est la différence entre no-cache et no-store ?

C'est la confusion la plus courante sur Cache-Control. no-cache signifie que le navigateur peut stocker la réponse localement mais doit vérifier auprès du serveur avant de la servir — en utilisant la validation ETag ou Last-Modified. Si le serveur dit que rien n'a changé, il renvoie 304 Not Modified et le navigateur sert la copie en cache. no-store signifie ne jamais stocker la réponse nulle part. Utilisez no-cache pour le contenu qui change fréquemment mais où la logique de cache est utile. Utilisez no-store pour le contenu vraiment sensible comme les sessions bancaires ou les données médicales personnelles.

Puis-je combiner Access-Control-Allow-Origin: * avec les credentials ?

Non, et le navigateur bloquera cette combinaison indépendamment de ce que votre serveur envoie. La spécification CORS exige que lorsque Access-Control-Allow-Credentials est true, l'en-tête Access-Control-Allow-Origin doit contenir une origine spécifique, pas un wildcard. C'est intentionnel : les credentials ne doivent jamais être envoyés en cross-origin à une origine inconnue arbitraire. Pour prendre en charge les requêtes cross-origin avec credentials, validez l'en-tête Origin de la requête par rapport à votre liste d'autorisation et renvoyez-le dans la réponse, avec un en-tête Vary: Origin pour éviter les problèmes de cache.

Comment vérifier mes en-têtes de sécurité actuels ?

Trois méthodes. La plus rapide : ouvrez les DevTools du navigateur, allez dans l'onglet Réseau, cliquez sur n'importe quelle requête vers votre serveur, et regardez la section En-têtes de réponse. Depuis le terminal : curl -I https://votresite.com affiche tous les en-têtes de réponse. Ou utilisez l'Analyseur d'en-têtes HTTP de DevToolbox — collez les en-têtes et obtenez une note de sécurité instantanée avec des explications pour chacun, ce qui manque, et quelle est la valeur recommandée.
