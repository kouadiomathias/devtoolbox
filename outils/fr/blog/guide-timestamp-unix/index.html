Le bug de production qui m'a fait découvrir ça

Il y a quelques années, je travaillais sur un tableau de bord qui récupérait des événements depuis une API tierce. L'API renvoyait les timestamps sous forme d'entiers — created_at: 1695132000 — et mon équipe frontend les affichait comme des dates. Pendant des semaines, chaque événement dans l'interface apparaissait comme "1er janvier 1970 à 00:28".

L'équipe backend stockait les timestamps Unix en secondes. Le frontend les traitait comme des millisecondes et les passait directement à new Date(1695132000). En JavaScript, le constructeur Date attend des millisecondes. 1 695 132 000 millisecondes depuis l'epoch correspondent à janvier 1970. 1 695 132 000 secondes correspondent à septembre 2023. Deux interprétations du même nombre, séparées par 53 ans.

La correction fut une seule multiplication : new Date(1695132000 * 1000). Le bug était présent depuis trois semaines avant que quelqu'un ne le remarque, simplement parce que personne n'avait regardé une date d'événement d'assez près. Cette erreur est si courante que la plupart des questions de débogage de timestamps sur Stack Overflow y ramènent.

Ce guide contient tout ce dont vous avez besoin pour ne plus jamais faire cette erreur, comprendre tous les formats de timestamp que vous rencontrerez dans des systèmes réels, et extraire des dates depuis des identifiants qui embarquent des timestamps — y compris UUID v1, UUID v7 et MongoDB ObjectIDs.

Comment détecter automatiquement secondes vs millisecondes

L'heuristique la plus fiable consiste à compter les chiffres. Un timestamp Unix en secondes pour toute date entre 2001 et 2286 est un nombre à 10 chiffres. Un timestamp en millisecondes pour la même période est un nombre à 13 chiffres. Cela fonctionne parce que l'epoch Unix progresse de manière prévisible — en secondes, nous n'atteindrons pas 11 chiffres avant le 20 novembre 2286.

# Python — heuristique avec bornes
def detecter_unite_timestamp(ts: int) -> str:
    """
    Retourne l'unité probable d'un timestamp Unix.
    Fonctionne pour toute date entre 2001 et 2286.
    """
    if 1_000_000_000 <= ts <= 9_999_999_999:
        return "secondes"        # 10 chiffres
    elif 1_000_000_000_000 <= ts <= 9_999_999_999_999:
        return "millisecondes"   # 13 chiffres
    elif 1_000_000_000_000_000 <= ts <= 9_999_999_999_999_999:
        return "microsecondes"   # 16 chiffres
    elif ts >= 1_000_000_000_000_000_000:
        return "nanosecondes"    # 19 chiffres (Go, Rust, pg_timestamp_ns de PostgreSQL)
    else:
        return "inconnue"

# Utilisation
print(detecter_unite_timestamp(1749376800))          # secondes
print(detecter_unite_timestamp(1749376800000))        # millisecondes
print(detecter_unite_timestamp(1749376800000000))     # microsecondes

Si vous construisez une API et voulez éviter cette ambiguïté, la solution correcte est d'utiliser des chaînes ISO 8601 pour les timestamps dans vos charges JSON. "2025-06-08T10:00:00Z" est sans ambiguïté, lisible par un humain sans conversion, et prise en charge nativement par toutes les bibliothèques de dates modernes. Le léger surcoût en taille de charge utile est presque toujours justifié par la clarté.

Attention aux valeurs par défaut des langages. Date.now() en JavaScript, System.currentTimeMillis() en Java et Time.now.to_i * 1000 en Ruby renvoient des millisecondes. time.time() en Python, les appels système Unix, EXTRACT(EPOCH FROM NOW()) de PostgreSQL et la plupart des pilotes de base de données renvoient des secondes. Lorsque vous mélangez des langages ou des services dans un même système, documentez l'unité utilisée et appliquez-la avec un alias de type ou un wrapper.

Fuseaux horaires : le piège du datetime naïf

Les bugs de fuseau horaire sont la classe la plus courante de problèmes liés aux dates dans les systèmes backend, et ils sont presque invisibles — ils produisent des valeurs erronées d'apparence plausible, pas des plantages. La cause racine est presque toujours la même : un datetime naïf traité comme s'il était UTC.

En Python, il existe deux types d'objets datetime. Un datetime naïf n'a aucune information de fuseau horaire. Un datetime conscient en a un. Le problème est que les datetimes naïfs n'annoncent pas leur ambiguïté — ils existent simplement, flottants, et votre code suppose qu'ils sont UTC parce que c'est l'intention. Mais datetime.fromtimestamp() sans argument tz crée un datetime naïf dans le fuseau horaire local du serveur, pas UTC.

import datetime, zoneinfo

ts = 1749376800  # 2025-06-08 10:00:00 UTC

# ❌ Bug : datetime naïf interprété dans le fuseau horaire local
# Sur un serveur réglé sur Europe/Paris (UTC+2), cela donne 12:00:00 local
# Si vous le sérialisez ensuite comme UTC, vous obtenez 12:00:00Z au lieu de 10:00:00Z
dt_naif = datetime.datetime.fromtimestamp(ts)
print(dt_naif)                    # 2025-06-08 12:00:00 (FAUX si vous attendez UTC)

# ✅ Correct : spécifiez toujours le fuseau horaire explicitement
dt_utc = datetime.datetime.fromtimestamp(ts, tz=datetime.timezone.utc)
print(dt_utc)                      # 2025-06-08 10:00:00+00:00 ✓

# ✅ Convertir vers un fuseau spécifique après avoir obtenu UTC
paris = zoneinfo.ZoneInfo("Europe/Paris")
dt_paris = dt_utc.astimezone(paris)
print(dt_paris)                    # 2025-06-08 12:00:00+02:00 ✓

# ✅ Retour vers le timestamp Unix depuis un datetime conscient
print(int(dt_utc.timestamp()))     # 1749376800 ✓

Le même piège existe en JavaScript. new Date(timestamp) crée un timestamp UTC en interne, mais les méthodes comme getHours() renvoient des valeurs dans le fuseau horaire local du navigateur ou du serveur. Si vous exécutez Node.js sur un serveur dans un fuseau non-UTC, new Date(ts * 1000).getHours() ne vous donne pas l'heure UTC.

const ts = 1749376800; // 2025-06-08 10:00:00 UTC

const d = new Date(ts * 1000);

// ❌ Ces méthodes dépendent du fuseau horaire système — faux si vous attendez UTC
console.log(d.getHours());               // Varie selon le fuseau horaire du système

// ✅ Celles-ci renvoient toujours des valeurs UTC, indépendamment du fuseau système
console.log(d.getUTCHours());            // 10 ✓
console.log(d.toISOString());            // "2025-06-08T10:00:00.000Z" ✓
console.log(d.toUTCString());            // "Sun, 08 Jun 2025 10:00:00 GMT" ✓

// ✅ Formatage avec fuseau horaire via Intl (moderne, recommandé)
const formateur = new Intl.DateTimeFormat('fr-FR', {
  timeZone: 'Europe/Paris',
  dateStyle: 'full',
  timeStyle: 'long'
});
console.log(formateur.format(d));
// "dimanche 8 juin 2025 à 12:00:00 heure avancée d'Europe centrale"

Heure d'été et l'heure ambiguë

Deux fois par an, les horloges changent. Quand les horloges reculent en automne, une heure est "répétée" — la même heure d'horloge se produit deux fois. Si vous stockez des événements en heure locale sans information de fuseau horaire, vous ne pouvez pas distinguer les deux 2:30 du matin. C'est pourquoi tout système sérieux stocke les timestamps en UTC (ou en timestamps Unix) et ne convertit en heure locale que pour l'affichage.

Règle d'or : Stockez en UTC, affichez en local. Utilisez toujours les noms de fuseaux IANA (Europe/Paris, America/New_York) plutôt que des décalages (+02:00) pour les fuseaux horaires destinés aux utilisateurs — les décalages ne tiennent pas compte des changements d'heure d'été.

Timestamps cachés dans les identifiants

Plusieurs formats d'identifiants largement utilisés embarquent un timestamp dans leur structure. C'est utile : vous pouvez déterminer quand un enregistrement a été créé directement depuis son ID, sans colonne created_at séparée. Mais cela signifie aussi que si vous voyez un UUID ou un ObjectID que vous ne reconnaissez pas, vous pouvez décoder quand il a été créé.

UUID version 1

L'UUID v1 encode un timestamp de 60 bits comptant les intervalles de 100 nanosecondes depuis le 15 octobre 1582 — date d'adoption du calendrier grégorien. Le timestamp est réparti sur trois champs non contigus dans l'UUID, son extraction nécessite donc de réassembler les morceaux.

Étant donné l'UUID 6ba7b810-9dad-11d1-80b4-00c04fd430c8 :

  • time_low = 6ba7b810 (8 premiers car. hex, 32 bits)
  • time_mid = 9dad (car. 9–12, 16 bits)
  • time_hi = 11d1 avec le quartet de version (1) masqué → 01d1 (12 bits)
# Python — extraire le timestamp Unix d'un UUID v1
import uuid

def uuid1_vers_unix_ms(uuid_str: str) -> int:
    """Extrait le timestamp Unix (ms) d'une chaîne UUID v1."""
    u = uuid.UUID(uuid_str)
    if u.version != 1:
        raise ValueError(f"UUID v1 attendu, reçu v{u.version}")

    # u.time est le timestamp 60 bits en intervalles de 100ns depuis 1582-10-15
    # Décalage grégorien : intervalles de 100ns entre 1582-10-15 et 1970-01-01
    DECALAGE_GREGORIEN = 0x01b21dd213814000

    unix_100ns = u.time - DECALAGE_GREGORIEN
    unix_ms = unix_100ns // 10_000  # Convertir 100ns en ms

    return unix_ms

exemple = "6ba7b810-9dad-11d1-80b4-00c04fd430c8"
ms = uuid1_vers_unix_ms(exemple)
from datetime import datetime, timezone
print(datetime.fromtimestamp(ms / 1000, tz=timezone.utc))
# 1997-02-03 17:26:46.009320+00:00

UUID version 7

L'UUID v7 (standardisé dans la RFC 9562, mai 2024) a adopté une approche bien plus simple : les 48 premiers bits encodent directement le timestamp Unix en millisecondes. Pas de décalage grégorien, pas de réassemblage de champs. Il suffit de lire les 12 premiers caractères hexadécimaux.

# Python — extraire le timestamp Unix d'un UUID v7
import uuid

def uuid7_vers_unix_ms(uuid_str: str) -> int:
    """Extrait le timestamp Unix (ms) d'une chaîne UUID v7."""
    u = uuid.UUID(uuid_str)
    if u.version != 7:
        raise ValueError(f"UUID v7 attendu, reçu v{u.version}")

    # Les 48 premiers bits (12 car. hex) = timestamp Unix en millisecondes
    # uuid.int est l'entier 128 bits complet ; décaler à droite de 80 pour obtenir les 48 bits supérieurs
    return u.int >> 80

exemple = "018f4d1a-3b2c-7def-9abc-123456789abc"
ms = uuid7_vers_unix_ms(exemple)
from datetime import datetime, timezone
print(datetime.fromtimestamp(ms / 1000, tz=timezone.utc))

# JavaScript — encore plus simple
const uuid7 = "018f4d1a-3b2c-7def-9abc-123456789abc";
const hex48 = uuid7.replace(/-/g, '').slice(0, 12); // 12 premiers car. hex
const unixMs = parseInt(hex48, 16);
console.log(new Date(unixMs).toISOString());

MongoDB ObjectID

Un ObjectID MongoDB est une chaîne hexadécimale de 24 caractères. Les 4 premiers octets (8 caractères hex) encodent le timestamp Unix de création en secondes. Contrairement à l'UUID v7, il s'agit de secondes — pas de millisecondes — donc la résolution est de 1 seconde.

# Python — extraire le timestamp de création d'un ObjectID MongoDB
def objectid_vers_unix_s(object_id: str) -> int:
    """Extrait le timestamp Unix (secondes) d'un ObjectID MongoDB."""
    if len(object_id) != 24:
        raise ValueError("L'ObjectID doit comporter exactement 24 caractères hexadécimaux")
    # 8 premiers car. hex = 4 premiers octets = timestamp Unix en secondes
    return int(object_id[:8], 16)

oid = "507f1f77bcf86cd799439011"
ts = objectid_vers_unix_s(oid)
from datetime import datetime, timezone
print(datetime.fromtimestamp(ts, tz=timezone.utc))
# 2012-10-16 21:10:47+00:00

# JavaScript
const oid = "507f1f77bcf86cd799439011";
const unixSeconds = parseInt(oid.substring(0, 8), 16);
console.log(new Date(unixSeconds * 1000).toISOString());
// "2012-10-16T21:10:47.000Z"

package main

import (
    "fmt"
    "time"
)

func main() {
    // ── Timestamps actuels ─────────────────────────────────────
    maintenantUnixS  := time.Now().Unix()      // int64, secondes
    maintenantUnixMs := time.Now().UnixMilli() // int64, millisecondes (Go 1.17+)
    maintenantUnixNs := time.Now().UnixNano()  // int64, nanosecondes

    // ── Secondes Unix → time.Time ──────────────────────────────
    ts := int64(1749376800)
    t  := time.Unix(ts, 0).UTC()   // Toujours spécifier UTC explicitement

    // ── Formater en chaînes standards ───────────────────────────
    fmt.Println(t.Format(time.RFC3339))     // "2025-06-08T10:00:00Z"
    fmt.Println(t.Format(time.RFC3339Nano)) // "2025-06-08T10:00:00.000000000Z"
    fmt.Println(t.Format(time.RFC1123Z))    // "Sun, 08 Jun 2025 10:00:00 +0000"
    fmt.Println(t.Format("2006-01-02 15:04:05"))  // Format personnalisé (date de référence de Go)

    // ── Avec fuseau horaire ─────────────────────────────────────────
    paris, _ := time.LoadLocation("Europe/Paris")
    tParis   := t.In(paris)
    fmt.Println(tParis.Format(time.RFC3339)) // "2025-06-08T12:00:00+02:00"

    // ── time.Time → timestamp Unix ────────────────────────────
    d := time.Date(2025, 6, 8, 10, 0, 0, 0, time.UTC)
    fmt.Println(d.Unix())    // 1749376800

    // ── Windows FILETIME ──────────────────────────────────────
    filetime := (ts + 11_644_473_600) * 10_000_000
    fmt.Printf("FILETIME: %d\n", filetime)

    _ = maintenantUnixMs
    _ = maintenantUnixNs
    _ = maintenantUnixS
}
import java.time.*;
import java.time.format.*;

public class ExemplesTimestamp {
    public static void main(String[] args) {
        // ── Timestamps actuels ─────────────────────────────────
        long maintenantMs  = System.currentTimeMillis();  // millisecondes
        long maintenantS   = Instant.now().getEpochSecond(); // secondes

        // ── Secondes Unix → Instant ─────────────────────────────
        long ts = 1749376800L;
        Instant instant = Instant.ofEpochSecond(ts);

        // ── Formater en ISO 8601 ─────────────────────────────────
        System.out.println(instant.toString());
        // "2025-06-08T10:00:00Z"

        // ── Formatage avec fuseau horaire ──────────────────────────
        ZoneId paris = ZoneId.of("Europe/Paris");
        ZonedDateTime zdt = instant.atZone(paris);
        System.out.println(zdt.format(DateTimeFormatter.RFC_1123_DATE_TIME));
        // "Sun, 8 Jun 2025 12:00:00 +0200"

        // ── Format personnalisé ──────────────────────────────────────
        DateTimeFormatter fmt = DateTimeFormatter
            .ofPattern("yyyy-MM-dd HH:mm:ss z")
            .withZone(paris);
        System.out.println(fmt.format(instant));
        // "2025-06-08 12:00:00 CEST"

        // ── LocalDateTime → timestamp Unix (toujours spécifier la zone) ──
        LocalDateTime ldt = LocalDateTime.of(2025, 6, 8, 10, 0, 0);
        long unix = ldt.toEpochSecond(ZoneOffset.UTC); // 1749376800

        // ── Windows FILETIME ─────────────────────────────────────
        long filetime = (ts + 11_644_473_600L) * 10_000_000L;
        System.out.printf("FILETIME: %d%n", filetime);

        // ── Depuis FILETIME vers Instant ─────────────────────
        Instant depuisFiletime = DateTimeOffset.fromFileTime(filetime).toInstant();
        // (En termes .NET ; en Java, inverser manuellement la formule)

        System.out.println(maintenantS);
        System.out.println(maintenantMs);
    }
}

Le problème de l'an 2038

Le 19 janvier 2038 à 03:14:07 UTC, les timestamps Unix stockés sur 32 bits signés débordent. La valeur maximale d'un entier signé 32 bits est 2 147 483 647. Après cela, il reboucle vers un grand nombre négatif — ce qui correspond au 13 décembre 1901.

Chronologie du timestamp Unix (entier signé 32 bits)
1970 → 2025 (sûr)
Débordement 2038 ⚠️
Valeur max : 2 147 483 647 = 19 janvier 2038 à 03:14:07 UTC

Vous pouvez démontrer le débordement sur n'importe quel système 32 bits :

/* C — Démonstration du débordement de l'an 2038 */
#include <stdio.h>
#include <time.h>
#include <stdint.h>

int main() {
    // Valeur maximale d'un entier signé 32 bits
    int32_t max32 = 2147483647;

    // Convertir en struct tm
    time_t t = (time_t)max32;
    struct tm *tm_info = gmtime(&t);
    printf("Timestamp max 32 bits : %s", asctime(tm_info));
    // "Tue Jan 19 03:14:07 2038"

    // Une seconde plus tard : débordement
    int32_t overflow = max32 + 1;  // Rebouble vers -2147483648
    time_t t2 = (time_t)overflow;
    struct tm *tm_overflow = gmtime(&t2);
    printf("Après débordement : %s", asctime(tm_overflow));
    // "Fri Dec 13 20:45:52 1901"

    return 0;
}

Qui est encore concerné en 2026 ?

La plupart des logiciels modernes tournent sur des systèmes 64 bits et utilisent time_t 64 bits, qui ne débordera pas avant l'an 292 277 026 596 — bien au-delà de toute préoccupation pratique. Mais le problème persiste dans des contextes spécifiques :

  • Systèmes embarqués — microcontrôleurs, systèmes de contrôle industriel, ECU automobiles et appareils IoT utilisent souvent des architectures 32 bits avec des timestamps 32 bits gravés dans le firmware.
  • Bases de données anciennes — certaines installations MySQL plus anciennes utilisent des colonnes TIMESTAMP 32 bits, qui refusent les dates après le 19 janvier 2038. Utilisez DATETIME à la place.
  • Linux 32 bits — le noyau Linux a ajouté le support de time_t 64 bits pour les architectures 32 bits dans la version 5.6 (mars 2020). Les distributions basées sur des noyaux plus anciens ont encore le problème.
  • Systèmes de fichiers — FAT32 stocke les dates de modification dans un champ 32 bits. Les fichiers ne seront pas directement corrompus, mais certaines opérations sur les timestamps peuvent échouer après 2038.
# Vérifier si votre système utilise time_t 64 bits
python3 -c "import struct; print(struct.calcsize('l') * 8, 'bit time_t')"
# 64 bit time_t  → sûr
# 32 bit time_t  → vulnérable

# Vérifier en C
#include <time.h>
#include <stdio.h>
int main() {
    printf("sizeof(time_t) = %zu octets (%zu bits)\n",
           sizeof(time_t), sizeof(time_t) * 8);
    return 0;
}

Foire aux questions

Comment savoir si un timestamp Unix est en secondes ou en millisecondes ?

Comptez les chiffres. Un nombre à 10 chiffres est presque certainement des secondes Unix — cela couvre toute date entre septembre 2001 (1 000 000 000) et novembre 2286 (9 999 999 999). Un nombre à 13 chiffres est des millisecondes. La confusion existe parce que Date.now() en JavaScript et System.currentTimeMillis() en Java renvoient des millisecondes, tandis que time.time() en Python et la plupart des appels système Unix renvoient des secondes. Si vous contrôlez l'API, documentez explicitement l'unité et envisagez d'utiliser des chaînes ISO 8601 à la place — elles sont sans ambiguïté.

Qu'est-ce que l'epoch Unix ?

L'epoch Unix est le 1er janvier 1970 à 00:00:00 UTC — le point de référence à partir duquel les timestamps Unix sont comptés. Il a été choisi par les développeurs originaux d'UNIX parce que c'était une date récente et ronde qui précédait les premières installations commerciales d'UNIX. Ce choix n'a pas de signification profonde ; c'était une décision pratique. La valeur exacte a été formalisée dans le standard POSIX (IEEE Std 1003.1).

Puis-je extraire une date d'un UUID v1 ?

Oui. L'UUID v1 (défini dans la RFC 4122) encode un timestamp de 60 bits en intervalles de 100 nanosecondes depuis le 15 octobre 1582. Le timestamp est stocké dans trois champs non contigus (time_low, time_mid, time_hi_and_version), vous devez donc les réassembler et soustraire le décalage grégorien (122 192 928 000 000 000 en unités de 100ns) pour obtenir le timestamp Unix. Le module uuid de Python expose cela via uuid.UUID(s).time. L'extracteur MAC UUID v1 de DevToolbox fait cela pour vous et identifie également si l'adresse MAC est réelle ou générée aléatoirement.

Qu'est-ce que le Windows FILETIME et comment le convertir ?

Le FILETIME Windows est un entier 64 bits comptant les intervalles de 100 nanosecondes depuis le 1er janvier 1601 à 00:00:00 UTC. Il est utilisé dans toute l'API Windows (FindFirstFile, GetFileTime), .NET DateTime, LDAP et Active Directory. La constante 11 644 473 600 est le nombre de secondes entre le 1er janvier 1601 et le 1er janvier 1970. Conversion depuis des secondes Unix : FILETIME = (unixSeconds + 11644473600) × 10 000 000. En .NET : DateTime.FromFileTimeUtc(filetime) ou DateTimeOffset.FromUnixTimeSeconds(unixSeconds).

Qu'est-ce que le problème de l'an 2038 ?

Sur les systèmes 32 bits qui stockent les timestamps Unix comme des entiers signés de 32 bits, la valeur maximale représentable est 2 147 483 647, correspondant au 19 janvier 2038 à 03:14:07 UTC. Après cet instant, l'entier reboucle vers un grand nombre négatif (−2 147 483 648), qui représente le 13 décembre 1901. Les systèmes 64 bits modernes ne sont pas affectés — leur timestamp maximal est l'an 292 277 026 596. La zone à risque concerne les systèmes embarqués, les bases de données anciennes et les builds Linux 32 bits.

Pourquoi les objets datetime de Python causent-ils des bugs de fuseau horaire ?

Python a deux types d'objets datetime : naïf (sans information de fuseau) et conscient (avec fuseau). datetime.fromtimestamp(ts) sans argument tz crée un datetime naïf dans le fuseau horaire local du serveur — pas UTC. Si votre serveur est en UTC+2 et que vous sérialisez ce datetime naïf comme UTC, vous obtenez une heure décalée de deux heures. La solution est d'être toujours explicite : datetime.fromtimestamp(ts, tz=datetime.timezone.utc). Pour les conversions de fuseau horaire, utilisez zoneinfo.ZoneInfo (Python 3.9+) avec les noms de fuseaux IANA plutôt que des décalages fixes, qui ne tiennent pas compte des changements d'heure d'été.
