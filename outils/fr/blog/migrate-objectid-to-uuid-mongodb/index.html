Migrer de MongoDB ObjectID vers UUID dans une application Node.js — Guide complet

Publié le 6 juin 2026 · Mis à jour le 8 juin 2026 par Mathias Kouadio — 12 min de lecture

Votre application MongoDB tourne en production depuis des mois. Les ObjectID font le travail, mais vous commencez à ressentir leurs limites. Vous prévoyez d'ajouter PostgreSQL, vous exposez vos IDs dans une API REST, ou vous voulez simplement des identifiants plus standards. Migrer vers UUID est la solution, mais comment faire sans casser votre base existante ?

Dans cet article, je vous guide à travers une migration par phases avec des scripts prêts pour la production, la gestion de la double écriture, la surveillance, la validation et un plan de rollback en cas de problème.

Pourquoi abandonner ObjectID ?

L'ObjectID est propre à MongoDB. Il est compact (12 octets) et contient un timestamp, mais il n'est pas facilement compatible avec d'autres SGBD ou avec les attentes des développeurs frontend. Un UUID, lui, est universel, aléatoire (v4) ou triable (v7), et bien mieux adapté aux API publiques.

Trois raisons concrètes de migrer :

  • Vous voulez exposer vos IDs dans des URLs sans qu'ils trahissent la date de création ou la séquence.
  • Vous ajoutez PostgreSQL ou un autre SGBD qui ne comprend pas l'ObjectID.
  • Vous souhaitez standardiser vos identifiants sur l'ensemble de votre architecture.

Phase 1 : ajouter le champ UUID sans toucher à _id

La règle d'or : ne supprimez jamais _id. MongoDB en a besoin comme clé primaire. Nous allons ajouter un nouveau champ uuid à chaque document. Si vous utilisez Mongoose, commencez par mettre à jour le schéma :

const userSchema = new mongoose.Schema({
  uuid: { type: String, unique: true, sparse: true },
  name: String,
  email: String
});

Le flag sparse: true permet d'indexer uniquement les documents qui possèdent déjà le champ. Exécutez ensuite le script optimisé avec bulkWrite — c'est 10 à 50 fois plus rapide qu'une mise à jour document par document :

const { v4: uuidv4 } = require('uuid');
const BATCH_SIZE = 1000;

async function migrateUUIDs(db) {
  let processed = 0;
  
  while (true) {
    const docs = await db.collection('users')
      .find({ uuid: { $exists: false } })
      .limit(BATCH_SIZE)
      .toArray();
    
    if (docs.length === 0) break;
    
    const operations = docs.map(doc => ({
      updateOne: {
        filter: { _id: doc._id },
        update: { $set: { uuid: uuidv4() } }
      }
    }));
    
    await db.collection('users').bulkWrite(operations, { ordered: false });
    processed += docs.length;
    console.log(`Documents migrés : ${processed}`);
  }
  
  console.log(`Migration terminée : ${processed} au total`);
}

Phase 2 : créer l'index unique

Une fois tous les documents pourvus d'un UUID, créez un index unique :

await db.collection('users').createIndex({ uuid: 1 }, { unique: true });

Phase 3 : mettre à jour les références

Le plus délicat : les collections liées. Si un document orders référence un user_id sous forme d'ObjectID, vous devez migrer ces références. La méthode la plus sûre est d'ajouter un champ user_uuid tout en conservant user_id pendant la transition.

// Pour chaque commande, retrouver l'UUID du user correspondant
const orders = db.collection('orders').find({ user_uuid: { $exists: false } });
while (await orders.hasNext()) {
  const order = await orders.next();
  const user = await db.collection('users').findOne({ _id: order.user_id });
  if (user) {
    await db.collection('orders').updateOne(
      { _id: order._id },
      { $set: { user_uuid: user.uuid } }
    );
  }
}
⚠ Important : Testez cette étape sur un environnement de staging avec un volume de données réaliste. Pour les grandes collections, découpez cette opération en lots comme à la Phase 1.

Surveiller la progression en temps réel

Suivez le nombre de documents migrés pendant l'exécution du script :

async function checkProgress(db) {
  const total = await db.collection('users').countDocuments();
  const migrated = await db.collection('users')
    .countDocuments({ uuid: { $exists: true } });
  const remaining = total - migrated;
  const pct = ((migrated / total) * 100).toFixed(1);
  
  console.log(`Total : ${total} | Migrés : ${migrated} | Restants : ${remaining} | ${pct}%`);
}

// Appeler toutes les 30 secondes pendant la migration
setInterval(() => checkProgress(db), 30000);

Phase 4 : la double écriture pendant la transition

Pour éviter les incohérences pendant la migration, modifiez votre code applicatif pour qu'il écrive à la fois l'ancien _id et le nouveau uuid. Pour les nouveaux documents, préférez UUID v7 (triable dans le temps) au lieu de v4 si vous avez besoin de requêtes chronologiques :

// Pour les nouveaux documents après la migration
// Préférer UUID v7 (trié dans le temps) à UUID v4 (purement aléatoire)
// si vous avez besoin de requêtes chronologiques

const { v7: uuidv7 } = require('uuid'); // uuid@9.0+
const newUser = new User({
  name: 'Alice',
  email: 'alice@example.com',
  uuid: uuidv7() // Trié dans le temps, pas d'impact sur l'index MongoDB
});
await newUser.save();
// L'ancien code continue à utiliser _id, le nouveau utilise uuid

Phase 5 : basculer les requêtes sur UUID

Une fois toutes les données migrées et les index en place, modifiez vos endpoints API pour utiliser uuid au lieu de _id. Supprimez progressivement les anciennes références.

Validation — ne sautez pas cette étape

Exécutez une validation statistiquement significative avant de déclarer la migration réussie :

async function validateMigration(db, sampleSize = 1000) {
  const results = {
    total_checked: 0,
    uuid_missing: 0,
    uuid_invalid: 0,
    broken_references: 0
  };
  
  // 1. Vérifier que tous les documents ont un UUID valide
  const uuidRegex = /^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i;
  
  const sample = await db.collection('users')
    .aggregate([{ $sample: { size: sampleSize } }])
    .toArray();
  
  for (const doc of sample) {
    results.total_checked++;
    if (!doc.uuid) results.uuid_missing++;
    else if (!uuidRegex.test(doc.uuid)) results.uuid_invalid++;
  }
  
  // 2. Vérifier l'intégrité des références
  const orderSample = await db.collection('orders')
    .aggregate([{ $sample: { size: sampleSize } }])
    .toArray();
    
  for (const order of orderSample) {
    if (order.user_uuid) {
      const user = await db.collection('users')
        .findOne({ uuid: order.user_uuid });
      if (!user) results.broken_references++;
    }
  }
  
  console.log('Résultats de validation :', results);
  
  const isValid = results.uuid_missing === 0 && 
                  results.uuid_invalid === 0 && 
                  results.broken_references === 0;
  
  console.log(isValid ? '✅ Migration valide' : '❌ Problèmes détectés — corrigez avant de passer en production');
  return isValid;
}

Rollback : comment revenir en arrière

Si la migration échoue, supprimez simplement le champ uuid et les index associés. Tant que vous n'avez pas supprimé _id ni les anciennes références, le rollback est instantané.

// Supprimer l'index unique
await db.collection('users').dropIndex('uuid_1');

// Supprimer le champ uuid de tous les documents
await db.collection('users').updateMany(
  { uuid: { $exists: true } },
  { $unset: { uuid: "" } }
);

Gardez toujours une sauvegarde de la base avant de lancer la migration.

Foire Aux Questions

Puis‑je supprimer le champ _id après la migration ?

Non, vous ne pouvez pas supprimer _id car c'est la clé primaire imposée par MongoDB. Mais vous pouvez ne plus l'utiliser dans votre code applicatif et baser toutes vos requêtes sur le champ uuid.

Quel est l'impact sur les performances ?

Un index UUID est légèrement plus grand qu'un index ObjectID (16 octets contre 12). Pour la plupart des applications avec des collections de moins de 10 millions de documents, la différence est négligeable sur les requêtes simples. Les pipelines d'agrégation complexes peuvent être plus sensibles — testez toujours avec votre propre volume de données.

Dois‑je migrer toutes les collections en même temps ?

Non, vous pouvez migrer collection par collection. Commencez par les collections sans relations, puis attaquez les collections avec références une fois que la double écriture est en place.
