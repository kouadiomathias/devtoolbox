ObjectID MongoDB vs UUID — Structure, portabilité et migration pratique

Publié le 23 mai 2026 par Kouame Kouadio Mathias — 9 min de lecture
En résumé : L'ObjectID de MongoDB encode un timestamp, une machine et un compteur sur 12 octets. Il est ultra‑compact mais difficilement portable hors de l'écosystème MongoDB. Nous expliquons sa structure, le risque réel de collision, et comment migrer progressivement vers UUID v7 avec des exemples de code exécutables.

Table de décision — ObjectID ou UUID pour votre projet ?

ContexteIdentifiant recommandé
Vous restez 100 % dans l'écosystème MongoDB, sans exposer les IDsObjectID (par défaut)
Vous prévoyez de migrer ou de cohabiter avec PostgreSQL / MySQLUUID v7 (portabilité maximale)
Vous exposez vos identifiants dans une API REST publiqueUUID v7 (non énumérable, standard)
Vous avez besoin d'un tri chronologique intégré à l'IDObjectID ou UUID v7 (tous deux contiennent un timestamp)
Vous évoluez dans un cluster MongoDB à haute concurrenceUUID v7 (pas de risque de collision lié au compteur)

Structure interne d'un ObjectID — Ce que contiennent réellement 12 octets

Contrairement à ce que beaucoup pensent, un ObjectID n'est pas une valeur aléatoire. Sa structure est déterministe :

ObjectID : 507f1f77bcf86cd799439011
507f1f77 bcf86cd799 439011
├─ Timestamp ─┤ ├── Machine+PID ──┤ ├─ Compteur ─┤
4 octets 5 octets 3 octets

Le timestamp de 4 octets est un horodatage Unix en secondes (pas en millisecondes). Les 5 octets suivants identifient la machine et le processus qui ont généré l'ID. Enfin, un compteur incrémental de 3 octets permet de différencier plusieurs ObjectID créés dans la même seconde par le même processus.

Cette structure rappelle UUID v1 (timestamp + node) mais dans un format plus compact de 12 octets au lieu de 16.

Portabilité — Le point faible d'ObjectID

ObjectID est parfaitement intégré à MongoDB : il est léger, triable par date de création et garanti unique au sein d'une collection. Mais dès que vous devez exposer cet ID dans une API REST, le stocker dans PostgreSQL, ou l'utiliser comme identifiant global entre plusieurs services, les problèmes commencent.

  • Pas de type natif : PostgreSQL, MySQL et les autres bases n'ont pas de type ObjectID. Vous devez le stocker en VARCHAR(24) ou en binaire, perdant la validation automatique.
  • Format non standard : les validateurs JSON Schema, les ORM et les bibliothèques clientes reconnaissent format: uuid, pas format: objectid.
  • Encodage hexadécimal : la représentation textuelle de 24 caractères est moins dense que les 36 caractères d'un UUID, mais l'absence de tirets peut surprendre les développeurs habitués aux UUID.

En bref, ObjectID est un excellent identifiant local qui devient un boulet dès que vous sortez de MongoDB.

Le piège du compteur incrémental

Dans des scénarios de très forte concurrence (plusieurs milliers d'insertions par seconde sur une même machine), le compteur de 3 octets (16,7 millions de valeurs) peut déborder à l'intérieur d'une même seconde. Si le timestamp n'a pas encore changé, deux documents peuvent recevoir le même ObjectID, provoquant des erreurs de clé dupliquée.

Problème documenté : Sur des clusters MongoDB avec des horloges désynchronisées ou lors de pics de charge extrêmes, la combinaison timestamp + compteur n'est plus suffisante pour garantir l'unicité. UUID v7, avec 74 bits d'aléa, élimine ce risque.

Ce que nous avons changé d'avis — L'évolution de nos recommandations

En 2022, nous recommandions ObjectID pour tout projet MongoDB, et ULID pour les projets multi‑bases. En 2024, avec la publication de la RFC 9562 et la disponibilité de bibliothèques UUID v7 stables dans la plupart des langages, notre recommandation a évolué :

  • ObjectID reste acceptable pour les projets purement MongoDB qui n'exposent pas leurs identifiants à l'extérieur.
  • UUID v7 est devenu notre recommandation par défaut pour tout nouveau projet, y compris sur MongoDB, en raison de sa portabilité, de sa standardisation et de son tri chronologique.

Cette section documente ce changement pour que vous puissiez comprendre le contexte de nos choix actuels.

Migration pratique d'ObjectID vers UUID v7 dans MongoDB

Si vous avez déjà une collection avec des ObjectID, vous pouvez introduire progressivement UUID v7 sans interrompre votre service. Voici une approche éprouvée :

  1. Ajoutez un champ uuid_v7 à vos documents existants. Vous pouvez générer un UUID v7 basé sur le timestamp de l'ObjectID pour conserver un ordre chronologique approximatif.
  2. Modifiez votre code applicatif pour générer un UUID v7 à la création de chaque nouveau document.
  3. Doublez les index pendant la transition : conservez l'index sur _id et ajoutez un index sur uuid_v7.
  4. Lorsque tous les anciens documents sont migrés ou purgés, remplacez _id par la valeur du champ uuid_v7 et supprimez l'ancien index.
// Exemple : migration partielle avec ajout d'un champ uuid_v7
db.collection('users').aggregate([
  {
    $addFields: {
      uuid_v7: { $function: {
        body: function(oid) {
          // Utilisez une librairie UUID v7 côté serveur
          // Ici, un exemple simplifié qui extrait le timestamp et le combine avec un aléa
          const ts = Math.floor(oid.getTimestamp().getTime() / 1000);
          // ... génération UUID v7 (non montrée pour la concision)
          return generateUUIDv7(ts);
        },
        args: ["$_id"],
        lang: "js"
      }}
    }
  }
]);

Foire Aux Questions

Est‑ce que ObjectID est plus rapide qu'UUID pour les index MongoDB ?

ObjectID est légèrement plus compact (12 octets contre 16), mais la différence de performance d'index est négligeable en pratique. Le véritable gain est la portabilité, pas la vitesse.

Puis‑je décoder un ObjectID sans MongoDB ?

Oui, les 4 premiers octets donnent le timestamp Unix. Vous pouvez extraire la date de création sans interroger la base.

Est‑il sûr de changer le type de l'_id sur une collection existante ?

Non, ne changez pas le type de _id sur place. Ajoutez plutôt un nouveau champ (comme uuid_v7) et migrez progressivement, comme décrit ci‑dessus.
