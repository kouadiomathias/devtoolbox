UUID vs BIGINT : le vrai impact sur vos requêtes PostgreSQL à 100M de lignes

7 juin 2026 · Mis à jour le 9 juin 2026 · 14 min de lecture · Par Mathias Kouadio

Le débat revient à chaque nouveau projet : faut‑il utiliser un BIGSERIAL tout simple ou un UUID pour les clés primaires ? Les réponses qu'on lit oscillent entre « BIGINT est plus rapide » et « UUID est indispensable en distribué ». Mais à partir de quel volume la différence devient‑elle perceptible ?

J'ai voulu trancher avec des mesures concrètes sur 100 millions de lignes, sur un PostgreSQL 16 non tuné, avec un SSD NVMe et 32 Go de RAM. Les UUID v7 ont été générés avec le paquet npm uuid v9.0 (ou uuid-utils v0.9 en Python). Voici les résultats détaillés et les conclusions qu'il faut en tirer pour vos prochains schémas.

Protocole de test reproductible (DDL inclus)

Trois tables principales, chacune avec 100 millions de lignes, représentant trois stratégies de clé primaire. Les valeurs UUID v7 ont été générées en application avec le paquet uuid npm v9.0 (ou uuid-utils v0.9 en Python).

-- Tables pour PostgreSQL 16
CREATE TABLE events_bigint (
  id      BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  payload JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE events_uuid4 (
  id      UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  payload JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE events_uuid7 (
  id      UUID PRIMARY KEY,  -- généré côté application
  payload JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Table FK pour le benchmark de jointures
CREATE TABLE event_details_bigint (
  id       BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  event_id BIGINT REFERENCES events_bigint(id),
  detail   TEXT
);
CREATE INDEX ON event_details_bigint(event_id);

-- Après 100M d'insertions, mesurer la taille des index
SELECT
  tablename,
  indexname,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE tablename IN ('events_bigint', 'events_uuid4', 'events_uuid7')
ORDER BY pg_relation_size(indexrelid) DESC;

Benchmark 1 : insertions pures

100 millions d'INSERTs en batch de 1 000 lignes, exécutés avec \copy depuis des fichiers CSV pré‑générés.

Type de PKTemps total (min)Latence moyenne / ligne
BIGINT38~ 0,023 ms
UUID v752~ 0,031 ms
UUID v4128~ 0,077 ms

BIGINT est 3,3× plus rapide qu'UUID v4 et environ 1,4× plus rapide qu'UUID v7. L'absence de fragmentation d'index profite énormément au BIGINT, mais UUID v7 s'en sort très honorablement car les écritures restent séquentielles.

La concurrence compte : avec 16 workers d'insertion parallèles, les performances d'UUID v4 chutent fortement à cause des contentions de page split sur l'index B‑tree. Dans notre test, UUID v4 a pris 215 minutes contre 58 minutes pour UUID v7 — presque 4× plus lent. BIGINT est resté stable à 42 minutes.

Benchmark 2 : jointures sur clé étrangère

La table event_details référence la table d'événements via une FK. On exécute une jointure simple SELECT * FROM events JOIN event_details ON events.id = event_details.event_id sur 10 millions de lignes (avec un index sur la FK).

FK pointant versTemps d'exécution (s)Taille de l'index FK
BIGINT12,4~ 316 Mo
UUID v721,7~ 632 Mo
UUID v422,1~ 632 Mo

Ici, BIGINT gagne largement : l'index FK est deux fois plus petit, et le parcours est nettement moins coûteux en I/O. L'UUID v4 et v7 sont équivalents en lecture : la fragmentation n'impacte pas les jointures, seule la taille de l'index compte.

Benchmark 3 : range queries sur created_at

Requête typique : SELECT * FROM events WHERE created_at BETWEEN '2026-01-01' AND '2026-01-02'. On mesure l'impact de l'ordre des PK sur le clustering.

Type de PKTemps (ms)Pages lues
BIGINT180~ 3 400
UUID v7205~ 3 900
UUID v41 240~ 41 000

UUID v4 est dramatiquement mauvais : les lignes créées le même jour sont dispersées sur tout le disque, multipliant les lectures par 12. UUID v7 récupère presque les performances de BIGINT, parce que l'ordre de la PK suit l'ordre du created_at. Pour les requêtes temporelles, la PK triée joue un rôle de quasi‑index de clustering.

Benchmark 4 : point lookups par PK

SELECT * FROM events WHERE id = '...' exécuté 100 000 fois avec des valeurs aléatoires.

Type de PKTemps moyen par lookup (ms)
BIGINT0,31
UUID v70,38
UUID v40,40

La différence est négligeable. Un point lookup traverse un B‑tree de hauteur quasi identique. Une fois l'index warm en cache, l'écart fond à moins de 0,1 ms. Ne choisissez pas votre PK sur ce critère.

Mesurer la fragmentation d'index

Utilisez l'extension pgstattuple pour voir l'état interne réel :

CREATE EXTENSION IF NOT EXISTS pgstattuple;

SELECT
  indexrelid::regclass AS index_name,
  avg_leaf_density,
  avg_fragmentation_in_percent
FROM pgstatindex('events_uuid4_pkey');

-- Résultats attendus après 100M d'insertions aléatoires :
-- UUID v4 : avg_fragmentation_in_percent ~85-99%
-- UUID v7 : ~2-5%
-- BIGINT  : ~0-1%

Après un REINDEX TABLE CONCURRENTLY events_uuid4; la fragmentation tombe à zéro, mais elle réapparaît rapidement avec de nouvelles insertions aléatoires. Ce coût de maintenance caché est une raison de plus pour préférer UUID v7 ou BIGINT pour les tables avec un fort volume d'écriture.

Le vrai coût : la taille de stockage des clés étrangères

Ce n'est pas la PK qui coûte cher : c'est chaque FK qui la référence. Si vous avez 20 tables qui pointent vers users.id, vous multipliez l'écart par 20.

ScénarioVolume PK + 5 FK (Go)
Tout BIGINT (8 octets)12,8
Tout UUID (16 octets)25,6
Hybride (BIGINT interne + UUID exposé)15,4

Sur 100M d'utilisateurs et 5 tables liées, la facture disque double en full UUID. La RAM utilisée pour le cache suit la même proportion. Si vous gérez un budget cloud serré, le BIGINT a encore de beaux jours devant lui.

Conclusion nuancée : la meilleure approche en 2026

UUID v7 réduit massivement l'écart avec BIGINT pour les écritures et les requêtes temporelles. Mais BIGINT reste roi pour les jointures et l'empreinte mémoire.

La stratégie que je recommande aujourd'hui :

  • BIGINT auto‑incrément en interne (jointures rapides, FK compactes).
  • UUID v7 exposé publiquement dans vos API (sécurité, non‑énumérabilité).

Cette approche hybride ajoute une colonne, mais elle vous donne le meilleur des deux mondes. L'overhead est minime face aux gains en sécurité et en flexibilité.

Foire Aux Questions

Est-ce que BIGINT est toujours plus rapide qu'UUID dans PostgreSQL ?

Pour les insertions et les jointures massives, oui, BIGINT reste plus rapide en raison d'une taille d'index plus faible (8 octets contre 16). Mais UUID v7 réduit considérablement la fragmentation d'index et s'en approche.

Puis-je utiliser UUID v7 comme clé primaire sans regret ?

Oui, UUID v7 donne d'excellentes performances en écriture tout en offrant les avantages d'un identifiant global. L'impact en espace disque et mémoire cache est le principal compromis.

Quelle est la meilleure approche pour les clés primaires en 2026 ?

L'approche hybride : BIGINT auto-incrément en interne (jointures rapides) + UUID v7 exposé publiquement. Elle combine vitesse et sécurité.
