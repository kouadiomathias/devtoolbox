Pourquoi les en‑têtes HTTP sont plus importants qu’on ne le pense

Les en‑têtes HTTP sont la première chose qu’un navigateur lit lorsqu’il reçoit une réponse — avant même le moindre octet de HTML. Ils indiquent au navigateur comment restituer le contenu, ce qu’il doit mettre en cache, quelles origines sont fiables et comment protéger l’utilisateur. Un seul en‑tête manquant peut exposer les utilisateurs au détournement de clic, au cross‑site scripting ou à des attaques de l’homme du milieu. Un en‑tête Cache-Control mal configuré peut diffuser des données obsolètes pendant des heures.

Le problème, c’est que les en‑têtes sont invisibles lors de la navigation habituelle. Ils vivent dans le panneau Réseau des DevTools, dans la sortie de curl -I ou dans les fichiers de configuration Nginx — pas dans la page elle‑même. Cet outil les rend visibles, explique chacun d’entre eux en langage clair et vous indique ceux qui vous manquent avant qu’un attaquant ne s’en aperçoive.

Les en‑têtes de sécurité que tout serveur de production devrait avoir

Strict-Transport-Security (HSTS)

Indique au navigateur de toujours utiliser HTTPS pour ce domaine, même si l’utilisateur tape http://. Sans cela, un attaquant réseau peut intercepter la première requête HTTP avant la redirection vers HTTPS. La valeur recommandée est max-age=31536000; includeSubDomains (un an). L’ajout de preload enregistre également le domaine dans les listes de préchargement des navigateurs, empêchant toute connexion HTTP.

Content-Security-Policy (CSP)

Définit quelles sources de scripts, styles, images et polices sont autorisées. C’est la principale défense contre le cross‑site scripting (XSS). Une CSP stricte comme default-src 'self'; script-src 'self' empêche l’exécution de scripts injectés. La CSP est l’en‑tête le plus complexe à configurer correctement — l’outil met en évidence les valeurs dangereusement permissives telles que unsafe-inline et unsafe-eval.

X-Frame-Options

Empêche votre page d’être chargée dans une <iframe> sur un domaine différent — la défense classique contre le détournement de clic. Utilisez DENY pour bloquer complètement l’affichage dans une iframe, ou SAMEORIGIN pour l’autoriser uniquement depuis votre propre domaine. Note : l’équivalent moderne est la directive CSP frame-ancestors, plus flexible.

X-Content-Type-Options

Régler cet en‑tête à nosniff empêche les navigateurs de deviner le type MIME du contenu. Sans cela, un navigateur pourrait exécuter un fichier texte comme du JavaScript si une balise script pointe dessus. Une ligne, une énorme protection. Il n’y a aucune raison de ne pas le définir.

Referrer-Policy

Contrôle la quantité d’informations incluses dans l’en‑tête Referer lors de la navigation entre pages. Sans cet en‑tête, l’URL complète (y compris les paramètres de requête, qui peuvent contenir des jetons de session) est envoyée aux sites tiers. strict-origin-when-cross-origin est la valeur par défaut recommandée en 2026.

Permissions-Policy

Le successeur de Feature-Policy. Restreint l’accès aux API du navigateur comme la caméra, le microphone, la géolocalisation et les paiements. Si votre application n’utilise pas la caméra, définir camera=() signifie qu’un script compromis ne pourra jamais y accéder.

En‑têtes qui fuient des informations — supprimez‑les ou masquez‑les

Server

Des en‑têtes comme Server: Apache/2.4.51 (Ubuntu) ou Server: nginx/1.18.0 révèlent votre logiciel serveur et sa version à quiconque les consulte. Les attaquants s’en servent pour rechercher des CVE connues pour cette version précise. Définissez une valeur générique ou supprimez‑le entièrement : dans Nginx, server_tokens off;.

X-Powered-By

Ajouté par défaut par des frameworks comme Express.js et PHP. X-Powered-By: Express ou X-Powered-By: PHP/8.1.0 est un renseignement gratuit pour un attaquant. Dans Express : app.disable('x-powered-by'). Dans PHP : expose_php = Off dans php.ini.

Comprendre les directives Cache-Control

Cache-Control est l’un des en‑têtes les plus importants pour la performance, mais ses directives sont souvent mal comprises :

  • max-age=N — mettre en cache la réponse pendant N secondes
  • no-store — ne jamais mettre en cache cette réponse (données sensibles)
  • no-cache — mettre en cache mais toujours revalider auprès du serveur avant de servir
  • private — seul le navigateur peut mettre en cache (pas les CDN ni les proxys)
  • public — les CDN et les proxys peuvent mettre en cache
  • stale-while-revalidate=N — servir du contenu périmé pendant N secondes tout en récupérant une version fraîche en arrière‑plan
  • immutable — la ressource ne changera jamais ; saute complètement la revalidation

Comment la note de sécurité est calculée

Le score commence à 100 et des points sont déduits pour chaque en‑tête de sécurité manquant ou mal configuré. Voici l’échelle de notation :

  • A+ (95–100) : tous les en‑têtes de sécurité critiques présents et bien configurés
  • A (85–94) : la plupart des en‑têtes de sécurité présents, aucun problème critique
  • B (70–84) : quelques en‑têtes de sécurité présents, des lacunes mineures
  • C (50–69) : des en‑têtes de sécurité importants sont manquants
  • D (30–49) : la plupart des en‑têtes de sécurité sont absents
  • F (0–29) : configuration de sécurité critique manquante

Foire aux questions

Comment obtenir les en‑têtes HTTP de mon site web ?

Trois méthodes : (1) Outils de développement du navigateur → onglet Réseau → cliquez sur une requête → panneau En‑têtes → copiez la section En‑têtes de réponse. (2) Terminal : curl -I https://votresite.com. (3) Terminal avec tous les en‑têtes : curl -D - https://votresite.com -o /dev/null. Collez la sortie directement dans cet outil.

Mes données sont‑elles envoyées à un serveur lorsque j’utilise cet outil ?

Non. Toute l’analyse s’effectue dans votre navigateur grâce à JavaScript. Vos en‑têtes — qui peuvent contenir des détails de configuration sensibles — ne quittent jamais votre machine. C’est pourquoi nous avons conçu cet outil en pur côté client plutôt que de demander une URL qui nécessiterait un proxy côté serveur.

Ma CSP contient unsafe-inline. Est‑ce toujours mauvais ?

Presque toujours. unsafe-inline autorise les balises <script> en ligne et les attributs d’événements JavaScript, exactement ce qu’injectent les attaques XSS. Si vous avez besoin de scripts en ligne, utilisez plutôt une CSP basée sur un nonce : script-src 'nonce-{aléatoire}'. Pour les styles en ligne, envisagez de les déplacer dans une feuille de style. Il y a très peu de raisons légitimes d’utiliser unsafe-inline dans une application de production.

Quelle est la différence entre X-Frame-Options et CSP frame-ancestors ?

X-Frame-Options est l’ancien en‑tête (2008) avec trois options : DENY, SAMEORIGIN et la désormais dépréciée ALLOW-FROM. La directive CSP frame-ancestors est l’équivalent moderne et plus flexible — elle prend en charge plusieurs origines autorisées, les jokers et 'none' (équivalent à DENY). Si vous utilisez CSP avec frame-ancestors, vous pouvez omettre X-Frame-Options, mais conserver les deux offre une défense en profondeur pour les navigateurs plus anciens.

Comment ajouter des en‑têtes de sécurité dans Nginx ?

Ajoutez ces directives dans votre bloc server ou location de votre configuration Nginx :

add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
server_tokens off;

Puis rechargez : nginx -t && systemctl reload nginx
