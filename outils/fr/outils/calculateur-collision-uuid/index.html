Comprendre la probabilité de collision des UUID

Les UUID (Universally Unique Identifiers) sont conçus pour être uniques, mais ils ne sont pas infinis. La probabilité que deux UUID générés aléatoirement entrent en collision est extrêmement faible, mais pas nulle. La probabilité suit le paradoxe des anniversaires : avec n identifiants et d valeurs possibles, la probabilité approximative d'au moins une collision est :

p(n) ≈ 1 − exp(−n² / (2d))

Pour UUID v4, d = 2122 (environ 5,3 × 1036) car 6 bits sont réservés pour la version et la variante. Cela signifie qu'il faudrait générer environ 2,7 × 1018 UUID pour avoir 50 % de chance de rencontrer une seule collision. C'est 2,7 milliards de milliards.

Probabilité de collision pour les identifiants courants

Utilisez le calculateur pour voir comment les différents types d'identifiants se comparent :

  • UUID v4 – 122 bits d'entropie, l'UUID aléatoire standard.
  • UUID v7 – mêmes 122 bits, mais ordonné dans le temps ; probabilité de collision identique à v4 pour la partie aléatoire.
  • NanoID (21 caractères) – 126 bits d'entropie avec l'alphabet par défaut (64 symboles).
  • ULID – 80 bits aléatoires, horodatage de 48 bits ; total de 128 bits, mais l'unicité repose sur les 80 bits aléatoires.

Foire Aux Questions

Deux UUID peuvent-ils vraiment être identiques ?

Oui, c'est théoriquement possible, mais la probabilité est astronomiquement faible. Avec 122 bits aléatoires, vous devez générer environ 2,7 × 1018 UUID pour atteindre 50 % de chance de collision. À titre de comparaison, générer 1 milliard d'UUID par seconde pendant 100 ans ne produirait que 3,2 × 1018 UUID.

Les collisions d'UUID v4 sont-elles vraiment un problème en pratique ?

Non. Le nombre d'UUID nécessaire pour qu'une collision ait une probabilité significative est tellement élevé que ce n'est pas une préoccupation pratique dans un système réel. D'autres modes de défaillance (corruption de base de données, bugs) sont bien plus probables.

Comment UUID v7 se compare-t-il à v4 en termes de collision ?

UUID v7 possède la même taille de partie aléatoire (74 bits après l'horodatage), donc la probabilité de collision est identique à v4 pour la portion aléatoire. L'horodatage garantit l'ordre mais n'augmente pas l'entropie aléatoire.
