    YAML, JSON, TOML — quand utiliser quel format

    Ces trois formats résolvent le même problème : sérialiser des données structurées dans un format texte lisible par un humain. Mais ils ont été conçus avec des priorités différentes, et choisir le mauvais format cause de réelles frictions.

    FormatIdéal pourMoins adapté pourUtilisé par
    JSONDonnées d'API, échange de données, stockage webLongs fichiers de configuration édités manuellement (pas de commentaires)APIs REST, npm, package.json
    YAMLConfiguration complexe, chaînes multi-lignes, ancresGrands volumes de données (l'analyse est lente), équipes débutantesKubernetes, GitHub Actions, Ansible, Docker Compose
    TOMLFichiers de configuration simples, typage explicite, débutantsStructures imbriquées complexes, valeurs nullesCargo.toml (Rust), pyproject.toml, Hugo, modules Go

    Le problème des dates en YAML — et comment cet outil le gère

    Le schéma par défaut de YAML analyse automatiquement les dates nues. Une valeur comme 2024-01-15 est analysée comme un objet Date JavaScript, pas comme une chaîne. Lorsque vous la convertissez ensuite en JSON, vous obtenez une chaîne d'horodatage UTC ("2024-01-15T00:00:00.000Z") — pas la chaîne que vous aviez saisie. Cela piège constamment les développeurs qui convertissent des manifestes Kubernetes ou des workflows GitHub Actions.

    Activez « Dates en tant que chaînes » dans les options d'analyse pour utiliser JSON_SCHEMA au lieu du schéma par défaut, ce qui conserve les valeurs de type date sous forme de chaînes tout au long de la conversion.

    # Entrée YAML
release_date: 2024-01-15
version: "2.1.0"

# ❌ Comportement par défaut — sortie JSON
{ "release_date": "2024-01-15T00:00:00.000Z", "version": "2.1.0" }

# ✅ Avec "Dates en tant que chaînes" — sortie JSON
{ "release_date": "2024-01-15", "version": "2.1.0" }

    Ancres et alias YAML — ce qu'ils sont et ce qu'ils deviennent

    Les ancres YAML vous permettent de définir une valeur une fois et de la référencer plusieurs fois, évitant ainsi les répétitions. C'est très courant dans les pipelines CI/CD où l'on souhaite réutiliser des définitions d'environnement ou des étapes de configuration.

    # YAML avec ancres
defaults: &default-settings
  timeout: 30
  retries: 3
  log_level: info

service_a:
  <<: *default-settings
  port: 8080

service_b:
  <<: *default-settings
  port: 9090

    Pendant la conversion, les ancres sont résolues — l'alias est remplacé par la valeur réelle. L'option « Résoudre les ancres » rend cela explicite dans la sortie YAML (supprime les références & et *). JSON et TOML ne supportent pas les ancres, ils reçoivent donc toujours les valeurs résolues.

    Limitations de TOML — ce qui ne peut pas être converti

    TOML est moins expressif que JSON et YAML sur certains points. Si vos données source contiennent l'un des éléments suivants, la conversion vers TOML échouera :

    • Valeurs nulles — TOML n'a pas de concept de null. Convertissez en YAML à la place.
    • Tableaux de types mixtes — les tableaux TOML doivent contenir des valeurs du même type. [1, "deux", true] est invalide en TOML.
    • Clés non-chaînes — les clés TOML doivent être des chaînes (ce que JSON utilise toujours, mais certains documents YAML peuvent avoir des clés entières).
    • Tableaux anonymes profondément imbriqués — la syntaxe de tableaux de tables de TOML a des limites sur l'imbrication des structures.

    Comparaison sémantique — pourquoi l'égalité textuelle ne suffit pas

    Deux fichiers YAML représentant la même configuration peuvent sembler complètement différents en tant que texte : ordre des clés différent, indentation différente, styles de guillemets différents. Un outil de diff montrerait des centaines de changements. La comparaison sémantique de cet outil normalise les deux documents — les analyse, trie toutes les clés récursivement, puis compare les structures résultantes. Deux documents sémantiquement identiques produisent la même structure normalisée, quel que soit le formatage.

    C'est particulièrement utile pour vérifier qu'une migration de configuration a conservé toutes les valeurs, ou pour confirmer que deux fichiers générés automatiquement par des outils différents sont équivalents.

    Foire aux questions

    Pourquoi YAML convertit-il les dates en valeurs inattendues ?

    Par défaut, js-yaml analyse les dates nues comme 2024-01-01 en tant qu'objets Date JavaScript, qui sont ensuite sérialisés en chaînes d'horodatage UTC dans JSON. Activez Dates en tant que chaînes dans les options d'analyse. Cela utilise JSON_SCHEMA au lieu du schéma YAML complet, conservant les valeurs de type date sous forme de texte brut tout au long du pipeline de conversion.

    Qu'est-ce qu'une ancre et un alias YAML ?

    Une ancre YAML (&nom) marque une valeur afin de pouvoir la réutiliser ailleurs via un alias (*nom). Les ancres évitent de répéter le même bloc dans un fichier de configuration. Ce convertisseur résout les ancres pendant l'analyse — les sorties JSON et TOML contiennent toujours les valeurs résolues. Dans la sortie YAML, l'activation de « Résoudre les ancres » remplace tous les alias par leurs valeurs réelles.

    Quelle est la différence entre YAML et TOML ?

    YAML est un sur-ensemble flexible de JSON axé sur la sérialisation de données lisibles par un humain, souvent utilisé pour la configuration complexe (Kubernetes, GitHub Actions, Ansible). TOML est plus simple et plus explicite, conçu spécifiquement pour les fichiers de configuration où la clarté est plus importante que la puissance expressive — il est utilisé par Rust (Cargo.toml), Python (pyproject.toml), Hugo et Go. TOML est moins ambigu que YAML mais aussi moins capable (pas de null, pas de tableaux de types mixtes).

    TOML peut-il représenter toutes les données que JSON peut ?

    Non. TOML ne peut pas représenter les valeurs null — le null JSON n'a pas d'équivalent TOML. TOML exige également que les tableaux soient homogènes (tous les éléments du même type). Si votre JSON contient null ou des tableaux de types mixtes, la conversion vers TOML échouera. Dans ce cas, convertissez en YAML à la place, qui prend en charge null et les tableaux mixtes nativement.

    Mes données sont-elles envoyées à un serveur ?

    Non. Toute l'analyse, la conversion, la validation et la comparaison s'exécutent entièrement dans votre navigateur en utilisant js-yaml et smol-toml. Vos fichiers de configuration, secrets et données ne quittent jamais votre machine. C'est particulièrement important pour les fichiers pouvant contenir des clés API, des mots de passe de base de données ou des noms d'hôte internes.

    Pourquoi ma conversion YAML diffère-t-elle selon l'option de tri des clés ?

    Les objets JSON et les mappings YAML ne garantissent pas l'ordre des clés, donc l'ordre dans la sortie dépend de la façon dont l'analyseur a construit l'objet. L'option « Trier les clés » dans la sortie JSON applique un tri alphabétique récursif à toutes les clés d'objet. C'est utile pour produire une sortie déterministe pour les diffs et le contrôle de version, mais la sortie triée est sémantiquement identique à celle non triée.
