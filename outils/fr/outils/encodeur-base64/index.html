L'encodage Base64 en pratique : là où ça casse et comment le corriger

Sur cette page

  3. Pièges courants du Base64
  4. FAQ
  5. Références

Un pipeline de contenu traduit fonctionnait parfaitement en préproduction, où chaque chaîne de test était en anglais. En production, le premier contenu français contenant un caractère accentué — café — est passé par btoa() et toute la requête a planté avec InvalidCharacterError: Failed to execute 'btoa' on 'Window'. btoa() n'accepte que les caractères Latin1 (octet unique) ; toute chaîne avec une lettre accentuée, un emoji ou un caractère non-ASCII lève immédiatement une exception. La solution n'est pas d'éviter les accents — c'est d'encoder les véritables octets UTF-8 de la chaîne plutôt que ses caractères JavaScript bruts.

Ce seul piège provoque un nombre disproportionné de rapports de bugs liés à Base64, mais il est loin d'être le seul. Base64 semble trivial — trois octets en entrée, quatre caractères en sortie — pourtant les détails environnants (sécurité URL, remplissage, encodage, stockage) sont là où les vrais problèmes de production apparaissent.

Base64 n'est pas du chiffrement. Il n'a ni clé ni secret — c'est un encodage de texte réversible. Ne l'utilisez jamais pour protéger des mots de passe, des jetons ou des données personnelles. Quiconque reçoit la chaîne peut la décoder en une seule ligne.

Pièges courants du Base64

01btoa() lève une exception sur du texte non-ASCII
Élevée

btoa() opère sur des unités de code UTF-16 et exige que chacune tienne sur un seul octet (0–255). Toute lettre accentuée, écriture non latine ou emoji lève InvalidCharacterError immédiatement.

À éviter
const b64 = btoa("café"); // lève InvalidCharacterError
Correction
function encodeUtf8Base64(str) { const bytes = new TextEncoder().encode(str); let binary = ''; bytes.forEach(b => binary += String.fromCharCode(b)); return btoa(binary); // fonctionne pour toute chaîne Unicode valide }
02Le Base64 standard casse à l'intérieur d'une URL
Élevée

+ est interprété comme un espace dans les chaînes de requête, et / ressemble à un séparateur de chemin. Une chaîne Base64 contenant l'un ou l'autre corrompra silencieusement l'URL dans laquelle elle est intégrée.

À éviter
https://api.exemple.com/verify?token=a+b/c== // '+' et '/' sont mal interprétés par l'analyse d'URL
Correction
b64url = b64.replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, ''); // c'est exactement ce que fait le bouton URL-safe ci-dessus
03Oublier de retirer le préfixe d'URL de données
Moyenne

Une URL de données complète ressemble à data:image/png;base64,iVBORw0KG.... Passer toute la chaîne à atob() ou base64.b64decode() échoue ou produit des données incorrectes, car le préfixe de métadonnées n'est pas du Base64 valide.

Correction
const raw = dataUrl.split(',')[1]; // supprime tout ce qui précède la première virgule const bytes = atob(raw);
04Stocker des blobs Base64 dans des colonnes relationnelles
Moyenne

Base64 ajoute déjà environ 33% de surcoût par rapport aux octets bruts. Stocker ce texte dans une colonne TEXT/VARCHAR au lieu d'un type binaire natif ajoute un surcoût supplémentaire dû à la collation, à l'indexation et à la planification des requêtes que les types binaires évitent.

Correction
-- PostgreSQL : stockez les octets bruts, pas le texte Base64 CREATE TABLE attachments (id uuid PRIMARY KEY, data bytea); -- ou mieux : stockez le fichier dans un stockage objet (S3, GCS) et ne gardez qu'une URL/clé

Foire aux questions

Est-ce que Base64 est un chiffrement ?
Non. Base64 est un encodage entièrement réversible sans clé. Quiconque possède la chaîne peut la décoder instantanément. Utilisez un vrai chiffrement (AES, RSA) pour tout ce qui doit rester secret.
Quelle est la différence entre Base64 et Base64url ?
Base64url remplace les deux caractères non sûrs pour les URL — + devient -, / devient _ — et supprime le remplissage =, de sorte que le résultat soit sûr à l'intérieur des URL, des noms de fichiers et des en-têtes HTTP. Les segments d'en-tête et de charge utile JWT utilisent toujours Base64url.
Comment convertir une image ou un fichier en Base64 dans le navigateur ?
Déposez un fichier dans le panneau « Fichier / Image → Base64 » ci-dessus, ou dans votre propre code utilisez la méthode readAsDataURL() de l'API FileReader — elle retourne une URL de données prête à l'emploi.
Comment encoder en Base64 en JavaScript sans planter sur les accents ?
N'appelez pas btoa() directement sur une chaîne Unicode. Encodez-la d'abord en octets UTF-8 avec TextEncoder, convertissez ces octets en chaîne binaire, puis appelez btoa() sur celle-ci. Le décodage est le même processus inverse avec atob() et TextDecoder.
Comment encoder en Base64 en Python ?
Utilisez le module base64 : base64.b64encode(b"Hello"). Pour une sortie URL-safe, utilisez base64.urlsafe_b64encode(b"Hello"). Les deux acceptent et retournent des bytes, donc encodez/décodez les chaînes avec .encode()/.decode() autour.
Pourquoi Base64 se termine-t-il parfois par == ?
Base64 traite l'entrée par blocs de 3 octets pour produire 4 caractères. Lorsque la longueur d'entrée n'est pas un multiple de 3, un remplissage = est ajouté pour que la longueur de sortie soit un multiple de 4 — un = pour 1 octet manquant, deux == pour 2 octets manquants. Base64url omet complètement ce remplissage.
Mes données sont-elles envoyées à un serveur lorsque j'utilise cet outil ?
Non. Tous les encodages, décodages et traitements de fichiers se produisent entièrement dans votre navigateur via JavaScript. Rien de ce que vous tapez, collez ou déposez ici ne quitte jamais votre machine.
