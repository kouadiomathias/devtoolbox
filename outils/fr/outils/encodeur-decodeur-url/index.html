Pourquoi l'encodage d'URL piège les développeurs

L'encodage d'URL (encodage en pourcentage, défini dans la RFC 3986) convertit les caractères non sûrs en une séquence % suivie de leur code hexadécimal UTF-8 sur deux chiffres. Un espace devient %20, une esperluette devient %26, un signe égal devient %3D.

Le problème n'est pas l'encodage lui-même — c'est de le faire mal ou de le faire deux fois. Le double encodage est la cause la plus fréquente d'appels API cassés : encoder une chaîne déjà encodée produit %2520 (le % est encodé en %25), qui ne se décode jamais à la valeur attendue et provoque des erreurs 400 difficiles à diagnostiquer.

Pièges de l'analyse des chaînes de requête

Une chaîne de requête comme ?nom=Jean+Dupont&role=admin&tags=css&tags=js présente trois subtilités que la plupart des développeurs apprennent à leurs dépens : les espaces sont encodés en + (encodage formulaire) ou %20 (RFC 3986), les clés répétées (tags) représentent un tableau, et les caractères spéciaux dans les valeurs (&, =) doivent être encodés sinon ils seront interprétés comme des délimiteurs.

Foire aux questions

Quelle est la différence entre encodeURIComponent et encodeURI en JavaScript ?
encodeURIComponent encode tous les caractères sauf les lettres, les chiffres et les caractères non réservés - _ . ! ~ * ' ( ). Il est conçu pour encoder la valeur d'un seul paramètre d'URL. encodeURI préserve les caractères réservés qui font partie de la structure de l'URL comme : / ? # [ ] @ ! $ & ' ( ) * + , ; =. Utilisez encodeURIComponent pour les valeurs de paramètres et encodeURI pour une URL complète que vous souhaitez garder structurellement intacte.
Qu'est-ce que le double encodage d'URL et pourquoi est-ce un bug ?
Le double encodage se produit lorsqu'une chaîne déjà encodée en pourcentage est encodée une seconde fois. Par exemple, encoder %20 (qui représente un espace) produit %2520, car le caractère % est lui-même encodé en %25. Résultat : lorsque l'URL est décodée une fois, le destinataire obtient %20 au lieu d'un espace — la valeur attendue n'arrive jamais. Le double encodage casse les appels API et les soumissions de formulaires d'une manière difficile à repérer visuellement.
Pourquoi les espaces apparaissent-ils parfois sous forme de + au lieu de %20 ?
Le signe + pour les espaces est une convention du format application/x-www-form-urlencoded utilisé par les formulaires HTML. Dans les URL standard et les API REST, %20 est la représentation correcte selon la RFC 3986. Les deux sont largement acceptés dans les chaînes de requête, mais %20 est plus sûr pour les API, car certains backends ne décodent qu'une seule forme. Le décodeur de cet outil gère les deux.
Cet outil est-il sécurisé pour encoder des données sensibles ?
Oui. Tout l'encodage, le décodage et la génération de code s'exécutent entièrement en JavaScript dans votre navigateur. Aucun texte que vous saisissez n'est transmis à un serveur. Vous pouvez encoder en toute sécurité des URL contenant des clés API, des tokens ou d'autres paramètres sensibles.
Comment encoder une URL complète sans casser sa structure ?
Utilisez l'onglet Encodage intelligent d'URL. Il analyse votre URL en ses composants (schéma, hôte, chemin, chaîne de requête), laisse les caractères de structure (://, /, ?, &, =) intacts et n'encode que les valeurs à l'intérieur de chaque paramètre de requête. Cela évite de double-encoder les séparateurs tout en encodant correctement les caractères spéciaux dans les valeurs.
