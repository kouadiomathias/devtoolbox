L'erreur GUID la plus courante en .NET

Ces deux lignes semblent presque identiques lors d'une revue de code. L'une d'elles est un bug critique :

// ❌ FAUX — produit 00000000-0000-0000-0000-000000000000 à chaque fois
public Guid Id { get; set; } = new Guid();

// ✅ CORRECT — génère un nouvel UUID v4 unique
public Guid Id { get; set; } = Guid.NewGuid();

new Guid() appelle le constructeur sans paramètre, qui initialise chaque bit à zéro — identique à Guid.Empty et default(Guid). Chaque entité que vous créez reçoit la même clé primaire tout à zéro. La première insertion en base de données réussit. Toutes les suivantes rencontrent une violation de contrainte de clé primaire — ou pire, écrasent silencieusement le premier enregistrement si votre gestion d'erreurs n'est pas stricte.

Foire aux questions

Un GUID est-il identique à un UUID en C# ?
Oui. GUID (Globally Unique Identifier) est le terme Microsoft pour le standard UUID défini dans la RFC 4122. System.Guid.NewGuid() génère un UUID v4 — 128 bits de données aléatoires cryptographiques. La disposition binaire est identique ; seule la convention de nommage diffère entre l'écosystème .NET et le reste de l'industrie du logiciel.
Quelle est la différence entre Guid.NewGuid() et new Guid() ?
Guid.NewGuid() génère un UUID v4 aléatoire cryptographiquement unique à chaque appel. new Guid() — le constructeur sans paramètre — renvoie Guid.Empty, soit 00000000-0000-0000-0000-000000000000. Chaque appel produit la même valeur tout à zéro. C'est le bug lié aux GUID le plus courant en .NET : utiliser new Guid() là où vous vouliez Guid.NewGuid(), provoquant des conflits de clé primaire ou des identifiants en double dans chaque enregistrement.
Comment convertir un GUID en chaîne sans tirets ?
Appelez id.ToString("N"). Cela produit une chaîne hexadécimale minuscule de 32 caractères sans séparateur : 3f2504e04f8911d39a0c0305e82c3301. Pour la reconstituer, utilisez Guid.ParseExact(str, "N") — spécifier le format explicitement empêche toute ambiguïté et échoue rapidement avec une FormatException claire si l'entrée est incorrecte. L'onglet Valider de cet outil convertit également entre les formats de manière interactive.
Quel format de GUID utiliser dans une API REST JSON ?
Format D — la forme avec tirets par défaut. Appelez id.ToString() (sans argument) ou id.ToString("D"). System.Text.Json comme Newtonsoft.Json sérialisent les valeurs Guid au format D par défaut. C'est le format que chaque client HTTP de chaque langage peut analyser sans configuration supplémentaire. Utilisez le format N (sans tirets) uniquement pour intégrer un GUID dans un chemin d'URL afin d'éviter l'encodage des tirets.
Comment utiliser un GUID comme clé primaire dans Entity Framework Core ?
Déclarez la propriété de clé comme Guid. EF Core génère automatiquement un nouveau GUID à l'insertion en utilisant ValueGeneratedOnAdd() — aucune configuration requise. Pour SQL Server, envisagez d'utiliser HasDefaultValueSql("NEWSEQUENTIALID()") pour générer des GUID séquentiels au niveau de la base de données, ce qui élimine la fragmentation d'index B-tree causée par les valeurs UUID v4 aléatoires. Sur .NET 9+, Guid.CreateVersion7() fournit des UUID séquentiels dans le code applicatif sans configuration côté base de données.
Dois-je utiliser Guid.NewGuid() ou UUID v7 pour les clés primaires en 2026 ?
Pour les clés primaires de base de données, UUID v7 est le meilleur choix : son préfixe de timestamp en millisecondes maintient les valeurs en ordre croissant, ce qui élimine la fragmentation d'index B-tree et réduit de moitié la taille de l'index sur SQL Server. Guid.CreateVersion7() est intégré à .NET 9+. Pour les jetons, les identifiants de session et tout identifiant exposé à l'extérieur, utilisez Guid.NewGuid() (UUID v4) — il ne révèle aucune information temporelle, ce qui le rend plus sûr pour les contextes sensibles à la sécurité.
