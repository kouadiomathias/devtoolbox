UUID v4 vs v7 vs v1 vs ULID vs NanoID — choisir le bon identifiant

  
Ces cinq types d'identifiants résolvent le même problème : générer un identifiant unique sans coordination entre les serveurs. Mais ils font des compromis différents entre le hasard, la triabilité, la taille et la performance en base de données.

UUID v7 — pourquoi c'est important pour les bases de données

  
UUID v4 est purement aléatoire. Cela signifie que les insertions dans un index B-tree (que possède chaque clé primaire) sont aléatoires — chaque nouvelle ligne atterrit à un emplacement imprévisible dans l'index, provoquant des divisions de pages et une fragmentation constantes. À 100 millions de lignes, cela peut rendre les insertions d'UUID v4 3× plus lentes que les insertions d'entiers séquentiels.

  
UUID v7 intègre le timestamp Unix actuel dans les 48 premiers bits. Les nouvelles lignes se trient toujours après les existantes (dans la même milliseconde, elles se trient aléatoirement). Cela rend les insertions d'index presque séquentielles — beaucoup plus proches de la performance BIGINT tout en conservant tous les avantages d'un identifiant unique distribué.

  
// Structure UUID v7
018f4d1a-3b2c-7def-9abc-123456789abc
└──────────────┘ └──────────────────┘
   48 bits Unix        74 bits aléatoires
   timestamp (ms)     (bits de version + variant)


  
crypto.randomUUID() vs le paquet npm uuid

  
Utilisez crypto.randomUUID() lorsque vous n'avez besoin que d'UUID v4 et que vous ciblez des environnements modernes (tous les navigateurs, Node.js 19+). Il ne nécessite aucune dépendance et est l'option la plus rapide — implémenté nativement en C++ dans V8.

  
Utilisez le paquet npm uuid lorsque vous avez besoin d'autres versions d'UUID (v1, v3, v5, v7), lorsque vous devez prendre en charge des versions plus anciennes de Node.js (v10-v18), ou lorsque vous avez besoin d'UUID déterministes (v3/v5 utilisent un hachage espace de noms + nom).

  
// crypto.randomUUID() — aucune installation nécessaire
const id = crypto.randomUUID();

// uuid npm — toutes les versions
import { v4, v7 } from 'uuid'; // npm install uuid
const idV4 = v4();
const idV7 = v7(); // triable dans le temps, recommandé pour les bases de données


  
ULID vs UUID — quand choisir ULID

  
ULID (Universally Unique Lexicographically Sortable Identifier) fait 26 caractères en Base32 Crockford. Contrairement à UUID, il se trie correctement en tant que chaîne simple sans analyse spéciale. Cela le rend utile lorsque vous voulez des identifiants ordonnés chronologiquement qui fonctionnent dans les chemins d'URL, les noms de fichiers et les comparaisons de chaînes sans avoir besoin d'un type UUID spécial dans votre base de données.

  
// ULID se trie chronologiquement en tant que chaîne
const ids = ['01HZBQ7G4K...', '01HZBQ7G5M...', '01HZBQ7G7X...'];
ids.sort(); // ← déjà dans l'ordre de création ! Le tri UUID v4 est aléatoire.


  
Math.random() est-il sûr pour la génération d'UUID ?

  
Non. Math.random() est un générateur de nombres pseudo-aléatoires — il est déterministe et prévisible si la graine est connue. Les UUID générés avec Math.random() peuvent entrer en collision à des volumes bien plus faibles que ce que leur entropie de 122 bits suggère. Utilisez toujours crypto.getRandomValues() ou crypto.randomUUID(), qui utilisent le CSPRNG du système d'exploitation (générateur de nombres pseudo-aléatoires cryptographiquement sécurisé).


  
Foire aux questions

  

    

      Quelle est la différence entre UUID v4 et UUID v7 en JavaScript ?
      

        
UUID v4 est purement aléatoire (122 bits d'entropie) et n'a pas d'ordre temporel. UUID v7 (RFC 9562, 2024) intègre le timestamp Unix actuel en millisecondes dans les 48 premiers bits, ce qui le rend triable dans le temps. Pour les clés primaires de base de données, v7 est nettement meilleur car les insertions séquentielles provoquent beaucoup moins de fragmentation d'index B-tree par rapport aux insertions aléatoires de v4. Le paquet npm uuid (v9+) prend en charge les deux.

      

    

    

      Dois-je utiliser crypto.randomUUID() ou le paquet npm uuid ?
      

        
Utilisez crypto.randomUUID() pour UUID v4 si vous ciblez Node.js 19+ ou les navigateurs modernes — il ne nécessite aucune dépendance et est l'option la plus rapide. Utilisez le paquet npm uuid si vous avez besoin d'UUID v7 (recommandé pour les bases de données), v1, v3 ou v5, ou si vous devez prendre en charge des versions plus anciennes de Node.js (v10-v18).

      

    

    

      Qu'est-ce qu'un ULID et quand l'utiliser à la place d'un UUID ?
      

        
ULID est un identifiant de 26 caractères encodé en Base32. Ses 10 premiers caractères encodent le timestamp en millisecondes et les 16 restants sont aléatoires. Contrairement à UUID, les ULID se trient lexicographiquement dans l'ordre de création en tant que chaînes simples — aucune analyse n'est nécessaire. Utilisez ULID lorsque vous voulez des identifiants compatibles URL qui se trient chronologiquement et fonctionnent dans des environnements où le type UUID n'est pas pris en charge.

      

    

    

      La génération d'UUID est-elle sûre côté client dans le navigateur ?
      

        
Oui. Tous les navigateurs modernes exposent window.crypto.getRandomValues() qui utilise le CSPRNG du système d'exploitation. crypto.randomUUID() utilise cette même source. N'utilisez jamais Math.random() pour la génération d'UUID — il n'est pas cryptographiquement sécurisé. Les UUID générés par cet outil ont les mêmes propriétés de sécurité que ceux générés côté serveur.
