Six mois après le lancement, notre table users avait 40 millions de lignes et chaque insertion prenait deux fois plus de temps qu'au premier jour. La table utilisait uuid.uuid4() comme clé primaire — le choix « évidemment correct », puisqu'il ne nécessitait aucune coordination entre les serveurs d'application. Personne ne s'était demandé ce qu'une valeur aléatoire de 128 bits fait à un index B-tree qui s'attend à des insertions à peu près séquentielles. La réponse : des divisions de page constantes, une localité de cache désastreuse et un index de clé primaire plus volumineux que la table elle-même. Nous n'avons pas corrigé cela en abandonnant les UUID — nous l'avons corrigé en comprenant à quoi sert réellement chaque version d'UUID et en choisissant la bonne pour le travail.

C'est le vide que cette boîte à outils et cet article sont conçus pour combler. Le module uuid de Python vous donne cinq façons de générer un UUID et au moins sept façons de le représenter sous forme de chaîne, et la documentation de la bibliothèque standard ne vous dit pas quelle combinaison choisir. Voici une référence rapide, suivie des modes de défaillance qui apparaissent réellement dans le code Python en production.

Règle générale : v4 pour les identifiants aléatoires généraux, v7 pour tout ce qui devient une clé primaire de base de données ou nécessite un ordre chronologique approximatif, v5 pour les identifiants déterministes dérivés d'un nom existant ou d'un identifiant externe, et v1 uniquement lorsque vous avez spécifiquement besoin de l'horodatage intégré et de l'information de nœud et que vous ne vous souciez pas de les divulguer.

Pièges courants des UUID en Python

01Clés primaires UUIDv4 et fragmentation d'index
Élevée

Chaque valeur uuid.uuid4() est constituée de 128 bits de données aléatoires. Lorsqu'elle est utilisée comme clé primaire/clusterisée, chaque insertion atterrit à un point aléatoire de l'arbre B au lieu de la fin, forçant des divisions de page et réduisant le cache d'écriture de la base de données. Sur des tables de dizaines de millions de lignes, cela ralentit mesurablement les insertions et gonfle l'index.

À éviter
python
import uuid class User(Base): id = Column(Uuid, primary_key=True, default=uuid.uuid4) # aléatoire, fragmente l'index
Correction
python
# uuid.uuid7() arrive dans la stdlib à partir de Python 3.14 ; utilisez le paquet # `uuid7` PyPI sur les versions antérieures. v7 intègre un horodatage donc les # valeurs sont à peu près monotones — les insertions restent près de la fin de l'index. class User(Base): id = Column(Uuid, primary_key=True, default=uuid.uuid7)
02UUIDv1 divulguant l'adresse MAC et l'horodatage
Élevée

UUIDv1 intègre l'horodatage de création et, sur de nombreux systèmes, un identifiant de nœud dérivé de l'adresse MAC directement dans la valeur. Toute personne pouvant analyser l'UUID peut extraire exactement quand il a été créé et déduire des informations sur la machine génératrice.

À éviter
python
session_id = str(uuid.uuid1()) # expose l'heure de création + le nœud à quiconque le décode
Correction
python
session_id = str(uuid.uuid4()) # ou uuid7() si l'ordre est utile — aucun ne divulgue d'info sur l'hôte
03Comparaison de chaînes au lieu de comparaison d'UUID
Moyenne

Les UUID peuvent arriver en majuscules, minuscules, avec ou sans tirets. Comparer des chaînes brutes échoue silencieusement lorsque la casse ou le formatage diffère, même si l'identifiant sous-jacent est le même.

À éviter
python
if request_id == stored_id: # échoue si l'un est en majuscules et l'autre pas ...
Correction
python
if uuid.UUID(request_id) == uuid.UUID(stored_id): # normalise les deux avant de comparer ...
04Stocker les UUID en VARCHAR
Moyenne

Une chaîne UUID avec tirets fait 36 octets ; la même valeur compactée en octets bruts fait 16. Utiliser VARCHAR(36) au lieu d'un type de colonne UUID natif double plus que le stockage pour cette colonne et ralentit les comparaisons d'index et les jointures.

Correction
python
# SQLAlchemy : utilisez le type natif Uuid, pas String(36) from sqlalchemy import Uuid id = Column(Uuid, primary_key=True)

Foire aux questions

Quelle est la différence entre uuid1, uuid4 et uuid7 en Python ?
uuid1() encode l'horodatage actuel et un identifiant de nœud, révélant quand et (souvent) où il a été créé. uuid4() est entièrement aléatoire sans information intégrée, au prix d'une mauvaise localité d'index de base de données lorsqu'il est utilisé comme clé primaire. uuid7() combine un horodatage en millisecondes avec des bits aléatoires, vous donnant des identifiants à peu près triables et adaptés aux index sans exposer d'identifiant de nœud.
Un UUID Python est-il vraiment garanti unique ?
Pas mathématiquement garanti, mais la probabilité de collision pour un UUIDv4 aléatoire est suffisamment faible pour être traitée comme unique en pratique : il vous faudrait environ un milliard d'UUID générés par seconde pendant environ 85 ans avant d'atteindre 50 % de chance d'une collision. Les versions déterministes (v3, v5) ne sont « uniques » que par rapport à leur espace de noms et à leur nom d'entrée.
À quoi sert uuid.NAMESPACE_DNS en réalité ?
C'est un UUID fixe et normalisé qui agit comme un sel pour uuid5(). Hacher le même nom contre le même UUID d'espace de noms produit toujours le même UUID de sortie, ce qui rend v5 déterministe. La bibliothèque standard fournit également NAMESPACE_URL, NAMESPACE_OID et NAMESPACE_X500 pour d'autres types d'identifiants.
Comment stocker efficacement un UUID dans PostgreSQL ou MySQL ?
Dans PostgreSQL, utilisez le type de colonne natif uuid, qui le stocke sur 16 octets et prend en charge l'indexation directement. MySQL n'a pas de type UUID natif avant les helpers UUID_TO_BIN()/BIN_TO_UUID() de la version 8.0, donc stockez-le en BINARY(16) et convertissez à la frontière plutôt que d'utiliser CHAR(36).
uuid.uuid7() est-il disponible dans ma version de Python ?
Il fait partie de la bibliothèque standard à partir de Python 3.14. Sur les versions antérieures, les paquets uuid7 ou uuid6 sur PyPI implémentent la même spécification avec une interface identique.
Comment convertir une chaîne hexadécimale sans tirets en objet UUID ?
uuid.UUID(hex=ma_chaine_hex) accepte à la fois les chaînes hexadécimales de 32 caractères avec ou sans tirets et les normalise en un objet UUID standard.
La génération d'UUID est-elle sûre entre les threads et le multiprocessing ?
Oui pour uuid4 — il repose sur la source aléatoire cryptographiquement sûre du système d'exploitation, qui est sûre pour les threads et les processus. uuid1 est également sûr mais peut théoriquement répéter la même valeur si la résolution de l'horloge système est plus grossière que le taux de génération sur le même nœud ; c'est rarement un problème pratique.
