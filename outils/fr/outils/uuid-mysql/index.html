Clés primaires UUID MySQL : UUID_TO_BIN, le drapeau d'échange et pourquoi c'est important

Sur cette page
  1. Deux lignes, au mauvais endroit
  2. Tableau de décision rapide
  3. Pièges courants des UUID MySQL
  4. FAQ
  5. Références

Une table utilisant VARCHAR(36) DEFAULT (UUID()) PRIMARY KEY semblait parfaite dans tous les tests — quelques lignes, des insertions instantanées. Puis deux nouvelles lignes sont arrivées en production, et au lieu d'apparaître à la fin de la table, elles se sont retrouvées au milieu. InnoDB a dû physiquement déplacer les lignes existantes pour faire de la place, car les nouveaux UUID étaient triés avant les lignes déjà sur le disque. Sur une petite table, c'est invisible. Sur une table avec un milliard de lignes, c'est l'index clusterisé qui se rééquilibre constamment, incapable de suivre. Passer la colonne en BINARY(16) n'a fait que réduire l'empreinte de stockage — la ligne atterrissait toujours au milieu de la table, car les octets bruts d'un UUID v1 n'augmentent pas de manière linéaire. La partie que tout le monde oublie est l'argument swap_flag de UUID_TO_BIN(), qui est ce qui rend réellement les insertions séquentielles à nouveau.

Cette page génère un UUID dans la disposition v1 propre à MySQL, vous montre les octets exacts que UUID_TO_BIN() stockerait avec et sans ce drapeau, et vous donne du SQL que vous pouvez coller directement dans une migration.

La fonction UUID() intégrée de MySQL ne produit que des UUID de version 1 (horodatage + nœud) — style RFC 4122. Il n'existe pas de fonction v4 native ; MySQL ne la fournit tout simplement pas.

Pièges courants des UUID MySQL

01Stocker en BINARY(16) sans le drapeau d'échange
Élevée

La disposition des octets d'un UUID v1 place time_low — la partie qui change le plus vite — en premier. Convertir en binaire sans réordonner disperse toujours les nouvelles lignes dans l'index clusterisé, car les premiers octets n'augmentent toujours pas de manière monotone dans le temps.

À éviter
sql
CREATE TABLE users ( id BINARY(16) PRIMARY KEY DEFAULT (UUID_TO_BIN(UUID())), -- pas de swap_flag : toujours fragmenté name VARCHAR(100) NOT NULL );
Correction
sql
CREATE TABLE users ( id BINARY(16) PRIMARY KEY DEFAULT (UUID_TO_BIN(UUID(), 1)), -- swap_flag=1 : time-high passe en premier, les insertions restent séquentielles name VARCHAR(100) NOT NULL ); -- toujours le relire avec le drapeau correspondant SELECT BIN_TO_UUID(id, 1) AS id, name FROM users;
02VARCHAR(36) au lieu de BINARY(16)
Moyenne

Avec le jeu de caractères utf8mb4, une clé primaire VARCHAR(36) peut avoir une longueur de clé d'index allant jusqu'à 146 octets (36 caractères × 4 octets/caractère, plus un préfixe de longueur). BINARY(16) est fixe à 16 octets — cette différence est dupliquée dans chaque index secondaire de la table, car InnoDB stocke la clé primaire dans chacun d'eux.

Correction
sql
-- préférez BINARY(16) à CHAR(36)/VARCHAR(36) chaque fois que vous contrôlez le schéma id BINARY(16) PRIMARY KEY DEFAULT (UUID_TO_BIN(UUID(), 1))
03Besoin d'un UUID v4 comme valeur par défaut MySQL
Moyenne

MySQL n'autorise qu'un petit ensemble de fonctions intégrées, suffisamment déterministes, dans une expression DEFAULT. Les fonctions stockées personnalisées et les composants installés — y compris tout générateur v4 écrit manuellement — lèvent ER_DEFAULT_VAL_GENERATED_NAMED_FUNCTION_IS_NOT_ALLOWED lorsqu'ils sont utilisés comme valeur par défaut de colonne.

Ne fonctionne pas
sql
-- échoue : les fonctions personnalisées/de composant ne peuvent pas être une expression DEFAULT id BINARY(16) DEFAULT (UUID_TO_BIN(my_uuid_v4()))
Correction
sql
-- MySQL 8.0.30+ : mode GIPK — laissez InnoDB gérer une PK auto_increment cachée, -- et conservez votre v4 généré par l'application comme simple colonne UNIQUE SET sql_generate_invisible_primary_key = 1; CREATE TABLE users ( uuid BINARY(16) NOT NULL UNIQUE, -- v4, généré dans le code applicatif name VARCHAR(100) NOT NULL ); -- INSERT INTO users (uuid, name) VALUES (UUID_TO_BIN(?), ?); -- ? = v4 généré par l'application
04Considérer UUID() comme sûr à exposer publiquement
Élevée

Un UUID v1 encode l'horodatage exact de création et un identifiant de nœud dérivé de l'adresse MAC du serveur. Renvoyer des valeurs brutes de UUID() dans une réponse API divulgue les deux à quiconque peut analyser un UUID.

Correction
sql
-- conservez UUID() uniquement pour les clés primaires internes et ordonnées. -- pour tout ce qui est visible par l'utilisateur (jetons d'API, identifiants publics), générez un v4 -- dans le code applicatif et stockez-le séparément si nécessaire.

Foire aux questions

Quelle version d'UUID la fonction UUID() de MySQL produit-elle ?
Toujours la version 1 — horodatage plus un identifiant de nœud dérivé de la MAC, conformément à la RFC 4122. MySQL n'a pas de fonction intégrée pour les UUID de version 4 (aléatoires).
Que fait réellement l'argument swap_flag de UUID_TO_BIN ?
Il permute les groupes hexadécimaux time-low et time-high de l'UUID avant la conversion en binaire, de sorte que les bits d'horodatage les plus significatifs arrivent en premier. Cela fait que les UUID successifs se comparent comme des valeurs croissantes, ce qui maintient les insertions InnoDB à la fin de l'index clusterisé au lieu de les disperser.
Une clé primaire UUID est-elle plus lente qu'AUTO_INCREMENT dans MySQL ?
Oui, si c'est un UUID v1 ou v4 brut stocké sans ordre — les positions d'insertion aléatoires provoquent des divisions de page et un rééquilibrage. Avec BINARY(16) et le drapeau d'échange, les performances d'insertion se rapprochent de celles d'AUTO_INCREMENT, bien que la clé fasse toujours 16 octets contre 4 ou 8 pour un entier.
Puis-je utiliser UUID() comme valeur DEFAULT dans MySQL ?
Oui — depuis MySQL 8.0.13, les expressions par défaut comme DEFAULT (UUID_TO_BIN(UUID(), 1)) sont prises en charge directement dans CREATE TABLE, sans avoir besoin d'un trigger. Cela fonctionne parce que UUID() et UUID_TO_BIN() sont des fonctions intégrées ; les fonctions personnalisées ne sont pas autorisées de la même manière.
Comment obtenir de vraies valeurs UUID v4 dans MySQL ?
Générez-les dans le code applicatif (chaque langage courant possède un générateur v4 sécurisé) et passez-les à l'insertion. Si vous voulez tout de même qu'InnoDB gère une clé primaire séquentielle efficace, activez le mode GIPK et stockez le v4 comme une simple colonne UNIQUE BINARY(16) au lieu de la clé primaire.
Mon UUID est-il généré sur un serveur quelque part ?
Non — tout sur cette page s'exécute dans votre navigateur via JavaScript. Les octets du nœud sont aléatoires localement et ne reflètent aucune adresse matérielle réelle.
