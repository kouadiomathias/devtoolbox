1. Basic Structure and Operators per Field

The standard syntax (Vixie cron, cronie) consists of five fields: minute (0-59), hour (0-23), day of month (1-31), month (1-12), day of week (0-7, 0 and 7 both mean Sunday). Extended implementations (Quartz, AWS CloudWatch) add a sixth field (seconds) at the beginning. Kubernetes CronJob uses 5 fields but supports an explicit timezone since version 1.25.

2. The OR Behavior Trap (Day of Month + Day of Week)

This is one of the least documented pitfalls. When both day of month and day of week are specified (neither is *), cron runs the job when either matches — not both. Concrete example: 

0 0 1 * 1

This expression runs on the 1st of every month AND every Monday, not just the first Monday of the month. If you only wanted the first Monday, you'd need Quartz's # or L operator, or a check inside the script. To avoid this surprising behavior, always leave one of the two fields as * or use ? in a Quartz environment.

3. Textual Names for Months and Days

Linux cron accepts abbreviations: MON, TUE, WED for days, and JAN, FEB for months. This makes expressions more readable: 

0 9 * JAN-MAR MON-FRI

Note that these names are in English and only work with traditional cron (Vixie, cronie). Neither Quartz nor Kubernetes supports them. If you generate an expression for multiple environments, prefer numeric values.

5. Timezone Handling — In Detail

A task scheduled at 0 2 * * * runs at 2 AM — but in which timezone? The answer depends on the environment.

  • Classic Linux: cron uses the system time. Check with timedatectl or date +%Z. You can set CRON_TZ=Europe/Paris at the top of the crontab (cronie only).
  • AWS CloudWatch: always UTC. No variable possible. If you need execution at 8 AM Paris time in summer, write 0 6 * * ? * (UTC+2).
  • Kubernetes CronJob (≥1.25): specify .spec.timeZone: "Europe/Paris". The cron syntax remains 5 fields; the time is interpreted in that timezone.
  • Daylight Saving Time pitfall: a job at 0 2 * * * may run twice (in autumn) or never (in spring) if the timezone switches between CET and CEST. Prefer hours like 3:00 AM or use CRON_TZ with UTC to avoid this.

6. Overlap Behavior

If a cron job takes 10 minutes and is scheduled every 5 minutes, standard cron launches a new instance without waiting for the previous one to finish. This can cause dozens of concurrent processes. Solutions:

  • Linux: use flock to lock execution: flock -n /tmp/myjob.lock /path/script.sh
  • Ubuntu: the run-one command (run-one package) ensures only one instance runs.
  • Kubernetes: set .spec.concurrencyPolicy: Forbid in the CronJob.

7. Environment Variables in the Crontab

The crontab accepts variables in the header. This is essential because the default PATH in cron differs from the interactive shell — the source of 90% of "my script works manually but not in cron" problems. 

SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin
MAILTO=admin@example.com
CRON_TZ=Europe/Paris

*/10 * * * * /opt/scripts/my_job.sh

To disable system emails (standard output), set MAILTO="". Always redirect output to a file: >> /var/log/my_job.log 2>&1.

8. Cron vs systemd Timers — When to Switch

systemd timers offer a modern alternative with centralized journaling, service dependencies and catch-up after reboot. Direct comparison: 

# Cron: every 5 minutes
*/5 * * * *

# equivalent systemd timer
OnCalendar=*:0/5

Advantages of systemd timers:

  • RandomizedDelaySec=30 — adds a random delay to avoid load spikes.
  • Persistent=true — runs missed tasks after a reboot.
  • Logs centralized in journalctl -u my-timer.service.

If you manage modern machines (systemd), consider migrating complex tasks to timers. For simple tasks, cron remains perfectly suitable.

9. Debugging a Silent Cron Job — Ultimate Checklist

When a cron job doesn't run, no error appears by default. Here are the diagnostic steps:

  1. Check logs: grep CRON /var/log/syslog or /var/log/cron. Look for the line corresponding to your command.
  2. Distinguish user crontab vs root: crontab -l (current user) and sudo crontab -l (root) are two separate files.
  3. Redirect output: add >> /tmp/my_job.log 2>&1 at the end of the line to capture stdout and stderr.
  4. Check permissions: the script must be executable (chmod +x script.sh) and accessible within the PATH defined in the crontab.
  5. Test the PATH: run your script in a minimal environment: env -i /bin/bash -l -c "/path/script.sh". This simulates the cron environment.
  6. Enable MAILTO: MAILTO=you@email.com at the top of the crontab. Any error output will be sent by email (if the mail service is configured).
  7. Use the cron generator: our validation tool checks syntax and shows the next executions, helping to spot scheduling errors.

FAQ

How do I write a cron expression for every 5 minutes?

*/5 * * * * in standard cron. Each field: minute (every 5), hour (every), day of month (every), month (every), day of week (every).

What is the difference between Linux cron and AWS cron?

Linux uses 5 fields, AWS CloudWatch (and Quartz) can use 6 fields with seconds. Our generator handles both modes.

Why is my cron job not running?

Check PATH, permissions, redirect output to a file, and look at /var/log/syslog. Use MAILTO to receive errors.

How to prevent a cron job from overlapping?

Use flock, run-one or the concurrencyPolicy: Forbid property in Kubernetes.
