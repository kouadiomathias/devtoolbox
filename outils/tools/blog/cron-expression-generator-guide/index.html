The backup that ran every Monday, for three weeks on the wrong server

A team I worked with had a cron job set up to run database backups on the first Monday of each month. The expression was:

0 2 1 * 1

The intent was clear: "at 2 AM, on the 1st, on Monday." The actual behavior was different. For three weeks, the job ran every Monday at 2 AM — not just the first one. On months where the 1st happened to fall on a Monday, it also ran on that day. Some months it ran five times. The storage filled. The alert fired. The root cause took two hours to find.

The culprit was a single line of documentation nobody had read: when both day-of-month and day-of-week are specified, standard cron uses OR logic, not AND. The job ran when either field matched, not when both did simultaneously.

This guide is built around the real traps, the ones that look correct, run silently, and only reveal themselves when something breaks in production. We start with the OR trap, cover every operator, every platform difference, every timezone edge case, and end with a debugging checklist you can run through in under ten minutes.

The 5 fields and what each one actually controls

A standard cron expression has five fields separated by spaces. Each field controls one unit of time. Here is the exact anatomy:

*
Minute
0–59
*
Hour
0–23
*
Day of month
1–31
*
Month
1–12
*
Day of week
0–7 (0=7=Sun)

A few things that trip up everyone at least once:

  • Sunday is both 0 and 7 in standard cron. 0 0 * * 0 and 0 0 * * 7 both mean "midnight on Sunday." Some implementations only accept one — always test both if portability matters.
  • Month and day-of-week accept textual names in cronie: MON, TUE... and JAN, FEB... These are case-insensitive and English-only. Quartz and Kubernetes do not support them.
  • The minimum unit is one minute. There is no native way to run a cron job every 30 seconds on standard cron. Use */1 * * * * /script.sh; sleep 30; /script.sh inside the cron command, or switch to a systemd timer which supports second-level precision.
# Every minute
* * * * *

# Every day at midnight
0 0 * * *

# Every weekday (Monday to Friday) at 8:30 AM
30 8 * * 1-5

# First day of every month at 6 AM
0 6 1 * *

# Every 15 minutes between 9 AM and 5 PM on weekdays
*/15 9-17 * * 1-5

# At 2:30 AM every Sunday
30 2 * * 0

Operators: *, ,, -, /, ?, L, W, #

*Wildcard all values
Universal

Matches every possible value for that field. * * * * * runs every minute of every hour of every day.

0 * * * *    # Every hour on the hour (minute=0, any hour, any day)
,List multiple values
Universal

Specifies a list of discrete values. Order does not matter cron evaluates the union of all values.

0 9,12,17 * * 1-5   # At 9 AM, noon, and 5 PM on weekdays
-Range — consecutive values
Universal

Specifies a continuous range, inclusive on both ends.

0 9 * JAN-MAR MON-FRI   # 9 AM on weekdays in January, February, March
/Step every N units
Universal

Runs at regular intervals. */5 means "every 5 units." You can combine with a range: 0-30/5 means "every 5 minutes within the first 30 minutes of each hour."

*/5 * * * *      # Every 5 minutes
0-30/10 * * * *  # At minutes 0, 10, 20, 30 of every hour
QNo specific value Quartz/AWS only
Quartz / AWS

Used in day-of-month OR day-of-week to explicitly say "I don't care about this field." This is the correct way to avoid the OR trap in Quartz environments — set one field to a value and the other to ?.

# Quartz: run every Monday (not every 1st AND every Monday)
0 0 12 ? * MON

# Quartz: run on the 1st of every month (not on Mondays too)
0 0 12 1 * ?
LLast Quartz/AWS only
Quartz / AWS

In day-of-month: the last day of the month (handles February and months with 30 days automatically). In day-of-week: the last occurrence of a day in the month. 5L = last Friday.

# Quartz: last day of every month at midnight
0 0 0 L * ?

# Quartz: last Friday of every month at 6 PM
0 0 18 ? * 6L
#Nth weekday — Quartz only
Quartz only

The Nth occurrence of a given weekday in the month. 2#1 = first Monday (day 2 of week = Monday in Quartz, where 1=Sunday). This is the correct way to schedule "the first Monday of the month" without hitting the OR trap.

# Quartz: first Monday of every month at noon
0 0 12 ? * 2#1

# Quartz: third Wednesday of every month at 9 AM
0 0 9 ? * 4#3
Quartz day numbering: In Quartz, 1=Sunday, 2=Monday, 3=Tuesday, 4=Wednesday, 5=Thursday, 6=Friday, 7=Saturday. This differs from Linux where 1=Monday. Always double-check when migrating between platforms.

The OR trap the most misunderstood cron behavior

⚠Day-of-month OR day-of-week not AND
Critical wrong behavior

When both the day-of-month field and the day-of-week field contain a non-wildcard value, cron schedules the job when either condition is true. This is by design — and it's documented in the POSIX spec — but it surprises almost everyone who encounters it for the first time.

Trap this runs far more often than expected
# Intended: "run at midnight on the first Monday of each month"
# Actual: "run at midnight on the 1st of every month, AND on every Monday"
0 0 1 * 1

# In a month with 5 Mondays, this runs 6 times (5 Mondays + the 1st)
# In a month where the 1st IS a Monday, the 1st is not counted twice
Fix 1 leave one field as * (for simple cases)
# Run only on every Monday (ignore day-of-month entirely)
0 0 * * 1

# Run only on the 1st of every month (ignore day-of-week entirely)
0 0 1 * *
Fix 2 use Quartz # operator (for "Nth weekday of month")
# Quartz: genuinely the first Monday of every month, nothing else
0 0 0 ? * 2#1
Fix 3 check inside the script (for standard cron)
#!/bin/bash
# Run via: 0 0 * * 1 /path/first_monday_check.sh
# This script checks whether today is the first Monday of the month
day=$(date +%d)
if [ "$day" -le 7 ]; then
    # Only the first 7 days of the month can contain the first Monday
    /path/to/actual_job.sh
fi
The rule: In standard cron, if you need to combine day-of-month and day-of-week with AND logic, you must enforce it inside the script. The cron expression itself cannot express AND — only OR.

Timezones and DST why your 2 AM job runs at 3 AM or twice

🕐DST the clock that breaks your schedule
High silent wrong timing

Twice a year, Daylight Saving Time shifts local clocks by one hour. If your cron daemon uses a local timezone and your job is scheduled during the transition window, two things can happen:

  • Spring (clocks spring forward): The hour from 2:00 AM to 3:00 AM doesn't exist. A job at 0 2 * * * silently skips that day.
  • Autumn (clocks fall back): The hour from 2:00 AM to 3:00 AM exists twice. A job at 0 2 * * * runs twice that night.
Risky depends on system timezone and DST transitions
# If system timezone is Europe/Paris:
0 2 * * *
# → skips entirely on last Sunday of March
# → runs twice on last Sunday of October
Fix schedule in UTC, convert to local time in your app
# At the top of your crontab (cronie only):
CRON_TZ=UTC

# Now this runs at 2:00 AM UTC regardless of DST
0 2 * * *

# In Europe/Paris (UTC+1 winter, UTC+2 summer):
# → Winter: 3:00 AM Paris time
# → Summer: 4:00 AM Paris time
# This is predictable and repeatable. DST shifts affect the local display, not the actual run time.
Platform-specific timezone settings
# Linux cronie CRON_TZ in crontab header
CRON_TZ=America/New_York
0 9 * * 1-5   # 9 AM New York time, DST-aware

# Kubernetes CronJob (≥ 1.25) spec.timeZone
spec:
  schedule: "0 9 * * 1-5"
  timeZone: "America/New_York"   # IANA timezone name

# AWS CloudWatch always UTC, no override possible
# 9 AM New York in summer (UTC-4): 13:00 UTC
0 13 * * ? *   # summer
# 9 AM New York in winter (UTC-5): 14:00 UTC
0 14 * * ? *   # winter you need two separate rules, or use Lambda with timezone logic
Best practice: Store and run all cron jobs in UTC. Convert to local time only in your application or notification layer. This eliminates DST bugs completely and makes schedules portable across regions.

Overlap: what happens when a job takes longer than its interval

Standard cron has no built-in awareness of how long a job runs. If a job scheduled every 5 minutes takes 8 minutes, by the time it finishes, two more instances have already started. On a busy server this can cascade into dozens of concurrent processes within an hour.

🔒Preventing overlap three approaches
High server stability
Linux — flock (most reliable)
# flock acquires an exclusive lock on a file before running the script
# -n means "non-blocking" if the lock is held, skip this run
*/5 * * * * flock -n /tmp/myjob.lock /path/to/script.sh

# With a timeout (wait up to 10 seconds before giving up):
*/5 * * * * flock -w 10 /tmp/myjob.lock /path/to/script.sh
Ubuntu run-one (simplest)
# run-one ensures only one instance of a command runs at a time
# install: sudo apt install run-one
*/5 * * * * run-one /path/to/script.sh
Kubernetes CronJob concurrencyPolicy
spec:
  schedule: "*/5 * * * *"
  concurrencyPolicy: Forbid   # Skip new run if previous is still running
  # Other options:
  # Allow    default, multiple instances can run simultaneously
  # Replace  cancel the running job, start a new one
Script-level PID file (portable, no dependencies)
#!/bin/bash
PIDFILE=/tmp/myjob.pid

if [ -f "$PIDFILE" ]; then
    pid=$(cat "$PIDFILE")
    if kill -0 "$pid" 2>/dev/null; then
        echo "Job already running (PID $pid). Skipping."
        exit 0
    fi
fi

echo $$ > "$PIDFILE"
trap "rm -f $PIDFILE" EXIT

# Your actual job below
/path/to/actual_work.sh

Environment variables that make or break cron jobs

The number one reason a cron job fails silently while working perfectly in your terminal is the PATH. The cron environment is minimal by design — it has none of your shell aliases, none of your .bashrc exports, and a PATH that typically only includes /usr/bin:/bin.

⚙The production-safe crontab header
Best practice 
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
MAILTO=ops@example.com
CRON_TZ=UTC
HOME=/home/deploy

# Format: minute hour day-of-month month day-of-week command
*/10 * * * * /opt/scripts/sync.sh >> /var/log/sync.log 2>&1
0 2 * * * /opt/scripts/backup.sh >> /var/log/backup.log 2>&1
  • SHELL specifies the shell used to run commands. Default is /bin/sh, which may lack bash features.
  • PATH list every directory where your scripts and their dependencies live. When in doubt, use absolute paths in your commands.
  • MAILTO any output that is not redirected to a file is emailed to this address. Set to "" to suppress emails entirely.
  • CRON_TZ timezone for this crontab (cronie only). Use IANA names like America/New_York or UTC.
How to simulate the cron environment manually
# This runs your script in an environment close to what cron uses
# Use this to reproduce "works in terminal, fails in cron" bugs
env -i HOME=/root SHELL=/bin/bash PATH=/usr/bin:/bin /bin/bash -l -c "/path/to/script.sh"

Cron vs systemd timers when to switch

systemd timers are the modern replacement for cron on Linux systems that run systemd (which is most distributions since 2015). They offer several features that cron simply can't provide, but they're also more complex to set up for simple tasks.

Featurecronsystemd timer
Setup complexity✅ One line in crontab❌ Two files (.timer + .service)
Log access❌ /var/log/syslog only✅ journalctl -u myservice
Catch-up after reboot❌ Missed runs are lost✅ Persistent=true
Random delay to spread load❌ Not built-in✅ RandomizedDelaySec=
Service dependencies❌ Not supported✅ After=, Requires=
Second-level precision❌ Minute minimum✅ OnCalendar=*:*:0/30
Portability✅ Any Unix❌ systemd only
Multi-user crontabs✅ Each user has their own❌ System-level only by default
→Equivalent expressions
Migration reference 
# cron: every 5 minutes
*/5 * * * * /opt/scripts/sync.sh

# systemd timer equivalent: /etc/systemd/system/sync.timer
[Unit]
Description=Sync every 5 minutes

[Timer]
OnCalendar=*:0/5
Persistent=true          # Run missed jobs after reboot
RandomizedDelaySec=30    # Spread load: run anywhere in a 30s window

[Install]
WantedBy=timers.target
# The matching service file: /etc/systemd/system/sync.service
[Unit]
Description=Sync service

[Service]
Type=oneshot
ExecStart=/opt/scripts/sync.sh
User=deploy
# Enable and start the timer
systemctl enable --now sync.timer

# Check status and next run time
systemctl list-timers --all | grep sync

# View logs
journalctl -u sync.service -f
Recommendation: Use cron for simple, portable, or per-user scheduled tasks. Switch to systemd timers when you need logging, catch-up runs, dependencies, or second-level precision on a managed Linux server.

Debugging checklist step by step when nothing runs

When a cron job doesn't run, there's no error by default. The following checklist covers every common cause in order of frequency.

🔍The 7-step cron debugging process
Start here
Step 1 Is the cron daemon running?
systemctl status cron      # Debian/Ubuntu
systemctl status crond     # RHEL/CentOS
pgrep -x cron || pgrep -x crond   # Check the process exists
Step 2 Check the system logs
# Ubuntu/Debian
grep CRON /var/log/syslog | tail -30

# RHEL/CentOS
grep CRON /var/log/cron | tail -30

# Look for: "(deploy) CMD (/opt/scripts/sync.sh)"
# If the line appears → cron ran it, the problem is in the script
# If the line is absent → the expression, user, or daemon is the issue
Step 3 Redirect all output to a file
# Add to your cron line:
*/5 * * * * /opt/scripts/sync.sh >> /tmp/sync.log 2>&1
# 2>&1 redirects both stdout and stderr to the same file
# Without this, output is silently discarded or emailed (if mail is configured)
Step 4 Check which crontab you're editing
crontab -l           # Current user's crontab
sudo crontab -l      # Root's crontab (completely separate file)
cat /etc/cron.d/*    # System-wide cron jobs
Step 5 Verify permissions and PATH
ls -la /opt/scripts/sync.sh    # Must be executable
chmod +x /opt/scripts/sync.sh  # Fix if missing

# Test with an absolute path in the cron line
*/5 * * * * /usr/bin/python3 /opt/scripts/sync.py >> /tmp/sync.log 2>&1
Step 6 Simulate the cron environment
# Run this in the terminal to reproduce the exact cron environment
env -i HOME=/home/deploy USER=deploy LOGNAME=deploy \
  SHELL=/bin/bash PATH=/usr/bin:/bin \
  /bin/bash -c '/opt/scripts/sync.sh' 2>&1
Step 7 Enable email notifications
# Add to the top of your crontab:
MAILTO=you@example.com
# Any unredirected output (including error messages) will be emailed
# Requires a local mail service (postfix, sendmail) to be running

Frequently asked questions

How do I write a cron expression that runs every 5 minutes?

Use */5 * * * * on standard Linux cron. The / operator means "every N units" and * means "all possible values", so */5 in the minute field means "at minute 0, 5, 10, 15, 20, 25, 30, 35, 40, 45, 50, 55." To start at a specific minute, use a range: 2-59/5 runs at minutes 2, 7, 12, 17, etc.

Why does my cron job fire on wrong days when I set both day-of-month and day-of-week?

Standard cron applies OR logic when both day-of-month and day-of-week are non-wildcard values. The expression 0 0 1 * 1 means "run when it is the 1st OR when it is Monday" — not both simultaneously. To get true AND logic, either leave one field as *, or add a conditional check inside your script (check if the date meets both conditions before running the actual work). In Quartz environments, use the # operator or ? to be explicit about which field matters.

What is the difference between Linux cron and Quartz / AWS cron?

Linux cron uses 5 fields (minute, hour, day-of-month, month, day-of-week). Quartz (used in Java frameworks and Spring) adds a seconds field at the beginning, making 6 fields minimum, plus an optional year field (7 fields total). Quartz also adds the L, W, and # operators for "last day", "nearest weekday", and "Nth weekday" respectively. AWS CloudWatch Events uses a 6-field format similar to Quartz but with some differences: it always runs in UTC, uses ? differently, and requires you to explicitly specify ? in one of the day fields when the other is set.

Why does my cron job run twice in autumn or skip in spring?

This is a Daylight Saving Time bug. When a local timezone changes its UTC offset, the clock either jumps forward (spring, skipping an hour) or falls back (autumn, repeating an hour). If your cron job is scheduled during that transition window, it will be skipped or duplicated. The fix is to run all cron jobs in UTC: add CRON_TZ=UTC to the top of your crontab in cronie, or set the timezone in your Kubernetes CronJob spec. Then calculate the correct UTC time manually, accounting for the UTC offset of your target timezone.

Why isn't my cron job running even though the expression is correct?

The most common cause is a PATH problem. The cron environment only has a minimal PATH (/usr/bin:/bin), so commands that work in your terminal may not be found by cron. Add PATH=/usr/local/bin:/usr/bin:/bin at the top of your crontab, or use absolute paths for every command in your cron lines. Also check: the script has execute permission (chmod +x), the cron daemon is running (systemctl status cron), you're editing the correct user's crontab (crontab -l vs sudo crontab -l), and output is redirected (>> /tmp/job.log 2>&1) so errors are visible.

How do I prevent a cron job from overlapping with itself?

Standard cron starts a new instance regardless of whether the previous one is still running. On Linux, the most reliable solution is to wrap your command with flock: flock -n /tmp/myjob.lock /path/to/script.sh. The -n flag means "skip this run if the lock is held" — no queuing, no waiting. On Ubuntu, the run-one package provides a simpler alternative. In Kubernetes, set spec.concurrencyPolicy: Forbid in your CronJob manifest.

When should I use a systemd timer instead of cron?

Choose systemd timers when you need centralized logging (via journalctl), catch-up runs after a reboot (Persistent=true), service dependencies (After=network.target), second-level scheduling precision, or a random delay to avoid load spikes (RandomizedDelaySec). Stick with cron when you need portability across Unix systems without systemd, per-user scheduled tasks, or when you already have a large crontab that works well and doesn't need the extra features.
