The script that ran for two months on the wrong server

A team I know was migrating a Django application from their old VPS to a new cloud server. They moved the code, migrated the database, updated the DNS. The deployment checklist was thorough. One thing they didn't check: the crontab on the old server.

For two months after the migration, a nightly backup script kept running on the decommissioned server — pulling live data via an API token that had never been revoked, storing compressed archives in a directory nobody was watching, running up storage bills on a machine they thought was idle. When they finally noticed (a billing spike), they found 61 gigabytes of backups on a server that wasn't supposed to exist anymore.

The cron job itself was fine. The expression was correct. The script worked perfectly. What failed was the deployment process around it: no documentation of what cron jobs existed, no alert when the job ran, no log anyone was reading.

This guide is about the full lifecycle of a cron job not just writing the expression, but writing a solid script, testing it safely before it runs in production, and setting up monitoring so you know when it works and when it doesn't.

This guide is about building and deploying real jobs. If you want to understand cron expression syntax in depth — operators, the OR trap, timezone DST pitfalls — read our companion article: Cron Expressions: The Complete Guide. The two articles cover different ground deliberately.

Step 1 Write a script that works in isolation

The first rule of cron scripts: never write logic directly in the cron line. Inline commands are impossible to test, impossible to version-control, and impossible to read six months later. Every cron job should call a script file.

Before you even think about the cron expression, write your script so it can run standalone from the terminal, by hand — and produce a clear log of what it did. Here is the shell script template I use for every new cron job:

🐚Production-ready shell script template
Copy and adapt 
#!/bin/bash
# ─────────────────────────────────────────────────────────────────
# Script : /opt/scripts/backup.sh
# Purpose: Nightly PostgreSQL backup to S3
# Cron   : 0 2 * * * /opt/scripts/backup.sh
# Owner  : deploy@myserver.com
# ─────────────────────────────────────────────────────────────────
set -euo pipefail   # Exit on error, unset variable, pipe failure

LOG=/var/log/backup.log
DATE=$(date '+%Y-%m-%d_%H-%M-%S')
BACKUP_DIR=/tmp/backups
S3_BUCKET=s3://my-company-backups/postgres

mkdir -p "$BACKUP_DIR"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG"
}

log "=== Backup started ==="

# 1. Dump database
log "Dumping database..."
pg_dump -U postgres mydb > "$BACKUP_DIR/mydb_$DATE.sql"

# 2. Compress
log "Compressing..."
gzip "$BACKUP_DIR/mydb_$DATE.sql"

# 3. Upload to S3
log "Uploading to S3..."
aws s3 cp "$BACKUP_DIR/mydb_$DATE.sql.gz" "$S3_BUCKET/"

# 4. Clean local temp file
rm -f "$BACKUP_DIR/mydb_$DATE.sql.gz"

log "=== Backup completed successfully ==="
exit 0

Three things this template does that matter in a cron context:

  • set -euo pipefail the script stops immediately if any command fails. Without this, a failed pg_dump silently continues to the upload step, uploading an empty or corrupt file.
  • Explicit logging with timestamps every meaningful action is logged with a timestamp. When you read the log at 9 AM and see the backup completed at 02:03:17, you know it worked. When you see the log ends at "Dumping database..." you know exactly where it died.
  • Header comment block the cron line, purpose, and owner are documented inside the script itself. When someone finds this script six months later, they know what it does and where it runs.
Always make the script executable before adding it to crontab: 
chmod +x /opt/scripts/backup.sh
# Verify
ls -la /opt/scripts/backup.sh
# → -rwxr-xr-x 1 deploy deploy 1234 Jun 20 08:00 /opt/scripts/backup.sh

Step 3 Add to crontab safely

Never edit /etc/crontab directly for user jobs. Use crontab -e it opens the crontab in a temporary file, validates the syntax on save, and only installs the new version if it's valid. A syntax error with direct file editing silently breaks the entire crontab.

⚙The complete production crontab structure
Production template 
# Open for editing (use 'crontab -e' as the user who owns the scripts)
crontab -e
# ══════════════════════════════════════════════════════
# Crontab for user: deploy
# Server: prod-web-01, Last updated: 2026-06-20
# ══════════════════════════════════════════════════════

# Environment
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
MAILTO=ops@example.com
CRON_TZ=UTC
HOME=/home/deploy

#  Backups ──────────────────────────────────────────
# Daily PostgreSQL backup at 2 AM UTC
0 2 * * * /opt/scripts/backup.sh >> /var/log/backup.log 2>&1

#  Maintenance ──────────────────────────────────────
# Clean log files older than 30 days every Sunday at 3 AM
0 3 * * 0 /opt/scripts/cleanup_logs.sh >> /var/log/cleanup.log 2>&1

#  Sync ─────────────────────────────────────────────
# Sync product catalog from API every 15 minutes during business hours
*/15 8-18 * * 1-5 /opt/scripts/sync_catalog.py >> /var/log/sync.log 2>&1

A few conventions that make a crontab readable six months later:

  • One comment per job — say what the job does in plain language, not just the schedule.
  • Group jobs by category backups, maintenance, sync, reports. Each group gets a header comment.
  • Always redirect both stdout and stderr — >> /var/log/job.log 2>&1 on every line. Unredirected output goes to MAILTO if configured, or is silently discarded.
  • Document the file at the top server name, user, last updated. When you SSH into an unfamiliar server at 2 AM, this metadata is invaluable.
Useful crontab commands
crontab -l              # List current user's crontab
crontab -e              # Edit current user's crontab
sudo crontab -l         # List root's crontab (separate file!)
crontab -l > ~/crontab.backup  # Export before making changes
crontab ~/crontab.backup       # Restore from backup

Step 4 Test before the first scheduled run

The most common failure mode in cron jobs is something that works perfectly in the terminal and silently fails in the cron context. The reason is almost always the environment: cron runs with a minimal PATH, no loaded shell profile, and a different working directory than your terminal session.

You can reproduce this exact environment before the job fires for the first time:

🧪Simulating the cron environment
Test before deploy
Method 1. env -i (closest to real cron)
# This strips your current environment and runs the script
# with only the variables cron would have
env -i HOME=/home/deploy \
       USER=deploy \
       LOGNAME=deploy \
       SHELL=/bin/bash \
       PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin \
       /bin/bash -c '/opt/scripts/backup.sh'

# Read the output carefully any "command not found" error
# means you need to use an absolute path or expand PATH in the crontab
Method 2. temporary cron line with a 2-minute delay
# Find the current time
date '+%M %H'   # e.g. "32 14" → 14:32

# Add a cron line that fires 2 minutes from now
# If current time is 14:32, set minute to 34
34 14 * * * /opt/scripts/backup.sh >> /tmp/test_backup.log 2>&1

# Wait 3 minutes, then read the log
cat /tmp/test_backup.log

# Remove the test line from crontab when done
crontab -e
Method 3. check cron actually dispatched the job
# After the expected run time, check the system cron log
grep CRON /var/log/syslog | grep backup | tail -5
# You should see a line like:
# Jun 20 02:00:01 server CRON[12345]: (deploy) CMD (/opt/scripts/backup.sh)
# If this line is absent → expression didn't match, or cron daemon isn't running
# If this line is present but the job failed → check /var/log/backup.log
The two-log rule: Always check two logs when debugging a cron job. First the system cron log (/var/log/syslog) to confirm the job was dispatched. Then your application log (/var/log/backup.log) to see what the script did. If the first log has the dispatch line but the second log is empty, the script crashed before writing anything — check permissions and the set -e behavior.

Step 5 Monitor and receive alerts

A cron job that runs silently every night is invisible until it breaks. The goal of monitoring is to make success visible and failure loud. You want to know immediately when a job fails — not when the backup is missing at 3 AM on a Friday night or when a product catalog sync has been failing for six weeks.

🔔Three monitoring layers implement all three
Don't skip this step
Layer 1 Log files (minimum viable)
# In your crontab: redirect everything to a log with a timestamp prefix
0 2 * * * /opt/scripts/backup.sh >> /var/log/backup.log 2>&1

# In your script: prefix every log line with a timestamp
log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" >> /var/log/backup.log; }

# Rotate logs to prevent disk fill (logrotate config)
# /etc/logrotate.d/cron-jobs
/var/log/backup.log {
    daily
    rotate 30
    compress
    missingok
    notifempty
}
Layer 2. Email on failure (simple but effective)
#!/bin/bash
# In your script: send an email if something goes wrong
MAILTO=ops@example.com
SCRIPT_NAME=$(basename "$0")

on_error() {
    local line=$1
    echo "ERROR in $SCRIPT_NAME at line $line — $(date)" | \
        mail -s "CRON FAILURE: $SCRIPT_NAME" "$MAILTO"
}
trap 'on_error $LINENO' ERR

# The rest of your script below
# If any command fails, the trap fires and sends an email
Layer 3 — Healthcheck ping (most reliable)
#!/bin/bash
# Ping a healthcheck URL when the job SUCCEEDS
# Services like healthchecks.io or BetterUptime alert you
# if the ping doesn't arrive within the expected window

HEALTHCHECK_URL="https://hc-ping.com/your-unique-uuid"

# Your job logic
pg_dump -U postgres mydb | gzip > /tmp/backup.sql.gz
aws s3 cp /tmp/backup.sql.gz s3://my-bucket/

# Signal success — if this line is never reached, the healthcheck
# will alert you after the expected run window passes
curl -fsS --retry 3 "$HEALTHCHECK_URL" > /dev/null 2>&1

# To signal failure explicitly:
# curl -fsS "$HEALTHCHECK_URL/fail" > /dev/null 2>&1
Healthcheck services like healthchecks.io (free tier available) work the opposite way from traditional monitoring: instead of checking if a service is running, they alert you when an expected ping doesn't arrive. This catches jobs that stop running silently — no code change, no error, just a job that was somehow removed from the crontab during a server rebuild.

4 real cron jobs with full code

1Nightly database backup to S3
Shell + AWS CLI
Cron line
0 2 * * * /opt/scripts/db_backup.sh >> /var/log/db_backup.log 2>&1
Script
#!/bin/bash
set -euo pipefail
DATE=$(date '+%Y-%m-%d')
DB_NAME=myapp_production
BUCKET=s3://my-company-backups

log() { echo "[$(date '+%H:%M:%S')] $*"; }

log "Starting backup of $DB_NAME"
pg_dump -U postgres "$DB_NAME" | gzip > /tmp/"$DB_NAME"_"$DATE".sql.gz
log "Dump complete. Uploading..."
aws s3 cp /tmp/"$DB_NAME"_"$DATE".sql.gz "$BUCKET"/postgres/
rm /tmp/"$DB_NAME"_"$DATE".sql.gz

# Delete backups older than 30 days from S3
aws s3 ls "$BUCKET"/postgres/ | awk '{print $4}' | \
  while read f; do
    age=$(( ($(date +%s) - $(date -d "${f:8:10}" +%s)) / 86400 ))
    [ "$age" -gt 30 ] && aws s3 rm "$BUCKET/postgres/$f"
  done

log "Backup complete and old files pruned"
curl -fsS "https://hc-ping.com/YOUR-UUID" > /dev/null 2>&1
2Python API sync every 15 minutes on weekdays
Python 3 + venv
Cron line
*/15 8-18 * * 1-5 /opt/venv/bin/python /opt/scripts/sync_products.py >> /var/log/sync.log 2>&1
Why /opt/venv/bin/python and not python3?

Using the venv's Python binary means all installed packages (requests, psycopg2, etc.) are available without activating the virtualenv. No source activate in the cron line — that requires a shell with profile loading, which cron doesn't do.

Script
#!/usr/bin/env python3
"""
Sync product catalog from supplier API.
Cron: */15 8-18 * * 1-5
"""
import requests, psycopg2, logging, sys
from datetime import datetime

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s %(levelname)s %(message)s',
    stream=sys.stdout
)
log = logging.getLogger(__name__)

API_URL = "https://api.supplier.com/v2/products"
API_KEY = "sk_live_xxxxx"   # Use env var in production: os.environ['SUPPLIER_API_KEY']
DB_DSN  = "postgresql://app:secret@localhost/myapp"

def sync():
    log.info("Fetching products from API...")
    r = requests.get(API_URL, headers={"Authorization": f"Bearer {API_KEY}"}, timeout=30)
    r.raise_for_status()
    products = r.json()["data"]
    log.info(f"Received {len(products)} products")

    conn = psycopg2.connect(DB_DSN)
    cur = conn.cursor()

    updated = 0
    for p in products:
        cur.execute("""
            INSERT INTO products (id, name, price, stock, updated_at)
            VALUES (%(id)s, %(name)s, %(price)s, %(stock)s, NOW())
            ON CONFLICT (id) DO UPDATE
            SET name=EXCLUDED.name, price=EXCLUDED.price,
                stock=EXCLUDED.stock, updated_at=NOW()
        """, p)
        updated += 1

    conn.commit()
    cur.close()
    conn.close()
    log.info(f"Sync complete {updated} products upserted")

if __name__ == "__main__":
    sync()
3Weekly log cleanup
Shell + find
Cron line
0 3 * * 0 /opt/scripts/cleanup_logs.sh >> /var/log/cleanup.log 2>&1
Script
#!/bin/bash
set -euo pipefail

LOG_DIRS=("/var/log/app" "/var/log/nginx" "/tmp/uploads")
MAX_AGE_DAYS=30

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*"; }
log "=== Log cleanup started ==="

total_freed=0
for dir in "${LOG_DIRS[@]}"; do
    if [ ! -d "$dir" ]; then
        log "Directory $dir not found skipping"
        continue
    fi

    # DRY RUN FIRST list files that would be deleted
    log "Files older than $MAX_AGE_DAYS days in $dir:"
    find "$dir" -name "*.log*" -mtime +"$MAX_AGE_DAYS" -type f | \
        while read f; do
            size=$(du -sh "$f" | cut -f1)
            log "  → $f ($size)"
        done

    # Actually delete
    count=$(find "$dir" -name "*.log*" -mtime +"$MAX_AGE_DAYS" -type f | wc -l)
    find "$dir" -name "*.log*" -mtime +"$MAX_AGE_DAYS" -type f -delete
    log "Deleted $count files from $dir"
done

log "=== Cleanup complete ==="
Always log the file list before deleting. The cleanup script above writes the name and size of every file it deletes. If something gets cleaned up by mistake, you at least know what it was and when it happened.
4Daily report email curl + Mailgun
Shell + curl
Cron line
0 8 * * 1-5 /opt/scripts/daily_report.sh >> /var/log/reports.log 2>&1
Script
#!/bin/bash
set -euo pipefail

MAILGUN_DOMAIN="mail.example.com"
MAILGUN_API_KEY="key-xxxxx"
RECIPIENT="team@example.com"
DATE=$(date '+%A %B %d, %Y')

log() { echo "[$(date '+%H:%M:%S')] $*"; }

# Build report body from database query
log "Generating report..."
REPORT=$(psql -U postgres myapp -t -c "
SELECT
    COUNT(*)          AS new_users,
    SUM(total)        AS revenue,
    COUNT(CASE WHEN status='failed' THEN 1 END) AS failed_orders
FROM orders
WHERE created_at >= NOW() - INTERVAL '24 hours';
")

SUBJECT="Daily Report — $DATE"
BODY="Good morning,

Here is yesterday's summary:

$REPORT

—
Sent automatically by cron on $(hostname)
"

log "Sending report to $RECIPIENT..."
curl -s -o /dev/null --user "api:$MAILGUN_API_KEY" \
    https://api.mailgun.net/v3/"$MAILGUN_DOMAIN"/messages \
    -F from="reports@$MAILGUN_DOMAIN" \
    -F to="$RECIPIENT" \
    -F subject="$SUBJECT" \
    -F text="$BODY"

log "Report sent successfully"

Frequently asked questions

How do I run a Python script with cron?

Use the absolute path to the Python interpreter from your virtual environment: /opt/venv/bin/python /opt/scripts/myscript.py. The venv's Python already knows about all installed packages, so you don't need to activate the virtualenv. If you're using the system Python without a venv, use /usr/bin/python3. Never use a bare python or python3 in a cron line — cron's minimal PATH may not find it, or may find the wrong version.

How do I test a cron job without waiting for the scheduled time?

Run the script in a minimal environment that mimics cron: env -i HOME=/home/youruser PATH=/usr/bin:/bin /bin/bash -c '/opt/scripts/myscript.sh'. This strips your current shell environment and runs the script with only the variables cron would provide. Any "command not found" error you get here is an error you'd get from cron too. You can also add a temporary cron line that fires 2 minutes from now, watch the log file, then remove the line.

How do I get an email alert when a cron job fails?

Add MAILTO=you@example.com at the top of your crontab. Any unredirected output from your cron lines will be emailed to that address — including error messages. This requires a local mail service (postfix or sendmail) to be running. For more reliable alerting, add a trap 'send_alert' ERR inside your script that sends a notification via curl to Mailgun, SendGrid, Slack webhook, or a healthcheck service. This way you control exactly what triggers an alert and what the message says.

What is the safest way to delete old files with a cron job?

Always run the find command without -delete first and redirect the output to a log file. Review the list carefully before adding the deletion flag. In your production script, log the name and size of every file before deleting it, so you have a record if something is cleaned up by mistake. Never run find -delete as root on a broad path like /var without a well-tested filter. The cleanup script in this guide shows a safe pattern with logging and per-directory processing.

How do I make a cron job run once after a server reboot?

Use the @reboot special string in your crontab: @reboot /opt/scripts/start_service.sh >> /var/log/start.log 2>&1. This fires once shortly after the cron daemon starts following a system boot. Note that @reboot is not available on all implementations and is not supported on AWS CloudWatch or Kubernetes CronJob. For production-critical startup tasks on a modern Linux server, a systemd service with WantedBy=multi-user.target is more reliable and easier to monitor.

How do I view the output of a cron job that already ran?

If you redirected output to a log file (which you should always do), read it with cat /var/log/myjob.log or tail -50 /var/log/myjob.log. To check whether cron dispatched the job at all, search the system cron log: grep CRON /var/log/syslog | grep myjob on Debian/Ubuntu, or grep CRON /var/log/cron | grep myjob on RHEL/CentOS. If the system log has the dispatch line but your application log is empty, the script crashed before writing any output — check the script's exit code and set -e behavior.

Can I run multiple commands in a single cron line?

Technically yes, using semicolons (always run next command) or && (stop on first failure). But for anything beyond a single simple command, write a shell script instead. Scripts are easier to test, easier to read, support proper error handling with set -euo pipefail, can be version-controlled with git, and produce structured logs. A cron line should ideally contain nothing but: schedule, script path, log redirect. Everything else belongs in the script.
