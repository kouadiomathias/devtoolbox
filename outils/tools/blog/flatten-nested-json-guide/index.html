The [object Object] that broke a client report

It was a Monday morning stand-up. A developer on a client project had spent the weekend building a data export feature. The product manager opened the Excel file, scrolled to the address column, and every cell said [object Object]. The meeting was paused. The export was pulled. Three hours were spent figuring out what happened.

What happened was straightforward: the API response was a JSON array where each record had a nested address object with city, zip, and country sub-fields. The export script had done row.address — and JavaScript had silently converted the object to its string representation instead of expanding the fields into separate columns.

This is the nested JSON problem. It happens every time a flat table format (CSV, Excel, SQL) meets a hierarchical data format (JSON). Knowing how to flatten correctly — and when not to — will save you from this conversation every time an API changes its response shape.

What flattening actually does

Flattening converts a hierarchical JSON structure into a single flat table. Every nested object becomes a set of additional columns, with keys joined by a separator (usually a dot). The depth of nesting becomes the length of the column name, not an additional level of structure.

❌ Before — nested (unusable in Excel)
[{
  "name": "Alice",
  "address": {
    "city": "Paris",
    "zip": "75001"
  }
}]
✅ After, flat (Excel-ready)
name,address.city,address.zip
Alice,Paris,75001

The address sub-object disappears entirely. Its fields become new columns with dot-notation headers. Any spreadsheet tool can now filter, sort, and pivot the data without parsing JSON.

The algorithm in Python and JavaScript

Flattening is a recursive function. For each key in the object: if the value is another object (not an array), recurse deeper and build a composite key by joining the parent key and child key with the separator. If the value is a primitive (string, number, boolean, null), store it directly.

PythonCore implementation
Ready to use 
def flatten(obj, prefix='', sep='.'):
    items = {}
    for key, val in obj.items():
        new_key = f"{prefix}{sep}{key}" if prefix else key
        if isinstance(val, dict):
            items.update(flatten(val, new_key, sep))
        else:
            items[new_key] = val
    return items

# Example
data = {"name": "Alice", "address": {"city": "Paris", "zip": "75001"}}
print(flatten(data))
# → {'name': 'Alice', 'address.city': 'Paris', 'address.zip': '75001'}

# Apply to an array of records
import csv, io
records = [{"name": "Alice", "address": {"city": "Paris", "zip": "75001"}},
           {"name": "Bob",   "address": {"city": "Lyon",  "zip": "69001"}}]

flat = [flatten(r) for r in records]
keys = sorted({k for row in flat for k in row})

output = io.StringIO()
writer = csv.DictWriter(output, fieldnames=keys)
writer.writeheader()
writer.writerows(flat)
print(output.getvalue())
JavaScriptBrowser and Node.js
Ready to use 
function flatten(obj, prefix = '', sep = '.') {
  return Object.keys(obj).reduce((acc, key) => {
    const newKey = prefix ? `${prefix}${sep}${key}` : key;
    if (
      typeof obj[key] === 'object' &&
      !Array.isArray(obj[key]) &&
      obj[key] !== null
    ) {
      Object.assign(acc, flatten(obj[key], newKey, sep));
    } else {
      acc[newKey] = obj[key];
    }
    return acc;
  }, {});
}

// Convert an array of records to CSV
function toCSV(records, sep = '.') {
  const flat = records.map(r => flatten(r, '', sep));
  const keys = [...new Set(flat.flatMap(Object.keys))].sort();
  const header = keys.join(',');
  const rows = flat.map(r => keys.map(k => JSON.stringify(r[k] ?? '')).join(','));
  return [header, ...rows].join('\n');
}

console.log(toCSV([
  { name: 'Alice', address: { city: 'Paris', zip: '75001' } },
  { name: 'Bob',   address: { city: 'Lyon',  zip: '69001' } }
]));

Edge cases that surprise developers

1Arrays of objects inside a record
Watch out

When a field contains an array of objects, "orders": [{"id": 1, "total": 50}, {"id": 2, "total": 30}], the simple flatten function has no single column to put them in. The DevToolbox converter expands each array element into its own row, duplicating all parent fields. This is usually what you want for Excel analysis, but produces repeated rows that can confuse a database import.

Result of row expansion
name,orders.id,orders.total
Alice,1,50
Alice,2,30    ← Alice appears twice

If you'd rather keep the array as a single cell, set the array handling option to Serialize as JSON string. The cell will contain "[{""id"":1,""total"":50},{""id"":2,""total"":30}]".

2Dots in original key names
Watch out

If your JSON has a key like "first.name", flattening with a dot separator creates an ambiguous column name, you cannot tell whether first.name was an original key or a flattened path. The DevToolbox converter detects this and shows a warning. Switch to __ as the separator or sanitize the keys before converting.

Ambiguous output, avoid this
// Original key contains a dot
{ "first.name": "Alice", "address": { "city": "Paris" } }
// Flattened with dot separator → ambiguous:
// first.name  ← is this "first" → "name" or the original key?
// address.city
Use double-underscore separator
// Safe: no ambiguity with __ separator
flatten(obj, '', '__')
// → { 'first.name': 'Alice', 'address__city': 'Paris' }
3Null values and missing keys
Good to know

When records in an array have different keys, one user has phone and another doesn't, the flattened CSV needs a column for phone that is empty for the second user. The DevToolbox converter handles this automatically, creating a union of all keys across all records and filling in empty cells for missing values.

[
  { "name": "Alice", "phone": "0612345678" },
  { "name": "Bob" }   ← no phone field
]
// Flattened output:
// name,phone
// Alice,0612345678
// Bob,              ← empty cell, not an error
Null vs missing: A field set to null and a field that is absent entirely both become empty cells in the CSV output. If the distinction matters for your pipeline, handle it before flattening by replacing null with a sentinel value.
4Deep nesting producing hundreds of columns
Good to know

A response with 8 levels of nesting can produce column names like data.users.profile.settings.notifications.email.digest.frequency. The converter lets you set a maximum flatten depth. Everything below that depth is serialized as a JSON string in a single cell, keeping the table manageable.

flatten(obj, '', '.', maxDepth=3)
// Fields deeper than 3 levels → serialized as JSON string in their parent column

For large files: pandas and jq

The online tool is built for interactive, quick conversions up to a few megabytes. For large exports from a data warehouse, a full database dump, or a CI/CD pipeline, use one of these approaches.

pandasPython, best for data pipelines
Recommended

pd.json_normalize is the standard library function for this. It handles nested objects, arrays of records, custom separators, and produces a DataFrame you can transform before exporting.

import pandas as pd
import json

with open('api_response.json', encoding='utf-8') as f:
    data = json.load(f)

# Simple case: root is an array
df = pd.json_normalize(data, sep='.')
df.to_csv('output.csv', index=False, encoding='utf-8-sig')

# Wrapped array: data is at data.users
df = pd.json_normalize(data['data']['users'], sep='.')
df.to_csv('output.csv', index=False, encoding='utf-8-sig')

# Array nested inside each record (orders per user)
# record_path: the nested array to expand into rows
# meta: parent fields to keep on every expanded row
df = pd.json_normalize(
    data['users'],
    record_path='orders',
    meta=['id', 'name', 'email'],
    sep='.'
)
df.to_csv('users_orders.csv', index=False, encoding='utf-8-sig')
jqTerminal, best for shell pipelines
Fast & scriptable

jq processes JSON as a stream, which means it can handle files much larger than available RAM. It's ideal for CI/CD pipelines, cron jobs, and any workflow where installing Python is impractical.

# Extract specific fields as CSV
jq -r '
  .data.users[] |
  [.name, .address.city, .address.zip, .email] |
  @csv
' api_response.json > output.csv

# Add a header row
(
  echo '"name","city","zip","email"'
  jq -r '.data.users[] | [.name, .address.city, .address.zip, .email] | @csv' input.json
) > output.csv

# Handle null values (replace with empty string)
jq -r '.data.users[] | [(.name // ""), (.address.city // ""), (.email // "")] | @csv' input.json
Which tool to use: Use the online converter for one-off exports and when you want immediate visual feedback. Use pandas when you need to chain transformations in a Jupyter notebook or scheduled script. Use jq in shell scripts and CI/CD pipelines where speed and low dependencies matter.

Frequently asked questions

Why does my flattened CSV have repeated rows for the same record?

Your JSON contains an array field, for example, a user with multiple orders. The flattening algorithm expands each array element into its own row and repeats all parent fields (name, email, etc.) on every row. This is intentional: it's the only way to represent a one-to-many relationship in a flat table. If you want to avoid duplication, either normalize the data into two separate CSV files, or configure the converter to serialize the array as a JSON string in a single cell.

Flatten vs normalize, which one should I use?

Flattening gives you one wide CSV, fast, simple, great for Excel and reporting tools. Normalizing gives you multiple related CSV files linked by foreign keys — correct for importing into a relational database without data redundancy. If the output goes to a spreadsheet, a BI tool, or a non-technical colleague, flatten. If the output goes into PostgreSQL, MySQL, or any SQL database, normalize.

My JSON keys contain dots. What should I do?

Switch the separator from dot to double underscore in the converter settings. With dot as the separator, a key named first.name in the original JSON produces the same column name as a flattened path first → name. This ambiguity breaks any downstream tool that tries to interpret column names as paths. Double underscore is safe because it rarely appears in original JSON keys.

What happens to null values in the output?

Null values become empty cells in the CSV, the cleanest representation for spreadsheet tools. A field set to null and a field that is entirely absent both produce an empty cell. If your downstream system requires the literal string "null" or a numeric zero, sanitize the data before converting or adjust the empty-cell handling in the converter settings.

How do I handle very large JSON files?

For files larger than a few megabytes, use pd.json_normalize in Python for scriptable pipelines, or jq for streaming from the command line. Both handle large files efficiently and support the same use cases as the online converter. Ready-to-copy code for both is in the article above.
