Line‑by‑line vs word‑by‑word: which diff do you need?

Most diff tools, including the command‑line diff and Git, default to comparing whole lines. But when a single line contains a tiny change — for example a modified IP address or a typo in a long prose sentence — a word‑by‑word diff becomes far more useful. It highlights only the changed characters or words inside the line, rather than marking the whole line as replaced.

Our Text Diff tool supports both modes. Switch to word‑by‑word when you’re reviewing configuration files, documentation, or any content where a single line may contain several independent changes. It’s the same idea as git diff --word-diff but directly in your browser.

What's happening behind the scenes? A quick look at diff algorithms

When you compare two texts, the tool doesn't simply line up the two versions word‑for‑word. It solves a classic computer science problem: finding the Longest Common Subsequence (LCS). The LCS is the longest sequence of lines (or words) that appear in both texts in the same order, allowing gaps. Once identified, everything outside that shared sequence is marked as added or removed.

Different tools can produce slightly different diffs because the LCS is not always unique. For example, if you delete a line and add a similar one elsewhere, the algorithm may interpret the change as a modification rather than a deletion + insertion — depending on how it was tuned. This is why git diff, VS Code, and an online tool can all show the same change differently. Our Text Diff uses a variant that tends to align code blocks more naturally, but for version‑control patches the unified view still matches the expected behavior.

Beyond plain text: when a simple diff isn't enough

Text diffs work perfectly for source code, config files, and prose. But they fall short with structured or binary data. Here’s what to do when a standard line‑by‑line comparison won't cut it:

  • JSON semantic diff: JSON objects are unordered key‑value maps. Two identical JSON files with different key ordering would appear as completely changed in a text diff. Use a semantic JSON comparator like jq with diff on sorted output, or our own JSON Formatter combined with a diff tool after sorting keys.
  • Image diff: Tools like imagemagick (compare command) or online image‑diff websites can overlay pixel differences. Command line: magick compare image1.png image2.png diff.png.
  • PDF diff: PDFs are not text files. Use diffpdf (GUI) or pdftotext + diff for text extraction, or specialized tools like Adobe Acrobat's compare feature.
  • Database diff: for schema or data comparison, use mysqldiff (MySQL), pgdiff (PostgreSQL), or graphical tools like DBeaver.

For everyday development work, a text diff covers 90% of needs. But knowing when to switch to a more appropriate comparator saves time and avoids confusion.

Command‑line diff: the power user's toolkit

When you're already in a terminal, or the files are too large for a browser, the diff command is your best ally. Here are the options I use every day:

# Simple comparison
diff file1.txt file2.txt

# Ignore case differences
diff -i file1.txt file2.txt

# Ignore whitespace changes
diff -w file1.txt file2.txt

# Unified output (like git diff)
diff -u file1.txt file2.txt

# Recursively compare directories
diff -r directory1/ directory2/

The -u flag produces a unified diff, which is exactly what Git shows. It's easier to read and can be applied as a patch. If you're comparing entire project folders, diff -r quickly identifies which files changed.

Git diff advanced: beyond the basics

Git users often rely on git diff without exploring its full capabilities. Here are a few commands that go beyond the default line‑by‑line view:

# Show word‑level changes
git diff --word-diff

# Ignore whitespace only
git diff --ignore-all-space

# Compare with the previous commit
git diff HEAD~1

# Compare two branches
git diff main..feature-branch

# See what changed for a specific file between branches
git diff main..feature-branch -- path/to/file

Word‑diff is especially valuable for code reviews: it pinpoints the exact variable or parameter that changed, without cluttering the diff with unchanged parts of the line. And comparing two branches lets you review an entire feature branch at once — our online diff tool can then be used on individual files you extract from there.
