Why two diff tools show different results for the same change

I noticed this for the first time during a code review. A colleague sent me a diff of our configuration file. I opened the same two files in my diff tool. We were looking at the same change, but the diffs looked different — mine showed three changed lines, his showed five. We were both right.

This happens because text comparison is not a single deterministic operation. When you ask "what changed between these two files?", the answer depends on which algorithm you use and, when multiple valid answers exist, how ties are broken. Two files might have several different valid Longest Common Subsequences — and choosing different ones leads to different diffs.

Understanding why this happens makes you a much better user of diff tools. It also explains some behaviors that look like bugs but are actually correct: a change being attributed to a different line than you expected, context lines appearing or disappearing, or two equivalent diffs looking completely different as text.

The LCS algorithm — how diff actually works

The problem diff solves is finding the Longest Common Subsequence (LCS) between two sequences of lines. The LCS is the longest sequence of lines that appears in both texts in the same relative order — not necessarily contiguous, just in order. Every line in the LCS is "unchanged." Everything not in the LCS is either an addition (only in the new version) or a deletion (only in the old version).

Here's a minimal example. Suppose the original file has four lines:

A
B
C
D

And the new file has four lines:

A
X
C
D

The LCS is A, C, D. Those three lines are unchanged. B is deleted (it was in the original, not in the new version). X is added (it's in the new version, not in the original).

To find the LCS efficiently, the algorithm builds a matrix. Each cell (i,j) contains the length of the LCS between the first i lines of the original and the first j lines of the new version.

Purple cells = matches (same line in both texts). The bottom-right value (3) is the LCS length — three lines are unchanged. Backtracking from that cell gives the LCS: A, C, D.

Why this explains the "different diff" problem

Consider a slightly different example where the word "error" appears three times in both files, and one occurrence moves from line 12 to line 15. The LCS could match the first and third occurrences (showing the middle one as deleted and readded), or it could match the second and third (showing the first as deleted and readded). Both are valid LCS solutions with the same length. Different algorithms, or different tie-breaking rules in the same algorithm, will choose differently — and the resulting diff looks completely different even though both are correct.

The Myers algorithm, published in 1986 and used by Git, GNU diff, and most modern tools, minimizes the number of edit operations (insertions + deletions). It's fast and produces natural-looking diffs for code. But even Myers can have ties in some edge cases.

Split view vs unified view — when to use each

Every serious diff tool offers two display modes. Understanding when each is appropriate saves time during code review and debugging.

Split view (side-by-side)

Both versions appear side by side. Changed lines are placed on the same row with visual highlighting. Lines that exist in one version but not the other show an empty slot on the opposite side. This makes it easy to see exactly what a line looked like before and after a change, without the mental overhead of tracking + and - prefixes.

Split view is best for: small changes in long files, changes where the before/after comparison matters (rename, rewrite, correction), and non-technical reviewers who haven't internalized unified diff syntax.

Unified view (git-style)

A single stream of lines. Unchanged context lines have no prefix. Added lines are prefixed with +. Deleted lines are prefixed with -. A header line (@@ -1,7 +1,8 @@) marks each "hunk" — a contiguous block of changes with some context lines around it.

Unified view is best for: patch files (the format diff -u and git diff produce), email-based code review, copying changes to apply elsewhere, and compact display of many small changes across many files.

Word-by-word diff — precision within a line

Line-level diff marks an entire line as changed even if only one word changed. On long lines — a SQL query, a sentence in documentation, a configuration value — this is unhelpful. You know something changed on that line, but you still have to read the whole line carefully to find the change.

Word-by-word diff goes further: it runs a second diff algorithm within the changed line, finding exactly which words were added and which were removed.

Before (word diff applied to a changed SQL line):

WHERE created_atupdated_at > '2024-01-01' AND status = 'pending''active'

This is equivalent to git diff --word-diff, which shows {-old-} and {+new+} markers around the exact words that changed. It's particularly useful for:

  • Configuration files — a value changed from true to false in a 60-character line
  • Documentation — a single word corrected in a long paragraph
  • SQL migrations — one column name changed in an ALTER TABLE statement
  • JSON/YAML values — a field value changed but the key and structure stayed the same

The diff command — complete reference

The Unix diff command is available on every Linux and macOS system, and on Windows via Git Bash or WSL. It's the most reliable way to compare large files, integrate diffs into scripts, and produce patch files for distribution.

# Basic diff — default output format (hard to read)
diff file1.txt file2.txt

# Unified format — readable, git-compatible, standard for patches
diff -u file1.txt file2.txt

# Unified with custom context lines (default is 3)
diff -U5 file1.txt file2.txt     # 5 lines of context around each change

# Ignore case differences
diff -i file1.txt file2.txt

# Ignore whitespace differences
diff -w file1.txt file2.txt      # All whitespace
diff -b file1.txt file2.txt      # Only leading/trailing whitespace

# Ignore empty lines
diff -B file1.txt file2.txt

# Recursive — compare entire directory trees
diff -r dir1/ dir2/

# Recursive + unified + brief listing of changed files first
diff -rq dir1/ dir2/             # Just list which files differ

# Produce a patch file
diff -u original.txt modified.txt > changes.patch

# Apply a patch
patch original.txt < changes.patch

# Apply in reverse (undo the patch)
patch -R original.txt < changes.patch

# Apply a git-style patch (strip one path component)
patch -p1 < changes.patch

# Check if patch would apply cleanly (dry run)
patch --dry-run -p1 < changes.patch

Combining with other tools

# Compare two JSON files after formatting for readability
diff <(python3 -m json.tool file1.json) <(python3 -m json.tool file2.json)

# Compare two YAML files after normalization
diff <(python3 -c "import yaml,sys; print(yaml.dump(yaml.safe_load(open('f1.yaml'))))") \
     <(python3 -c "import yaml,sys; print(yaml.dump(yaml.safe_load(open('f2.yaml'))))")

# Show only files that differ in a directory tree
diff -rq dir1/ dir2/ | grep "differ"

# Count changed lines
diff -u old.sql new.sql | grep -c "^[+-]"

# Color output in the terminal
diff --color=always -u file1.txt file2.txt | less -R

git diff — the commands developers use daily

git diff is the most-used diff tool in the world. It integrates deeply with Git's staging area, branches, and history. These are the commands that cover 95% of daily use.

Basic usage

# Changes in working directory not yet staged
git diff

# Changes staged for the next commit (what will be committed)
git diff --staged
# or equivalently:
git diff --cached

# All changes since the last commit (staged + unstaged)
git diff HEAD

# Changes in a specific file
git diff -- path/to/file.js

# Changes in a specific directory
git diff -- src/

Comparing branches and commits

# Compare two branches (what changed from main to feature-branch)
git diff main..feature-branch

# Equivalent: what commits are in feature-branch but not main
git diff main...feature-branch   # three dots = branch point

# Compare two specific commits
git diff abc1234..def5678

# Compare a commit to the current working directory
git diff abc1234

# What changed in a specific file between two branches
git diff main..feature-branch -- src/api/users.js

# Compare a branch to a tag
git diff v2.0.0..main

Output format options

# Word diff — highlights exact changed words within a line
git diff --word-diff

# Colored word diff (more visual)
git diff --word-diff=color

# Summary: just the list of changed files with stats
git diff --stat
git diff --stat main..feature-branch

# Name only — just filenames, no content
git diff --name-only main..feature-branch

# Ignore whitespace changes
git diff --ignore-all-space
git diff -w

# Show each change as a separate commit in the output
git diff -p

# Condensed: show only the names of files that differ
git diff --diff-filter=M --name-only   # M = modified files only
# Filters: A=added, D=deleted, M=modified, R=renamed, C=copied

Useful workflows

# See what changed in the last commit
git diff HEAD~1 HEAD

# See what changed across the last N commits
git diff HEAD~5 HEAD

# Find when a specific line was introduced
git log -S "string to search for" --source --all

# Generate a patch from the last commit to email to a colleague
git format-patch -1 HEAD

# Apply a patch from a colleague
git am < 0001-their-change.patch

# Interactive staging — stage only specific changes within a file
git add -p file.js   # Walk through each hunk and choose yes/no/split

Diff by content type — the right approach for each format

JSON files

Plain text diff of JSON is often misleading. Two JSON files can represent exactly the same data but look completely different as text: different key ordering, different indentation, different string quoting. A line-level diff would show hundreds of changes when nothing structurally changed.

The correct workflow: format both files consistently first, then diff the formatted output. Use the DevToolbox JSON Formatter or the command line:

# Format both files, then diff
diff <(python3 -m json.tool --sort-keys file1.json) \
     <(python3 -m json.tool --sort-keys file2.json)

# Node.js alternative
diff <(node -e "console.log(JSON.stringify(require('./file1.json'),null,2))") \
     <(node -e "console.log(JSON.stringify(require('./file2.json'),null,2))")

For semantic equality checks (ignoring formatting entirely), use the DevToolbox Text Diff Tool's similarity score or compare the formatted outputs — if the diff is empty, the data is identical.

SQL files (migrations)

When reviewing SQL migrations before deployment, format both scripts first. The DevToolbox SQL Formatter normalizes whitespace, keyword case, and indentation. After formatting, a diff shows only structural changes — not formatting noise.

# Compare two migration scripts in the terminal
diff -u --label "original" --label "modified" \
    migration_v1.sql migration_v2.sql

# In Git, always view SQL diffs with word-diff
git diff --word-diff -- migrations/

YAML configuration files

YAML anchors and multi-line strings can make raw text diffs confusing. The same principle applies: normalize first. Parse and re-dump both files with consistent formatting before diffing.

# Normalize and diff YAML files
diff <(python3 -c "import yaml; print(yaml.dump(yaml.safe_load(open('config-v1.yaml')), sort_keys=True))") \
     <(python3 -c "import yaml; print(yaml.dump(yaml.safe_load(open('config-v2.yaml')), sort_keys=True))")

Prose and documentation

For text documents, word-by-word diff is almost always more useful than line-by-line. A single sentence spanning 80 characters contains many words — marking the whole line as changed tells you less than showing exactly which word was corrected.

In the terminal, git diff --word-diff works well for Markdown. For plain text files, the DevToolbox Text Diff Tool's word diff mode is the most visual option.

Reading a unified diff — annotated line by line

Unified diff format is the standard output of diff -u and git diff. It's the format used in patch files, pull request diffs, and code review tools. Once you know how to read it, it becomes second nature.

--- a/src/config.js    2025-06-07 09:00:00
← Original file (a/ prefix added by git)
+++ b/src/config.js    2025-06-08 14:23:00
← Modified file

@@ -12,7 +12,8 @@ function loadConfig() {
← Hunk header: original lines 12-18 → new lines 12-19. Function name shown as context.

const host = process.env.HOST || 'localhost';
const port = process.env.PORT || 3000;
← Context lines (leading space): unchanged, shown for orientation

- const timeout = 5000;
← Deleted line (- prefix): was in original, not in new version
+ const timeout = parseInt(process.env.TIMEOUT) || 5000;
+ const maxRetries = parseInt(process.env.MAX_RETRIES) || 3;
← Added lines (+ prefix): in new version, not in original

return { host, port, timeout };
← Context line: unchanged. Note: maxRetries should probably be added here too.
}

The hunk header explained

The @@ -12,7 +12,8 @@ header is the most cryptic part for newcomers. Here's what each number means:

PartMeaningIn our example
-12,7In the original file: starts at line 12, spans 7 linesLines 12–18 of the original
+12,8In the new file: starts at line 12, spans 8 linesLines 12–19 of the new version (one line added)
Text after @@The nearest function or section heading (optional context)function loadConfig() {

The line count difference (7 vs 8 in our example) tells you immediately how many lines were added or removed: new_count - old_count = net lines added. A negative number means more lines were deleted than added in this hunk.

Applying and reversing a patch

# Save a diff to a patch file
git diff main..feature-branch > feature.patch
# or
diff -u original.conf new.conf > changes.patch

# Apply the patch
patch original.conf < changes.patch

# Verify the patch would apply cleanly before actually doing it
patch --dry-run original.conf < changes.patch

# Reverse the patch (undo the changes)
patch -R original.conf < changes.patch

# Apply a git format-patch output (different format, includes commit message)
git am < 0001-add-timeout-configuration.patch

Frequently asked questions

What is the difference between split view and unified view?

Split view shows both texts side by side with differences aligned on the same row. Unified view shows a single stream with added lines prefixed by + and removed lines by -, which is the format produced by diff -u and git diff. Split view is easier to read for small targeted changes; unified view is more compact, is the standard format for patch files, and is required for applying patches with the patch command.

Why do two diff tools show different results for the same change?

Text comparison finds the Longest Common Subsequence (LCS) between two files — the longest sequence of lines that appears in both in the same order. When multiple LCS solutions of equal length exist, different algorithms break ties differently. The Myers algorithm (used by Git, GNU diff, and most modern tools) minimizes the number of edit operations, but some texts have multiple equally minimal solutions that look completely different as output. Both are correct; neither is "more right" than the other.

What is word-by-word diff and when should I use it?

Word-by-word diff (also called "word diff") runs a second comparison within each changed line to highlight exactly which words were added or removed, instead of marking the entire line as changed. Use it for config files with long lines, documentation, SQL migrations, and any content where a line changed but most of it stayed the same. In git: git diff --word-diff. In DevToolbox's Text Diff Tool, this happens automatically within changed lines.

How does the share URL feature work?

The Share button in the DevToolbox Text Diff Tool encodes both texts and all options (ignore case, ignore whitespace, view mode, language) into the URL fragment using Base64 encoding. Nothing is stored on any server — the entire state is in the URL itself. Send the link to a colleague and they see the exact same diff, decoded and rendered locally in their browser. This also works for bookmarking a specific comparison.

What is the LCS algorithm?

LCS stands for Longest Common Subsequence. It finds the longest sequence of elements (lines, in the context of text diff) that appears in both inputs in the same relative order, but not necessarily contiguously. The diff algorithm uses LCS to identify which lines are unchanged — they form the LCS. Everything not in the LCS is either an addition or a deletion. The Myers algorithm, published in 1986 and used by Git, computes a minimal edit script efficiently using a diagonal path through the edit graph.

How do I apply a patch file from a unified diff?

Use the patch command: patch original.txt < changes.patch. For a git-style patch that includes a path prefix: patch -p1 < changes.patch (the -p1 strips the a/ prefix git adds). To test without modifying files: patch --dry-run -p1 < changes.patch. To reverse a patch: patch -R original.txt < changes.patch. For patches created with git format-patch (which include commit messages), use git am < patch_file instead.
