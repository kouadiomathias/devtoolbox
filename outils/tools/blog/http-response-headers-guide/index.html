The production incident that started this

In 2015, researchers discovered a vulnerability affecting a major banking application in the UK. The attack was simple: a third-party ad network had been compromised and was serving a page that loaded the bank's transfer confirmation screen inside a transparent <iframe>. When a user visited the malicious page, they thought they were clicking a "Win an iPhone" button. They were actually clicking "Confirm transfer" on a bank page they couldn't see.

The fix would have been two lines of configuration on the bank's server. Two lines that would have added X-Frame-Options: DENY to every response. They weren't there. This attack pattern — clickjacking — has been documented by the OWASP Foundation since 2008, and it still works on sites that haven't added this header.

This isn't an edge case. Security researchers at SecurityHeaders.com run automated scans of the Alexa top million sites. In their most recent analysis, only about 3% of websites received an A or A+ rating. The other 97% were missing at least one critical security header. On most of them, the misconfiguration wasn't complex — it was simply absent.

HTTP response headers are invisible in normal browsing. They don't appear in the page source. They live in the Network tab of DevTools, in curl -I output, in Nginx config files. Most developers never look at them until something breaks. This article changes that.

The 8 security headers every production server needs

1. Strict-Transport-Security (HSTS)

Strict-Transport-Security Critical

Tells the browser to always use HTTPS for this domain, even if the user types http:// or clicks an HTTP link.

max-age=31536000; includeSubDomains; preload

Without HSTS, here's what happens on public Wi-Fi: a user types yourbank.com in the address bar. The browser sends the first request over HTTP — it has no way to know it should use HTTPS before making contact. A network attacker sitting on the same network intercepts that HTTP request and never forwards it to the real server. They serve their own page instead. This is a protocol downgrade attack, and HSTS prevents it by making the browser refuse to make HTTP connections to the domain after the first successful HTTPS visit.

The max-age value is in seconds. 31536000 is one year — the standard recommendation. includeSubDomains extends the protection to all subdomains. preload submits your domain to the HSTS preload list maintained by Google and included in all major browsers, meaning HTTPS is enforced even before the first visit — no unprotected first request ever.

One important caveat: don't add preload unless you're certain every subdomain serves valid HTTPS and always will. Removing a domain from the preload list takes weeks and requires manual review.

2. Content-Security-Policy (CSP)

Content-Security-Policy Critical

Defines a whitelist of trusted sources for scripts, styles, images, and other resources. The browser enforces it — anything not on the list is blocked.

default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'self'

CSP is your primary defense against Cross-Site Scripting (XSS). If an attacker manages to inject a script into your page — through a comment field, an API response that's rendered without sanitization, or a compromised third-party library — a well-configured CSP will block that script from executing.

The most common mistake I see in real projects is using unsafe-inline to allow inline scripts and styles. This single directive completely disables XSS protection for scripts. If you have inline scripts, the modern approach is to use nonces: a random value generated per request, added to the CSP header and to each legitimate script tag.

# With nonce-based CSP (recommended for apps with inline scripts)
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-{random_per_request}'; object-src 'none'

If you're not sure where to start, use Content-Security-Policy-Report-Only first. It sends violation reports to a URL you specify without actually blocking anything. This lets you understand what your policy would break before enforcing it.

# Test without enforcing — add report-uri to collect violations
Content-Security-Policy-Report-Only: default-src 'self'; report-uri https://yourapp.com/csp-report

3. X-Frame-Options

X-Frame-Options High

Prevents your page from being loaded inside an <iframe> on another origin. The primary clickjacking defense.

DENY

The clickjacking scenario is straightforward: an attacker builds a page that overlays a transparent iframe loading your app. The user interacts with what they think is the attacker's page but is actually sending clicks to your app underneath. Sensitive actions — confirming transactions, changing passwords, granting permissions — can be triggered without the user's knowledge.

DENY blocks framing entirely, on all domains including your own. SAMEORIGIN allows framing only from the same origin. The deprecated ALLOW-FROM origin is ignored by modern browsers — if you need to allow a specific origin, use CSP's frame-ancestors directive instead.

CSP vs X-Frame-Options: CSP's frame-ancestors is more flexible and supersedes X-Frame-Options in supporting browsers. But X-Frame-Options still covers older browsers. Keeping both costs one line of config and costs nothing in performance.

4. X-Content-Type-Options

X-Content-Type-Options High

Prevents the browser from guessing the content type of a response. The only valid value is nosniff.

nosniff

Without this header, browsers try to be helpful by "sniffing" the content type when the declared type seems wrong. An attacker can upload a file with a .txt extension that contains JavaScript. If a script tag points to it, an older browser might execute it anyway, regardless of what the server says the Content-Type is. nosniff tells the browser: "Trust the Content-Type I declare. Don't guess."

This is one of the simplest headers to add — one line of config, zero side effects if your Content-Type headers are correct. There's no valid reason to not have it in production.

5. Referrer-Policy

Referrer-Policy Medium

Controls how much URL information is included in the Referer header when navigating to other sites.

strict-origin-when-cross-origin

Consider a URL like https://yourapp.com/reset-password?token=abc123. Without a Referrer-Policy, if a user clicks a link on that page to another site, the destination site's server logs receive the full URL — including the reset token. That token is now in a third-party server's logs, potentially accessible to their engineers, and possibly collected by analytics services.

The same applies to search query parameters, internal IDs, session tokens passed in URLs, and any other data in your URL structure. strict-origin-when-cross-origin sends the full URL to same-origin destinations but only the origin (no path, no query string) to cross-origin destinations. It's the browser default since Chrome 85, but explicitly setting it ensures consistent behavior across all browsers.

6. Permissions-Policy

Permissions-Policy Medium

Restricts browser API access for your page — camera, microphone, geolocation, payment, and more.

camera=(), microphone=(), geolocation=(), payment=()

If your application doesn't use the camera, there's no reason for any script on your page to be able to access it — including a compromised third-party library or a successful XSS payload. Permissions-Policy enforces this at the browser level. Even if an attacker manages to run code on your page, the browser will refuse to grant camera access if your policy says camera=().

This header replaced the older Feature-Policy header in 2020. The successor is supported in all modern browsers. List only the APIs your application actually uses, and explicitly deny everything else.

7. Cross-Origin Opener Policy (COOP)

Cross-Origin-Opener-Policy Medium

Isolates your browsing context from cross-origin windows, required for high-resolution timers and SharedArrayBuffer.

same-origin

After the Spectre CPU vulnerability was discovered in 2018, browsers restricted access to precise timing APIs that could be used for side-channel attacks. To re-enable features like SharedArrayBuffer and high-resolution performance.now(), your site needs to be in a cross-origin isolated context — which requires both COOP and COEP (Cross-Origin-Embedder-Policy: require-corp). If you don't use SharedArrayBuffer, COOP same-origin is still a good defense-in-depth header that prevents your page from being referenced by cross-origin windows.

8. Cross-Origin Resource Policy (CORP)

Cross-Origin-Resource-Policy Low

Controls which origins can load this resource, protecting against Spectre-style attacks via cross-origin reads.

same-origin (for sensitive resources)

This header is particularly relevant for APIs and resources that return sensitive data. same-origin means only same-origin documents can load the resource. Use cross-origin for resources intentionally shared (like a public CDN). Set it on APIs that should never be embedded or cross-origin loaded.

Cache-Control: every directive explained

Cache-Control is the header that most developers half-understand. They know no-cache means "don't cache" — except it doesn't. Getting this wrong has real consequences: users see stale data, sensitive pages get cached by CDNs, or your app makes unnecessary requests to the origin on every load. Let me go through each directive properly.

DirectiveWhat it actually meansWhen to use it
max-age=NCache for N seconds. After that, the browser revalidates with the server.Static assets, pages that change infrequently
no-storeNever store this response anywhere — not even temporarily. Every request goes to the server.Banking pages, personal data, pages with sensitive info
no-cacheThe browser can store it, but must revalidate with the server before serving it. Uses ETag or Last-Modified to check if fresh.Pages that change frequently but benefit from ETag validation
privateOnly the user's browser cache can store this. CDNs and shared proxies cannot.Authenticated pages, user-specific content
publicCDNs, proxies, and shared caches can store this.Static assets, public API responses, images
s-maxage=NCDN-specific max-age. Overrides max-age for shared caches only.When you want different TTLs for browsers vs CDNs
stale-while-revalidate=NServe the stale cached version for up to N seconds while fetching a fresh copy in the background.UI pages where a slightly stale response is acceptable
immutableThe resource will never change. The browser skips revalidation during max-age, even on hard refresh.Content-hashed assets (app.a3b1c2d4.js, logo.png?v=3)
must-revalidateOnce stale, the browser must revalidate with the origin before serving. Cannot serve stale content even if the server is unreachable.Data where serving stale content is unacceptable

The most common misunderstanding: no-cache ≠ "no caching". It means "cache it, but always check with the server first". If you truly don't want caching at all, use no-store. The confusion comes from the name, which was arguably a bad choice by the HTTP spec authors.

A practical strategy for modern web applications: use Cache-Control: public, max-age=31536000, immutable for content-hashed static assets (your bundler adds a hash to the filename on every build), and Cache-Control: no-cache for your HTML files (which are short and cheap to revalidate, and need to reference the latest assets).

# Nginx — different cache strategies per resource type

# HTML files: always revalidate (they reference the latest assets)
location ~* \.html$ {
    add_header Cache-Control "no-cache";
}

# Content-hashed assets: cache forever, never revalidate
location ~* \.(js|css|woff2)$ {
    add_header Cache-Control "public, max-age=31536000, immutable";
}

# API responses: private, 60 seconds max
location /api/ {
    add_header Cache-Control "private, max-age=60";
}

# Sensitive pages: never store
location /account/ {
    add_header Cache-Control "no-store";
}

Headers that reveal too much information

Some headers shouldn't be there at all. They're added by default by web servers and frameworks, and they hand attackers free intelligence about your infrastructure.

Server: Apache/2.4.51 (Ubuntu)

That response header tells an attacker exactly what to search for on NVD (National Vulnerability Database) or Exploit-DB. Apache 2.4.49 had a path traversal CVE (CVE-2021-41773) that was actively exploited within days of disclosure. A server advertising its version is essentially saying: "Here's a list of my known vulnerabilities."

Removing the version is trivial. Removing the server name entirely is slightly more involved but possible on all major servers.

# Nginx — hide version number
server_tokens off;
# Result: Server: nginx (version removed)

# Apache — httpd.conf or .htaccess
ServerTokens Prod
ServerSignature Off
# Result: Server: Apache (only product name, no version)

X-Powered-By: PHP/8.1.0 or X-Powered-By: Express

Added by default by PHP and Express.js. Same problem: you're advertising your framework and version. Removing it:

# Express.js — disable the header entirely
app.disable('x-powered-by');

# Or use Helmet, which handles this and other security headers automatically
const helmet = require('helmet');
app.use(helmet());

# PHP — php.ini
expose_php = Off

X-AspNet-Version and X-AspNetMvc-Version

Added by default in older ASP.NET configurations. Neither header serves any legitimate purpose in production.


<system.web>
  <httpRuntime enableVersionHeader="false" />
</system.web>

CORS: the most misunderstood security mechanism

Cross-Origin Resource Sharing (CORS) is the mechanism that controls which origins can make cross-origin requests to your server. It's enforced by browsers — not by your server itself. Understanding this distinction is critical, because CORS does not protect your server from server-to-server requests or from non-browser clients like curl.

The common mistake: wildcard + credentials

A pattern I've seen in production multiple times:

# ❌ This combination is rejected by all modern browsers
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true

The CORS spec explicitly forbids this combination. When Access-Control-Allow-Credentials: true, the browser will only include cookies and authorization headers in the cross-origin request if the server specifies a concrete origin — not a wildcard. The logic: a wildcard means "any origin", which is acceptable for truly public resources, but you'd never want to send your authentication cookies to a request from any random origin.

If you need credentials, you need to echo back the specific requesting origin (after validating it against your allowlist):

# ✅ Correct for credentialed cross-origin requests
# (Set dynamically in your server code after validating the Origin header)
Access-Control-Allow-Origin: https://app.yourcompany.com
Access-Control-Allow-Credentials: true
Vary: Origin  # ← Tell caches that responses differ by Origin
Always add Vary: Origin when you set Access-Control-Allow-Origin dynamically. Without it, a CDN might cache the response with one origin's header and serve it to a different origin — either leaking your credentials policy or causing legitimate requests to fail.

The CORS headers reference

HeaderWho sends itWhat it controls
Access-Control-Allow-OriginServer (response)Which origin can read the response
Access-Control-Allow-MethodsServer (preflight response)Which HTTP methods are allowed
Access-Control-Allow-HeadersServer (preflight response)Which request headers are permitted
Access-Control-Allow-CredentialsServer (response)Whether cookies/auth headers can be sent
Access-Control-Max-AgeServer (preflight response)How long to cache the preflight result (seconds)

Access-Control-Request-MethodBrowser (preflight)The method the actual request will use

Simple vs preflighted requests

Not every cross-origin request triggers a preflight OPTIONS request. "Simple" requests — GET and POST with standard content types like application/x-www-form-urlencoded — go directly. Complex requests — those with custom headers, non-standard content types (like application/json), or methods other than GET/POST — trigger a preflight first. The browser sends an OPTIONS request to check permissions, and your server must respond with the appropriate headers before the real request is sent.

Use Access-Control-Max-Age to cache preflight results and avoid the extra round trip on every request. A value of 86400 (one day) is reasonable for most APIs.

Ready-to-paste server configurations

These configurations give you a solid security baseline. They're not exhaustive — your CSP in particular needs to be tailored to your application's actual sources — but they address the most critical headers and will move you from an F to a B or better immediately.

Nginx — complete security headers block

server {
    listen 443 ssl http2;
    server_name example.com;

    # ── Security headers ──────────────────────────────────────

    # Force HTTPS for 1 year, including subdomains
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # Prevent clickjacking
    add_header X-Frame-Options "DENY" always;

    # Prevent MIME sniffing
    add_header X-Content-Type-Options "nosniff" always;

    # Control referrer information
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # Restrict browser API access
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=()" always;

    # Content Security Policy — tighten for your app
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; object-src 'none'; base-uri 'self';" always;

    # ── Hide server information ───────────────────────────────
    server_tokens off;

    # ── Your SSL, location blocks, etc. ──────────────────────
}

Apache — .htaccess or VirtualHost

# Requires mod_headers to be enabled: a2enmod headers

<IfModule mod_headers.c>
    Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
    Header always set X-Frame-Options "DENY"
    Header always set X-Content-Type-Options "nosniff"
    Header always set Referrer-Policy "strict-origin-when-cross-origin"
    Header always set Permissions-Policy "camera=(), microphone=(), geolocation=()"
    Header always set Content-Security-Policy "default-src 'self'; object-src 'none';"
</IfModule>

# Hide Apache version
ServerTokens Prod
ServerSignature Off

Express.js (Node.js) — with Helmet

const express = require('express');
const helmet = require('helmet');

const app = express();

// Helmet configures all the important security headers automatically
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],
      objectSrc: ["'none'"],
      baseUri: ["'self'"],
    },
  },
  hsts: {
    maxAge: 31536000,
    includeSubDomains: true,
    preload: true,
  },

    camera: [],
    microphone: [],

// Remove X-Powered-By (Helmet does this too, but explicit is better)
app.disable('x-powered-by');

Frequently asked questions

Do I need all security headers on every page?

HSTS, X-Content-Type-Options, Referrer-Policy, and Permissions-Policy should be set globally on every response. CSP and X-Frame-Options are strongly recommended everywhere but may require per-page customization if you embed legitimate iframes or load third-party scripts. The Server and X-Powered-By headers should be removed or neutralized globally — they serve no user-facing purpose in production.

Will a strict CSP break my website?

A poorly configured CSP can block legitimate resources — your own scripts, Google Analytics, embedded fonts. The right approach is to use Content-Security-Policy-Report-Only first. This mode sends violation reports to a URL you specify without enforcing anything. Monitor the reports for a week, add the necessary sources to your policy, then switch to enforcement. Also avoid unsafe-inline for scripts — use nonces instead, which allow specific legitimate inline scripts while blocking injected ones.

Is X-Frame-Options still needed if I use CSP frame-ancestors?

Strictly speaking, no — CSP frame-ancestors supersedes X-Frame-Options in browsers that support CSP Level 2 (all modern browsers). But keeping both headers provides defense in depth for older browsers and browser extensions that might not fully implement CSP. The cost is one extra line of config. The benefit is slightly broader coverage. Unless you have a specific reason to remove it, keep both.

What is the difference between no-cache and no-store?

This is the most common Cache-Control confusion. no-cache means the browser can store the response locally but must check with the server before serving it — using ETag or Last-Modified validation. If the server says nothing changed, it returns 304 Not Modified and the browser serves the cached copy. no-store means never store the response anywhere. Use no-cache for content that changes frequently but where caching logic is useful. Use no-store for truly sensitive content like banking sessions or personal medical data.

Can I combine Access-Control-Allow-Origin: * with credentials?

No, and the browser will block this combination regardless of what your server sends. The CORS spec requires that when Access-Control-Allow-Credentials is true, the Access-Control-Allow-Origin header must contain a specific origin, not a wildcard. This is by design: credentials should never be sent cross-origin to an arbitrary unknown origin. To support credentialed cross-origin requests, validate the Origin header from the request against your allowlist and echo it back in the response, along with a Vary: Origin header to prevent caching issues.

How do I check my current security headers?

Three ways. The quickest: open browser DevTools, go to the Network tab, click any request to your server, and look at the Response Headers section. From the terminal: curl -I https://yoursite.com shows all response headers. Or use the DevToolbox HTTP Header Analyzer — paste the headers and get an instant security grade with explanations for each one, what's missing, and what the recommended value is.
