Migrating from MongoDB ObjectID to UUID in a Node.js App — Complete Guide

Published on June 6, 2026 · Updated June 8, 2026 by Mathias Kouadio — 12 min read

Your MongoDB application has been running in production for months. The ObjectID values get the job done, but you're starting to feel their limits. You're planning to add PostgreSQL, you're exposing IDs in a REST API, or you simply want more standard identifiers. Migrating to UUID is the solution — but how do you do it without breaking your existing database?

In this article, I'll walk you through a phased migration with production-ready scripts, dual-write handling, monitoring, validation, and a rollback plan in case things go wrong.

Why move away from ObjectID?

The ObjectID is MongoDB-specific. It's compact (12 bytes) and contains a timestamp, but it doesn't easily interoperate with other DBMS or meet frontend developers' expectations. A UUID, on the other hand, is universal, random (v4) or sortable (v7), and far better suited for public APIs.

Three concrete reasons to migrate:

  • You want to expose IDs in URLs without revealing creation date or sequence.
  • You're adding PostgreSQL or another database that doesn't understand ObjectID.
  • You want to standardize your identifiers across your entire architecture.

Phase 1: add the UUID field without touching _id

The golden rule: never delete _id. MongoDB requires it as the primary key. We'll add a new uuid field to each document. If you're using Mongoose, start by updating your schema:

const userSchema = new mongoose.Schema({
  uuid: { type: String, unique: true, sparse: true },
  name: String,
  email: String
});

The sparse: true flag ensures that only documents that already have the field are indexed. Then run the optimized migration script using bulkWrite — this is 10–50× faster than updating documents one by one:

const { v4: uuidv4 } = require('uuid');
const BATCH_SIZE = 1000;

async function migrateUUIDs(db) {
  let processed = 0;
  
  while (true) {
    const docs = await db.collection('users')
      .find({ uuid: { $exists: false } })
      .limit(BATCH_SIZE)
      .toArray();
    
    if (docs.length === 0) break;
    
    const operations = docs.map(doc => ({
      updateOne: {
        filter: { _id: doc._id },
        update: { $set: { uuid: uuidv4() } }
      }
    }));
    
    await db.collection('users').bulkWrite(operations, { ordered: false });
    processed += docs.length;
    console.log(`Migrated: ${processed} documents`);
  }
  
  console.log(`Migration complete: ${processed} total`);
}

Phase 2: create the unique index

Once all documents have a UUID, create a unique index:

await db.collection('users').createIndex({ uuid: 1 }, { unique: true });

Phase 3: update references

The trickiest part: related collections. If an orders document references a user_id as an ObjectID, you need to migrate these references. The safest approach is to add a user_uuid field while keeping user_id during the transition.

// For each order, find the corresponding user UUID
const orders = db.collection('orders').find({ user_uuid: { $exists: false } });
while (await orders.hasNext()) {
  const order = await orders.next();
  const user = await db.collection('users').findOne({ _id: order.user_id });
  if (user) {
    await db.collection('orders').updateOne(
      { _id: order._id },
      { $set: { user_uuid: user.uuid } }
    );
  }
}
⚠ Important: Test this step on a staging environment with a realistic data volume. For large collections, break this operation into batches similar to Phase 1.

Monitoring progress in real time

Track how many documents have been migrated while the script is running:

async function checkProgress(db) {
  const total = await db.collection('users').countDocuments();
  const migrated = await db.collection('users')
    .countDocuments({ uuid: { $exists: true } });
  const remaining = total - migrated;
  const pct = ((migrated / total) * 100).toFixed(1);
  
  console.log(`Total: ${total} | Migrated: ${migrated} | Remaining: ${remaining} | ${pct}%`);
}

// Call every 30 seconds during migration
setInterval(() => checkProgress(db), 30000);

Phase 4: dual-write during the transition

To avoid inconsistencies during the migration, update your application code to write both the old _id and the new uuid. This way, new data is immediately compatible. For new documents, consider using UUID v7 (time-sortable) instead of v4 if you need chronological queries:

// For new documents after the migration
// Prefer UUID v7 (time-sorted) to UUID v4 (purely random)
// if you need chronological queries

const { v7: uuidv7 } = require('uuid'); // uuid@9.0+
const newUser = new User({
  name: 'Alice',
  email: 'alice@example.com',
  uuid: uuidv7() // Time-sorted, no index performance impact on MongoDB
});
await newUser.save();
// Old code continues to use _id, new code uses uuid

Phase 5: switch queries to UUID

Once all data is migrated and indexes are in place, update your API endpoints to use uuid instead of _id. Gradually remove the old references.

Validation — don't skip this

Run a statistically significant validation before declaring the migration successful:

async function validateMigration(db, sampleSize = 1000) {
  const results = {
    total_checked: 0,
    uuid_missing: 0,
    uuid_invalid: 0,
    broken_references: 0
  };
  
  // 1. Verify that all documents have a valid UUID
  const uuidRegex = /^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i;
  
  const sample = await db.collection('users')
    .aggregate([{ $sample: { size: sampleSize } }])
    .toArray();
  
  for (const doc of sample) {
    results.total_checked++;
    if (!doc.uuid) results.uuid_missing++;
    else if (!uuidRegex.test(doc.uuid)) results.uuid_invalid++;
  }
  
  // 2. Check the integrity of references
  const orderSample = await db.collection('orders')
    .aggregate([{ $sample: { size: sampleSize } }])
    .toArray();
    
  for (const order of orderSample) {
    if (order.user_uuid) {
      const user = await db.collection('users')
        .findOne({ uuid: order.user_uuid });
      if (!user) results.broken_references++;
    }
  }
  
  console.log('Validation results:', results);
  
  const isValid = results.uuid_missing === 0 && 
                  results.uuid_invalid === 0 && 
                  results.broken_references === 0;
  
  console.log(isValid ? '✅ Migration valid' : '❌ Issues detected — check before going live');
  return isValid;
}

Rollback: how to revert

If the migration fails, simply delete the uuid field and its associated indexes. As long as you haven't deleted _id or the old references, the rollback is instant.

// Drop the unique index
await db.collection('users').dropIndex('uuid_1');

// Remove the uuid field from all documents
await db.collection('users').updateMany(
  { uuid: { $exists: true } },
  { $unset: { uuid: "" } }
);

Always keep a database backup before running the migration.

Frequently Asked Questions

Can I remove the _id field after the migration?

No, you cannot remove _id because it is MongoDB's mandatory primary key. However, you can stop using it in your application code and base all queries on the uuid field instead.

What is the performance impact?

A UUID index is slightly larger than an ObjectID index (16 bytes vs 12). For most applications with collections under 10 million documents, the difference is negligible on simple queries. Complex aggregation pipelines may be more sensitive — always test with your own data volume.

Should I migrate all collections at once?

No, you can migrate collection by collection. Start with collections that have no relationships, then tackle collections with references once dual-write is in place.
