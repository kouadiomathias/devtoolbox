MongoDB ObjectID vs UUID — Structure, Portability, and Practical Migration

Published on May 23, 2026 by Kouame Kouadio Mathias — 9 min read
Quick summary: MongoDB's ObjectID encodes a timestamp, machine, and counter in 12 bytes. It's ultra‑compact but hard to port outside the MongoDB ecosystem. We explain its structure, the real collision risk, and how to gradually migrate to UUID v7 with executable code examples.

Decision Table — ObjectID or UUID for Your Project?

ObjectID Internal Structure — What 12 Bytes Really Contain

Contrary to what many believe, an ObjectID is not a random value. Its structure is deterministic:

ObjectID: 507f1f77bcf86cd799439011
507f1f77 bcf86cd799 439011
├─ Timestamp ─┤ ├── Machine+PID ──┤ ├─ Counter ─┤
4 bytes 5 bytes 3 bytes

The 4‑byte timestamp is a Unix timestamp in seconds (not milliseconds). The next 5 bytes identify the machine and process that generated the ID. Finally, a 3‑byte incremental counter differentiates multiple ObjectIDs created within the same second by the same process.

This structure is reminiscent of UUID v1 (timestamp + node) but in a more compact 12‑byte format instead of 16.

Portability — ObjectID's Weak Spot

ObjectID integrates perfectly with MongoDB: it's lightweight, sortable by creation date, and guaranteed unique within a collection. But as soon as you need to expose this ID in a REST API, store it in PostgreSQL, or use it as a global identifier across multiple services, problems arise.

  • No native type: PostgreSQL, MySQL, and other databases have no ObjectID type. You have to store it as VARCHAR(24) or binary, losing automatic validation.
  • Non‑standard format: JSON Schema validators, ORMs, and client libraries recognize format: uuid, not format: objectid.
  • Hexadecimal encoding: The 24‑character text representation is less dense than a UUID's 36 characters, but the lack of hyphens can surprise developers used to UUIDs.

In short, ObjectID is a great local identifier that becomes a burden as soon as you step outside MongoDB.

The Incremental Counter Trap

In very high concurrency scenarios (thousands of inserts per second on the same machine), the 3‑byte counter (16.7 million values) can overflow within a single second. If the timestamp hasn't advanced yet, two documents can receive the same ObjectID, causing duplicate key errors.

Documented issue: On MongoDB clusters with unsynchronized clocks or under extreme load spikes, the timestamp + counter combination is no longer sufficient to guarantee uniqueness. UUID v7, with 74 bits of randomness, eliminates this risk.

What We Changed Our Mind About — The Evolution of Our Recommendations

In 2022, we recommended ObjectID for every MongoDB project, and ULID for multi‑database projects. In 2024, with the publication of RFC 9562 and the availability of stable UUID v7 libraries in most languages, our recommendation evolved:

  • ObjectID remains acceptable for pure MongoDB projects that never expose their identifiers externally.
  • UUID v7 has become our default recommendation for any new project, including on MongoDB, because of its portability, standardization, and chronological sorting.

This section documents this change so you can understand the context of our current choices.

Practical Migration from ObjectID to UUID v7 in MongoDB

If you already have a collection with ObjectIDs, you can gradually introduce UUID v7 without service interruption. Here is a proven approach:

  1. Add a uuid_v7 field to your existing documents. You can generate a UUID v7 based on the ObjectID's timestamp to preserve approximate chronological order.
  2. Modify your application code to generate a UUID v7 for each new document at creation.
  3. Double‑index during the transition: keep the index on _id and add an index on uuid_v7.
  4. When all old documents are migrated or purged, replace _id with the value of the uuid_v7 field and remove the old index.

Frequently Asked Questions

Is ObjectID faster than UUID for MongoDB indexes?

ObjectID is slightly more compact (12 bytes vs. 16), but the index performance difference is negligible in practice. The real win is portability, not speed.

Can I decode an ObjectID without MongoDB?

Yes, the first 4 bytes give you the Unix timestamp. You can extract the creation date without querying the database.

Is it safe to change the _id type on an existing collection?

No, do not change the _id type in place. Instead, add a new field (like uuid_v7) and migrate gradually, as described above.
