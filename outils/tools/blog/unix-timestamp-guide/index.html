The production bug that introduced me to this

A few years ago I was working on a dashboard that pulled event data from a third-party API. The API returned timestamps as integers — created_at: 1695132000 — and my frontend team was rendering them as dates. For weeks, every single event in the interface was showing as "January 1, 1970 at 00:28".

The backend team had been storing Unix timestamps in seconds. The frontend was treating them as milliseconds and passing them directly to new Date(1695132000). In JavaScript, the Date constructor expects milliseconds. 1,695,132,000 milliseconds since the epoch is January 1970. 1,695,132,000 seconds is September 2023. Two interpretations of the same number, separated by 53 years.

The fix was one multiplication: new Date(1695132000 * 1000). The bug had been there for three weeks before anyone noticed, because no one had looked at an actual event date closely enough. This exact mistake is so common that most timestamp debugging questions on Stack Overflow trace back to it.

This guide is everything you need to never make that mistake, understand all timestamp formats you'll encounter in real systems, and extract dates from identifiers that embed timestamps — including UUID v1, UUID v7, and MongoDB ObjectIDs.

How to auto-detect seconds vs milliseconds

The most reliable heuristic is to count the digits. A Unix timestamp in seconds for any date between 2001 and 2286 is a 10-digit number. A timestamp in milliseconds for the same period is a 13-digit number. This works because the Unix epoch progresses predictably — in seconds, we won't hit 11 digits until November 20, 2286.

# Python — heuristic with boundaries
def detect_timestamp_unit(ts: int) -> str:
    """
    Returns the likely unit of a Unix timestamp.
    Works for any date between 2001 and 2286.
    """
    if 1_000_000_000 <= ts <= 9_999_999_999:
        return "seconds"        # 10 digits
    elif 1_000_000_000_000 <= ts <= 9_999_999_999_999:
        return "milliseconds"   # 13 digits
    elif 1_000_000_000_000_000 <= ts <= 9_999_999_999_999_999:
        return "microseconds"   # 16 digits
    elif ts >= 1_000_000_000_000_000_000:
        return "nanoseconds"    # 19 digits (Go, Rust, PostgreSQL pg_timestamp_ns)
    else:
        return "unknown"

# Usage
print(detect_timestamp_unit(1749376800))          # seconds
print(detect_timestamp_unit(1749376800000))        # milliseconds
print(detect_timestamp_unit(1749376800000000))     # microseconds

If you're building an API and want to avoid this ambiguity entirely, the correct solution is to use ISO 8601 strings for timestamps in your JSON payloads. "2025-06-08T10:00:00Z" is unambiguous, human-readable without conversion, and supported natively by every modern date library. The slight overhead in payload size is almost always worth the clarity.

Watch out for language defaults. JavaScript's Date.now(), Java's System.currentTimeMillis(), and Ruby's Time.now.to_i * 1000 return milliseconds. Python's time.time(), Unix system calls, PostgreSQL's EXTRACT(EPOCH FROM NOW()), and most database drivers return seconds. When you mix languages or services in a single system, document which unit you use and enforce it with a type alias or wrapper.

Timezones: the naive datetime trap

Timezone bugs are the most common class of date-related issues in backend systems, and they're almost invisible — they produce plausible-looking wrong values, not crashes. The root cause is almost always the same: a naive datetime being treated as if it were UTC.

In Python, there are two kinds of datetime objects. A naive datetime has no timezone information attached. An aware datetime has a timezone. The problem is that naive datetimes don't announce their ambiguity — they just exist, floating, and your code assumes they're UTC because that's the intention. But datetime.fromtimestamp() without a tz argument creates a naive datetime in the server's local timezone, not UTC.

import datetime, zoneinfo

ts = 1749376800  # 2025-06-08 10:00:00 UTC

# ❌ Bug: naive datetime interpreted in local timezone
# On a server set to Europe/Paris (UTC+2), this gives 12:00:00 local
# If you then serialize it as UTC, you get 12:00:00Z instead of 10:00:00Z
dt_naive = datetime.datetime.fromtimestamp(ts)
print(dt_naive)                    # 2025-06-08 12:00:00 (WRONG if you expect UTC)

# ✅ Correct: always specify timezone explicitly
dt_utc = datetime.datetime.fromtimestamp(ts, tz=datetime.timezone.utc)
print(dt_utc)                      # 2025-06-08 10:00:00+00:00 ✓

# ✅ Convert to a specific timezone after getting UTC
paris = zoneinfo.ZoneInfo("Europe/Paris")
dt_paris = dt_utc.astimezone(paris)
print(dt_paris)                    # 2025-06-08 12:00:00+02:00 ✓

# ✅ Back to Unix timestamp from an aware datetime
print(int(dt_utc.timestamp()))     # 1749376800 ✓

The same trap exists in JavaScript. new Date(timestamp) creates a UTC timestamp internally, but methods like getHours() return values in the local timezone of the browser or server. If you're running Node.js on a server in a non-UTC timezone, new Date(ts * 1000).getHours() does not give you the UTC hour.

const ts = 1749376800; // 2025-06-08 10:00:00 UTC

const d = new Date(ts * 1000);

// ❌ These depend on the system timezone — wrong if you expect UTC
console.log(d.getHours());               // Varies by system timezone

// ✅ These always return UTC values, regardless of system timezone
console.log(d.getUTCHours());            // 10 ✓
console.log(d.toISOString());            // "2025-06-08T10:00:00.000Z" ✓
console.log(d.toUTCString());            // "Sun, 08 Jun 2025 10:00:00 GMT" ✓

// ✅ Timezone-aware formatting with Intl (modern, recommended)
const formatter = new Intl.DateTimeFormat('en-US', {
  timeZone: 'Europe/Paris',
  dateStyle: 'full',
  timeStyle: 'long'
});
console.log(formatter.format(d));
// "Sunday, June 8, 2025 at 12:00:00 PM Central European Summer Time"

Daylight Saving Time and the ambiguous hour

Twice a year, clocks change. When clocks go back in autumn, one hour is "repeated" — the same wall clock time occurs twice. If you're storing events as local times without timezone info, you cannot distinguish between the two 2:30 AMs. This is why every serious system stores timestamps as UTC (or Unix timestamps) and converts to local time only for display.

Rule of thumb: Store as UTC, display as local. Always use IANA timezone names (Europe/Paris, America/New_York) rather than offsets (+02:00) for user-facing timezones — offsets don't account for DST changes.

Timestamps hidden inside identifiers

Several widely-used identifier formats embed a timestamp as part of their structure. This is useful: you can determine when a record was created directly from its ID, without a separate created_at column. But it also means that if you're seeing a UUID or ObjectID you don't recognize, you can decode when it was created.

UUID version 1

UUID v1 encodes a 60-bit timestamp counting 100-nanosecond intervals since October 15, 1582 — the date the Gregorian calendar was adopted. The timestamp is split across three non-contiguous fields in the UUID, so extracting it requires reassembling the pieces.

Given the UUID 6ba7b810-9dad-11d1-80b4-00c04fd430c8:

  • time_low = 6ba7b810 (first 8 hex chars, 32 bits)
  • time_mid = 9dad (chars 9–12, 16 bits)
  • time_hi = 11d1 with the version nibble (1) masked out → 01d1 (12 bits)
# Python — extract Unix timestamp from UUID v1
import uuid

def uuid1_to_unix_ms(uuid_str: str) -> int:
    """Extract Unix timestamp (ms) from a UUID v1 string."""
    u = uuid.UUID(uuid_str)
    if u.version != 1:
        raise ValueError(f"Expected UUID v1, got v{u.version}")

    # u.time is the 60-bit timestamp in 100ns intervals since 1582-10-15
    # Gregorian offset: 100ns intervals between 1582-10-15 and 1970-01-01
    GREGORIAN_OFFSET = 0x01b21dd213814000

    unix_100ns = u.time - GREGORIAN_OFFSET
    unix_ms = unix_100ns // 10_000  # Convert 100ns to ms

    return unix_ms

sample = "6ba7b810-9dad-11d1-80b4-00c04fd430c8"
ms = uuid1_to_unix_ms(sample)
from datetime import datetime, timezone
print(datetime.fromtimestamp(ms / 1000, tz=timezone.utc))
# 1997-02-03 17:26:46.009320+00:00

UUID version 7

UUID v7 (standardized in RFC 9562, May 2024) took a much simpler approach: the first 48 bits directly encode the Unix timestamp in milliseconds. No Gregorian offset, no field reassembly. Just read the first 12 hex characters.

# Python — extract Unix timestamp from UUID v7
import uuid

def uuid7_to_unix_ms(uuid_str: str) -> int:
    """Extract Unix timestamp (ms) from a UUID v7 string."""
    u = uuid.UUID(uuid_str)
    if u.version != 7:
        raise ValueError(f"Expected UUID v7, got v{u.version}")

    # The first 48 bits (12 hex chars) = Unix timestamp in milliseconds
    # uuid.int is the full 128-bit integer; shift right by 80 to get top 48 bits
    return u.int >> 80

sample = "018f4d1a-3b2c-7def-9abc-123456789abc"
ms = uuid7_to_unix_ms(sample)
from datetime import datetime, timezone
print(datetime.fromtimestamp(ms / 1000, tz=timezone.utc))

# JavaScript — even simpler
const uuid7 = "018f4d1a-3b2c-7def-9abc-123456789abc";
const hex48 = uuid7.replace(/-/g, '').slice(0, 12); // First 12 hex chars
const unixMs = parseInt(hex48, 16);
console.log(new Date(unixMs).toISOString());

MongoDB ObjectID

A MongoDB ObjectID is a 24-character hex string. The first 4 bytes (8 hex chars) encode the Unix creation timestamp in seconds. Unlike UUID v7, this is seconds — not milliseconds — so the resolution is 1 second.

# Python — extract creation timestamp from MongoDB ObjectID
def objectid_to_unix_s(object_id: str) -> int:
    """Extract Unix timestamp (seconds) from a MongoDB ObjectID."""
    if len(object_id) != 24:
        raise ValueError("ObjectID must be exactly 24 hex characters")
    # First 8 hex chars = first 4 bytes = Unix timestamp in seconds
    return int(object_id[:8], 16)

oid = "507f1f77bcf86cd799439011"
ts = objectid_to_unix_s(oid)
from datetime import datetime, timezone
print(datetime.fromtimestamp(ts, tz=timezone.utc))
# 2012-10-16 21:10:47+00:00

# JavaScript
const oid = "507f1f77bcf86cd799439011";
const unixSeconds = parseInt(oid.substring(0, 8), 16);
console.log(new Date(unixSeconds * 1000).toISOString());
# "2012-10-16T21:10:47.000Z"

The Year 2038 problem

On January 19, 2038 at 03:14:07 UTC, 32-bit signed integer Unix timestamps overflow. The maximum value of a 32-bit signed integer is 2,147,483,647. After that, it wraps to a large negative number — which represents December 13, 1901.

Unix timestamp timeline (32-bit signed integer)
1970 → 2025 (safe)
2038 overflow ⚠️
Max value: 2,147,483,647 = January 19, 2038 at 03:14:07 UTC

You can demonstrate the overflow yourself on any 32-bit system:

/* C — Year 2038 overflow demonstration */
#include <stdio.h>
#include <time.h>
#include <stdint.h>

int main() {
    // Max value of a 32-bit signed integer
    int32_t max32 = 2147483647;

    // Convert to struct tm
    time_t t = (time_t)max32;
    struct tm *tm_info = gmtime(&t);
    printf("Max 32-bit timestamp: %s", asctime(tm_info));
    // "Tue Jan 19 03:14:07 2038"

    // One second later: overflow
    int32_t overflow = max32 + 1;  // Wraps to -2147483648
    time_t t2 = (time_t)overflow;
    struct tm *tm_overflow = gmtime(&t2);
    printf("After overflow: %s", asctime(tm_overflow));
    // "Fri Dec 13 20:45:52 1901"

    return 0;
}

Who is still affected in 2026?

Most modern software runs on 64-bit systems and uses 64-bit time_t, which won't overflow until the year 292,277,026,596 — well beyond any practical concern. But the problem persists in specific contexts:

  • Embedded systems — microcontrollers, industrial control systems, automotive ECUs, and IoT devices often run 32-bit architectures with 32-bit timestamps baked into firmware.
  • Legacy databases — some older MySQL installations use 32-bit TIMESTAMP columns, which refuse dates after January 19, 2038. Use DATETIME instead.
  • 32-bit Linux — the Linux kernel added 64-bit time_t support for 32-bit architectures in version 5.6 (March 2020). Distributions based on older kernels still have the issue.
  • File systems — FAT32 stores modification times in a 32-bit field. Files won't be directly corrupted, but some timestamp operations may fail after 2038.

Frequently asked questions

How do I tell if a Unix timestamp is in seconds or milliseconds?

Count the digits. A 10-digit number is almost certainly Unix seconds — this covers any date between September 2001 (1,000,000,000) and November 2286 (9,999,999,999). A 13-digit number is milliseconds. The confusion exists because JavaScript's Date.now() and Java's System.currentTimeMillis() return milliseconds, while Python's time.time() and most Unix system calls return seconds. If you control the API, document the unit explicitly and consider using ISO 8601 strings instead — they're unambiguous.

What is the Unix epoch?

The Unix epoch is January 1, 1970 at 00:00:00 UTC — the reference point from which Unix timestamps are counted. It was chosen by the original UNIX developers because it was a recent, round date that predated the first commercial UNIX installations. There's no deep significance to the choice; it was a practical decision. The exact value was formalized in the POSIX standard (IEEE Std 1003.1).

Can I extract a date from a UUID v1?

Yes. UUID v1 (defined in RFC 4122) encodes a 60-bit timestamp in 100-nanosecond intervals since October 15, 1582. The timestamp is stored in three non-contiguous fields (time_low, time_mid, time_hi_and_version), so you have to reassemble them and subtract the Gregorian offset (122,192,928,000,000,000 in 100ns units) to get the Unix timestamp. The Python uuid module exposes this as uuid.UUID(s).time. The DevToolbox UUID v1 MAC Extractor does this for you and also identifies whether the MAC address is real or randomly generated.

What is Windows FILETIME and how do I convert it?

Windows FILETIME is a 64-bit integer counting 100-nanosecond intervals since January 1, 1601 00:00:00 UTC. It's used throughout the Windows API (FindFirstFile, GetFileTime), .NET DateTime, LDAP, and Active Directory. The constant 11,644,473,600 is the number of seconds between January 1, 1601 and January 1, 1970. Conversion from Unix seconds: FILETIME = (unixSeconds + 11644473600) × 10,000,000. In .NET: DateTime.FromFileTimeUtc(filetime) or DateTimeOffset.FromUnixTimeSeconds(unixSeconds).

What is the Year 2038 problem?

On 32-bit systems that store Unix timestamps as signed 32-bit integers, the maximum representable value is 2,147,483,647, corresponding to January 19, 2038 at 03:14:07 UTC. After that moment, the integer wraps to a large negative number (−2,147,483,648), which represents December 13, 1901. Modern 64-bit systems are not affected — their maximum timestamp is the year 292,277,026,596. The risk area is embedded systems, legacy databases, and 32-bit Linux builds.

Why do Python datetime objects cause timezone bugs?

Python has two kinds of datetime objects: naive (no timezone information) and aware (with timezone). datetime.fromtimestamp(ts) without a tz argument creates a naive datetime in the server's local timezone — not UTC. If your server is in UTC+2 and you serialize this naive datetime as UTC, you get a time that's two hours off. The fix is always explicit: datetime.fromtimestamp(ts, tz=datetime.timezone.utc). For timezone conversions, use zoneinfo.ZoneInfo (Python 3.9+) with IANA timezone names rather than fixed offsets, which don't account for DST changes.
