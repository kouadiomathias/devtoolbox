URL Encoding & Decoding: The Complete Developer Guide (2026)

June 8, 2026 · 14 min read · By Mathias Kouadio

In 2021, a single unencoded curly brace in a redirect_uri parameter enabled an open redirect attack on a widely-used OAuth service. The token was valid, the domain check passed — but the URL was never properly decoded before validation. One character, silently bypassing security. URL encoding is one of those topics that feels trivial until a production incident proves otherwise.

This guide covers everything you need to know: the exact difference between the three JavaScript encoding functions, the double-encoding bug that silently breaks APIs, + vs %20, code examples in five languages, and the security vulnerabilities that improper encoding enables. The three JavaScript encoding functions — exact differences

Most articles explain these vaguely. Here is the exact behavior with the same input string to make the comparison unambiguous:

const input = "hello world/path?key=val&other=a+b#anchor";

// encodeURI — preserves URL structure
encodeURI(input);
// "hello%20world/path?key=val&other=a+b#anchor"
// ✅ / ? & # = + preserved intentionally
// ⚠️  space → %20, but & and = are NOT encoded

// encodeURIComponent — encodes EVERYTHING except alphanumerics
encodeURIComponent(input);
// "hello%20world%2Fpath%3Fkey%3Dval%26other%3Da%2Bb%23anchor"
// ✅ / ? & # = + ALL encoded — correct for parameter values

// escape() — DO NOT USE, deprecated since ES3
escape(input);
// "hello%20world/path%3Fkey%3Dval%26other%3Da+b%23anchor"
// ❌ encodes differently, incompatible with modern standards

Never use escape() in modern code. It encodes Unicode characters using Latin-1 (%E9 for é) instead of UTF-8 (%C3%A9), breaking any non-ASCII content. It also fails completely on emoji and characters above U+00FF. It was deprecated in ES3 and exists today only for legacy script compatibility.

The double-encoding trap — the most common silent bug

This is the bug that generates the most confused Stack Overflow questions. You're getting %2520 in your URL when you expected %20. Here's exactly what happened and how to fix it.

// The scenario: you encode a string that was already encoded
const original = "hello world";
const encoded = encodeURIComponent(original);
// "hello%20world"

// Bug: encoding it a second time by mistake
const doubleEncoded = encodeURIComponent(encoded);
// "hello%2520world"
// The % character was itself encoded to %25
// So %20 became %2520

// When the server decodes it: it receives "hello%20world"
// not "hello world" as expected
decodeURIComponent("hello%2520world"); // "hello%20world" ← still encoded!

Detect and fix double-encoding programmatically

// Detect if a string is already encoded
function isAlreadyEncoded(str) {
  try {
    // If decoding changes the string, it was encoded
    return str !== decodeURIComponent(str);
  } catch {
    // Malformed URI — has invalid % sequences
    return false;
  }
}

// Encode idempotently — only encode if not already encoded
function safeEncode(str) {
  return isAlreadyEncoded(str) ? str : encodeURIComponent(str);
}

// Detect double-encoding in a URL
// %25XX pattern is a near-certain sign of double encoding
const doubleEncodingRegex = /%25[0-9A-Fa-f]{2}/gi;

function hasDoubleEncoding(url) {
  return doubleEncodingRegex.test(url);
}

console.log(hasDoubleEncoding("hello%2520world")); // true — double encoded
console.log(hasDoubleEncoding("hello%20world"));   // false — correct
Root cause: double-encoding usually happens at framework boundaries — a middleware encodes a value, then your code encodes it again before inserting it into a URL. The fix is to identify which layer is responsible for encoding and remove the duplicate.

The + vs %20 difference that breaks API integrations

Both represent a space in a URL — but they come from different standards and are not interchangeable in all contexts. Mixing them up causes silent failures in API calls.

// application/x-www-form-urlencoded standard (HTML forms)
// Spaces are encoded as +
const formData = new URLSearchParams({ q: "hello world", filter: "dev tools" });
console.log(formData.toString()); // "q=hello+world&filter=dev+tools"

// RFC 3986 standard (encodeURIComponent)
// Spaces are encoded as %20
encodeURIComponent("hello world"); // "hello%20world"

// The problem: mixing the two
const url1 = `https://api.example.com/search?q=${encodeURIComponent("hello world")}`;
// ✅ "https://api.example.com/search?q=hello%20world"
// Most APIs accept this — RFC 3986 standard

const url2 = `https://api.example.com/search?q=${"hello world".replace(/ /g, "+")}`;
// ⚠️ "https://api.example.com/search?q=hello+world"
// Some APIs require this, others don't decode + as a space

// Rule: use encodeURIComponent by default (%20)
// Only use + when the API specification explicitly requires it
// or when using URLSearchParams for form submission

The safe way to build URLs in JavaScript

// ❌ Never build URLs by string concatenation
const badUrl = "https://api.example.com/search?q=" + userInput;
// If userInput = "a&b" → breaks the query string
// If userInput = "<script>" → potential XSS

// ✅ Always use the URL class
const url = new URL("https://api.example.com/search");
url.searchParams.set("q", userInput);        // Encoded automatically
url.searchParams.set("filter", "price<100");  // < → %3C
url.searchParams.set("tags", "a&b");          // & → %26
console.log(url.toString());
// "https://api.example.com/search?q=hello%20world&filter=price%3C100&tags=a%26b"

URL encoding in 5 languages — copy-paste ready

Each language has its own encoding functions and its own traps. Here is the complete reference for the most common tasks in each ecosystem.

Python

from urllib.parse import (
    quote, unquote,
    quote_plus, unquote_plus,
    urlencode, urlparse, urlunparse, parse_qs
)

# Encode a parameter value (equivalent of encodeURIComponent)
quote("hello world/path", safe='')
# "hello%20world%2Fpath"
# safe='' means encode everything including /

# Encode for HTML forms (space → +)
quote_plus("hello world")
# "hello+world"

# Decode
unquote("hello%20world")       # "hello world"
unquote_plus("hello+world")    # "hello world"

# Encode a dictionary of parameters
params = {"q": "hello world", "page": 2, "filter": "price<100"}
urlencode(params)
# "q=hello+world&page=2&filter=price%3C100"

# Build a complete URL safely
from urllib.parse import urlencode, urlunparse

params = urlencode({"q": "hello world", "lang": "fr"})
url = urlunparse(("https", "api.example.com", "/search", "", params, ""))
# "https://api.example.com/search?q=hello+world&lang=fr"

# Parse an existing URL
parsed = urlparse("https://example.com/path?q=hello%20world#anchor")
print(parsed.scheme)    # "https"
print(parsed.netloc)    # "example.com"
print(parsed.path)      # "/path"
print(parsed.query)     # "q=hello%20world"
print(parsed.fragment)  # "anchor"

# Parse query parameters
parse_qs("q=hello+world&page=2")
# {'q': ['hello world'], 'page': ['2']}
Python trap: quote() has safe='/' by default, which means slashes are NOT encoded. Always pass safe='' when encoding parameter values. Otherwise quote("path/to/file") returns "path/to/file" unchanged.

JavaScript / Node.js

// Browser-native — no library needed
encodeURIComponent("hello world & more"); // "hello%20world%20%26%20more"

// URLSearchParams — the standard for query strings
const params = new URLSearchParams({
  q: "hello world",
  filter: "price<100",
  tags: "a&b"
});
console.log(params.toString());
// "q=hello+world&filter=price%3C100&tags=a%26b"
// Note: URLSearchParams uses + for spaces (form encoding)

// URL class — safe URL construction
const url = new URL("https://api.example.com/search");
url.searchParams.set("q", "hello world");
url.searchParams.set("category", "dev/tools");
console.log(url.toString());
// "https://api.example.com/search?q=hello+world&category=dev%2Ftools"

// Reading search parameters from a URL
const currentUrl = new URL("https://example.com/page?q=hello%20world&page=2");
console.log(currentUrl.searchParams.get("q"));    // "hello world" (auto-decoded)
console.log(currentUrl.searchParams.get("page")); // "2"

// Encoding a path segment (not a parameter)
const slug = "my article title/2026";

// "/blog/my%20article%20title%2F2026"

Java

import java.net.URLEncoder;
import java.net.URLDecoder;
import java.net.URI;
import java.net.http.HttpRequest;
import java.nio.charset.StandardCharsets;

// Encode a single parameter (always specify UTF-8)
String encoded = URLEncoder.encode("hello world/path", StandardCharsets.UTF_8);
// "hello+world%2Fpath"
// ⚠️ URLEncoder encodes space as + (form encoding)
// For %20, replace: encoded.replace("+", "%20")

// Decode
String decoded = URLDecoder.decode("hello+world%2Fpath", StandardCharsets.UTF_8);
// "hello world/path"

// Build a URI safely using the multi-argument constructor
// This handles encoding automatically
URI uri = new URI("https", "api.example.com", "/search", "q=hello world", null);
System.out.println(uri.toString());
// "https://api.example.com/search?q=hello%20world"

// Modern approach with HttpRequest.Builder (Java 11+)
HttpRequest request = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/search?q=" +
         URLEncoder.encode("hello world", StandardCharsets.UTF_8).replace("+", "%20")))
    .GET()
    .build();

Go

import (

    "net/url"
)

// QueryEscape — form encoding (space → +)
encoded := url.QueryEscape("hello world & more")
// "hello+world+%26+more"

URL encoding and security — vulnerabilities you need to know

Improper URL encoding is not just a formatting issue. It enables three categories of real attacks. Understanding them will change how you write validation code.

Open Redirect via incomplete encoding

// The attack: encoding only part of a malicious URL
// URL validation checks the domain — but on the encoded version

// Server-side validation (INCORRECT)
function isValidRedirect(url) {
  return url.startsWith("https://trusted.com");
}

// Attack URL 1: encoded colon
const attackUrl1 = "https%3A//evil.com/phishing";
isValidRedirect(attackUrl1); // false ← catches this

// Attack URL 2: the domain check passes, then the server decodes
const attackUrl2 = "https://trusted.com.evil.com/phishing";
isValidRedirect(attackUrl2); // true ← BYPASSED

// Attack URL 3: double-encoded slash
const attackUrl3 = "https://trusted.com%2F@evil.com";
// After decode: "https://trusted.com/@evil.com" → redirects to evil.com

// CORRECT approach: always decode THEN validate
function safeRedirect(encodedUrl) {
  let decoded;
  try {
    decoded = decodeURIComponent(encodedUrl);
  } catch {
    return null; // Malformed URL — reject
  }
  
  try {
    const parsed = new URL(decoded);
    const allowedDomains = ["trusted.com", "api.trusted.com"];
    
    if (!allowedDomains.includes(parsed.hostname)) {
      return null; // Domain not in allowlist
    }
    return decoded;
  } catch {
    return null; // Not a valid URL
  }
}

XSS via unencoded URL fragment

// Classic XSS bug: using the # fragment in HTML without encoding
// The fragment is never sent to the server — the browser handles it

// ❌ Vulnerable code
const fragment = location.hash.slice(1); // No server-side encoding
document.getElementById('title').innerHTML = fragment; // ← XSS

// Attack URL
// https://app.example.com/page#<img src=x onerror=alert(document.cookie)>

// ✅ Safe code
const fragment = decodeURIComponent(location.hash.slice(1));
document.getElementById('title').textContent = fragment;
//                                ^^^^^^^^^^^^^
//                         textContent instead of innerHTML
//                         — treats content as text, not HTML

Parameter injection via unvalidated query strings

// ❌ Building a backend URL with unvalidated user input
app.get('/proxy', (req, res) => {
  const target = req.query.url;
  // If url = "https://api.example.com/data?admin=true"
  // Attacker appends their own parameters

  const backendUrl = `https://internal-api.com/fetch?url=${target}`;
  // ← SSRF (Server-Side Request Forgery) risk
});

// ✅ Always validate the decoded value against an allowlist
app.get('/proxy', (req, res) => {
  const rawUrl = req.query.url;
  
  let decoded;
  try {
    decoded = decodeURIComponent(rawUrl);
  } catch {
    return res.status(400).json({ error: "Invalid URL encoding" });
  }
  
  const ALLOWED_ORIGINS = [
    "https://api.trusted.com",
    "https://data.trusted.com"
  ];
  
  const isAllowed = ALLOWED_ORIGINS.some(origin => decoded.startsWith(origin));
  if (!isAllowed) {
    return res.status(403).json({ error: "URL not allowed" });
  }
  
  // Safe to proceed
});

Security checklist — URL encoding audit

  • Decode BEFORE validating — never validate an encoded URL
  • Never inject unencoded URL parameters into HTML (XSS)
  • Use parameterized queries even with decoded values (SQLi)
  • Validate redirect URLs against an allowlist after full decoding (Open Redirect)
  • Check for %25XX in logs — likely sign of double-encoding
  • Use the native URL class — never concatenate URLs manually
  • Test all inputs with: & = + ? # / : @ % [ ]
  • Handle URIError from decodeURIComponent() — invalid sequences must not crash the app

Edge cases

Unicode, emoji, and accented characters

// Emoji → multi-byte UTF-8 sequences
encodeURIComponent("🔧");
// "%F0%9F%94%A7"  — 4 bytes, each → %XX

encodeURIComponent("🔧 fix it");
// "%F0%9F%94%A7%20fix%20it"

// Accented characters — important for FR/ES/DE markets
encodeURIComponent("déjà vu");
// "d%C3%A9j%C3%A0%20vu"
// é → %C3%A9 (UTF-8, correct)
// NOT %E9 (Latin-1, produced by the deprecated escape())

// Decode both UTF-8 and Latin-1 for legacy compatibility
function robustDecode(str) {
  try {
    return decodeURIComponent(str); // Handles UTF-8 %C3%A9
  } catch {
    try {
      // Fallback: unescape handles Latin-1 %E9 (deprecated but useful here)
      return unescape(str);
    } catch {
      return str; // Return unchanged if all decoding fails
    }
  }
}

Malformed percent sequences

// decodeURIComponent throws on invalid sequences
try {
  decodeURIComponent("hello%XYworld"); // %XY is not valid hex
} catch (e) {
  console.error(e.name);    // "URIError"
  console.error(e.message); // "URI malformed"
}

// Safe decode that never throws
function safeDecode(str) {
  try {
    return decodeURIComponent(str);
  } catch {
    // Return the original string if it can't be decoded
    // This is better than crashing the application
    return str;
  }
}

// Our online tool displays the error clearly without crashing
safeDecode("hello%XYworld"); // "hello%XYworld" (unchanged)

Already partially-encoded URLs

// Input mixed — partially encoded
const mixed = "hello%20world & café";

// Correct — decode the encoded parts first
decodeURIComponent("hello%20world & caf%C3%A9");
// "hello world & café" ✅

// Then re-encode cleanly if needed
encodeURIComponent("hello world & café");
// "hello%20world%20%26%20caf%C3%A9"

// Detect partial encoding before processing
function encodingStatus(str) {
  const hasPercent = /%[0-9A-Fa-f]{2}/.test(str);
  const hasUnencoded = /[&=+?#\[\] ]/.test(str);
  
  if (hasPercent && hasUnencoded) return 'partial';
  if (hasPercent) return 'encoded';
  if (hasUnencoded) return 'raw';
  return 'clean';
}

Frequently Asked Questions

What is the difference between encodeURI and encodeURIComponent?

encodeURI is designed to encode a full URL and preserves all characters that are structurally meaningful in a URL: /, ?, &, #, =, :, @. Use it when you have an already-formed URL and just want to ensure it's valid. encodeURIComponent encodes everything except alphanumeric characters and the safe set - _ . ! ~ * ' ( ). Use it for encoding individual parameter values before inserting them into a URL.

Why does my URL contain %2520 instead of %20?

This is a double-encoding bug. The % character itself was encoded to %25, turning %20 into %2520. It happens when a value that was already encoded (like hello%20world) is encoded again. The fix: find which layer is encoding the value twice and remove the duplicate. Use the safeEncode() pattern in this article to encode idempotently, or use the hasDoubleEncoding() detector to find the affected requests in your logs.

What is the difference between + and %20 in a URL?

Both represent a space but come from different standards. The + encoding comes from the application/x-www-form-urlencoded standard used in HTML forms — it's what URLSearchParams and most form processors produce. The %20 encoding follows RFC 3986, the general URL standard — it's what encodeURIComponent() produces. Most servers accept both, but some strict implementations only accept one. When in doubt, use %20 (encodeURIComponent). Only use + when an API explicitly specifies it or when submitting HTML forms.

Should I validate a URL before or after decoding?

Always decode first, then validate. Validating an encoded URL is a critical security mistake. An attacker can encode characters to bypass your validation rules, then the server decodes them into something malicious after the check passes. For example, https://trusted.com%2F@evil.com might pass a startsWith("https://trusted.com") check, but after decoding it becomes a redirect to evil.com. Always use new URL(decoded) to parse the URL, then check parsed.hostname against an allowlist.

How do I decode a URL in the terminal?

Python: python3 -c "import urllib.parse; print(urllib.parse.unquote('hello%20world'))". Node.js: node -e "console.log(decodeURIComponent('hello%20world'))". For encoding: python3 -c "import urllib.parse; print(urllib.parse.quote('hello world', safe=''))".

Is URL encoding case-sensitive?

The percent-encoding itself is case-insensitive: %2F and %2f are equivalent according to RFC 3986, and modern browsers normalize to uppercase. However, the decoded values (what the percent sequences represent) may be case-sensitive — it depends on the server's interpretation of the path or parameter. Stick to uppercase encoding (encodeURIComponent produces uppercase by default) to stay consistent.
