What a cron expression evaluator actually does

A cron expression is easy to write and hard to verify. You type */15 8-17 * * 1-5, it looks correct, and you deploy it. Three weeks later someone asks why the report job ran 47 times on a Sunday. The evaluator solves this by computing the actual next run dates from your expression — not an approximation, but the exact timestamps your cron daemon will fire.

Beyond listing dates, this tool does three things most online validators don't: it detects the OR trap when both day-of-month and day-of-week are non-wildcard, it catches impossible dates like February 31st that will silently never run, and it generates a natural-language description that reads like a sentence rather than a repeat of the technical syntax.

The OR trap the most common cron mistake

Standard cron documentation is ambiguous about what happens when both day-of-month and day-of-week contain non-wildcard values. The POSIX standard says: run when either field matches — not both. This surprises almost everyone.

# Intended: "first Monday of every month"
# Actual: "every Monday" AND "every 1st of the month"
0 0 1 * 1

The evaluator detects this pattern and displays an orange warning: "OR behavior: job runs when day-of-month OR day-of-week matches." To get true AND logic, add a check inside your script. In Quartz, use the ? operator to leave one day field unspecified.

Semantic warnings: impossible dates and invalid ranges

Syntax validation only checks that values are within the allowed ranges. Semantic validation goes further and asks: can this expression ever fire?

  • February 31 — 0 0 31 2 * is syntactically valid but February never has 31 days. The job will silently never run.
  • Day of week 8 — 0 0 * * 8 is out of range (0-7). Most cron daemons ignore the line entirely.
  • Step of 1 — */1 is equivalent to *; the evaluator suggests the simpler form.
  • Single-element range — 1-1 is equivalent to 1; the evaluator suggests the direct value.

Frequently asked questions

What formats does this cron evaluator support?
The evaluator supports standard Linux cron (5 fields), extended cron with a leading seconds field (6 fields), Quartz/Spring (6-7 fields with ?, L, W, # operators), and AWS CloudWatch Events format. Select the format in the dropdown or use Auto-detect — the tool infers the format from the operators and field count in your expression.
What is the OR trap in cron expressions?
When both day-of-month and day-of-week contain explicit non-wildcard values in standard cron, the job runs when either condition matches. 0 0 1 * 1 fires on the 1st of every month and every Monday — not just the first Monday. This is defined in the POSIX standard and affects all cron implementations. The evaluator shows an orange warning when it detects this pattern so you can decide whether the OR behavior is intentional.
How does timezone support work?
Select any IANA timezone from the dropdown (e.g. Europe/Paris, America/New_York). The next run dates are computed and displayed in that timezone, correctly handling DST transitions. If your cron daemon runs on a UTC server but you think in local time, this prevents the common mistake of scheduling a job at 2 AM local time and having it skip or run twice during DST transitions.
Can I export the next run dates?
Yes. Use the export buttons below the run list: CSV downloads a spreadsheet-compatible file with run number, date, day of week, and relative time. JSON produces an array of ISO 8601 timestamps for programmatic use. iCal (.ics) creates a calendar file you can import into Google Calendar or Outlook — each run appears as a separate calendar event.
Does the evaluator detect impossible dates?
Yes. 0 0 31 2 * (February 31) and 0 0 30 2 * (February 30) are detected as impossible — February never has those days, so the job will never run. 0 0 * * 8 (day of week 8) is flagged as out of range. Each warning explains why the expression is problematic and what to do instead.
Is any data sent to a server when I use this tool?
No. The entire evaluator — parsing, date generation, validation, export — runs in JavaScript in your browser. Your cron expressions are never transmitted to any server. This makes it safe to test expressions from sensitive production environments, crontabs containing internal hostnames or path structures, or any configuration you'd rather keep private.
