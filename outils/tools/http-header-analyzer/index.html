Why HTTP headers matter more than you think

HTTP headers are the first thing a browser reads when it receives a response — before a single byte of HTML is parsed. They instruct the browser on how to render content, what to cache, which origins to trust, and how to protect the user. A single missing header can expose users to clickjacking, cross-site scripting, or man-in-the-middle attacks. A misconfigured Cache-Control header can serve stale data for hours.

The problem is that headers are invisible in normal browsing. They live in the DevTools Network panel, in curl -I output, or in Nginx config files — not in the page itself. This tool makes them visible, explains each one in plain English, and tells you which ones you're missing before an attacker finds out first.

The security headers every production server should have

Strict-Transport-Security (HSTS)

Tells the browser to always use HTTPS for this domain, even if the user types http://. Without it, a network attacker can intercept the first HTTP request before the redirect to HTTPS. The recommended value is max-age=31536000; includeSubDomains (one year). Adding preload also registers the domain in browser preload lists, preventing any HTTP connection ever.

Content-Security-Policy (CSP)

Defines which sources of scripts, styles, images, and fonts are trusted. This is the primary defense against cross-site scripting (XSS). A strict CSP like default-src 'self'; script-src 'self' prevents injected scripts from loading. CSP is the most complex header to configure correctly — the tool highlights dangerously permissive values like unsafe-inline and unsafe-eval.

X-Frame-Options

Prevents your page from being loaded in an <iframe> on a different domain — the classic clickjacking defense. Use DENY to block framing entirely, or SAMEORIGIN to allow framing only from your own domain. Note: the modern equivalent is the CSP frame-ancestors directive, which is more flexible.

X-Content-Type-Options

Setting this to nosniff prevents browsers from MIME-sniffing the content type. Without it, a browser might execute a text file as JavaScript if a script tag points to it. One line, enormous protection. There's no reason not to set it.

Referrer-Policy

Controls how much information is included in the Referer header when navigating between pages. Without this header, the full URL (including query parameters, which may contain session tokens) is sent to third-party sites. strict-origin-when-cross-origin is the recommended default in 2026.

Permissions-Policy

The successor to Feature-Policy. Restricts access to browser APIs like camera, microphone, geolocation, and payment. If your application doesn't use the camera, setting camera=() means a compromised script can never access it either.

Headers that leak information — remove or obfuscate them

Server

Headers like Server: Apache/2.4.51 (Ubuntu) or Server: nginx/1.18.0 reveal your server software and version to anyone who looks. Attackers use this to look up known CVEs for that exact version. Set it to a generic value or remove it entirely: in Nginx, server_tokens off;.

X-Powered-By

Added by default by frameworks like Express.js and PHP. X-Powered-By: Express or X-Powered-By: PHP/8.1.0 is free intelligence for an attacker. In Express: app.disable('x-powered-by'). In PHP: expose_php = Off in php.ini.

Understanding Cache-Control directives

Cache-Control is one of the most important headers for performance, but its directives are often misunderstood:

  • max-age=N — cache the response for N seconds
  • no-store — never cache this response (for sensitive data)
  • no-cache — cache it but always revalidate with the server before serving
  • private — only the browser can cache this (not CDNs or proxies)
  • public — CDNs and proxies can cache this
  • stale-while-revalidate=N — serve stale content for N seconds while fetching a fresh version in the background
  • immutable — the resource will never change; skip revalidation entirely

How the security score is calculated

The score starts at 100 and points are deducted for each missing or misconfigured security header. The grading scale:

  • A+ (95–100): all critical security headers present and well-configured
  • A (85–94): most security headers present, no critical issues
  • B (70–84): some security headers present, minor gaps
  • C (50–69): significant security headers missing
  • D (30–49): most security headers absent
  • F (0–29): critical security configuration missing

Frequently Asked Questions

How do I get the HTTP headers from my website?

Three ways: (1) Browser DevTools → Network tab → click any request → Headers panel → copy the Response Headers section. (2) Terminal: curl -I https://yoursite.com. (3) Terminal with all headers: curl -D - https://yoursite.com -o /dev/null. Paste the output directly into this tool.

Is my data sent to a server when I use this tool?

No. All analysis happens inside your browser using JavaScript. Your headers — which may contain sensitive configuration details — never leave your machine. This is why we built it as a client-side tool instead of requiring a URL input that would need a server-side proxy.

My CSP has unsafe-inline. Is that always bad?

Almost always. unsafe-inline allows inline <script> tags and JavaScript event attributes, which is exactly what XSS attacks inject. If you need inline scripts, use a nonce-based CSP instead: script-src 'nonce-{random}'. For inline styles, consider moving them to a stylesheet. There are very few legitimate reasons to use unsafe-inline in a production application.

What is the difference between X-Frame-Options and CSP frame-ancestors?

X-Frame-Options is the older header (2008) with three options: DENY, SAMEORIGIN, and the deprecated ALLOW-FROM. The CSP frame-ancestors directive is the modern equivalent and is more flexible — it supports multiple allowed origins, wildcards, and 'none' (equivalent to DENY). If you use CSP with frame-ancestors, you can omit X-Frame-Options, but keeping both provides defense in depth for older browsers.

How do I add security headers in Nginx?

Add these to your server or location block in your Nginx config:

add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
server_tokens off;

Then reload: nginx -t && systemctl reload nginx
