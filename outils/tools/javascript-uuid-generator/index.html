UUID v4 vs v7 vs v1 vs ULID vs NanoID — choosing the right identifier

  
All five identifier types solve the same problem: generating a unique ID without coordination between servers. But they make different trade-offs between randomness, sortability, size, and database performance.

UUID v7 — why it matters for databases

  
UUID v4 is purely random. This means inserts into a B-tree index (which every primary key has) are random — each new row lands at an unpredictable location in the index, causing constant page splits and fragmentation. At 100M rows, this can make UUID v4 inserts 3× slower than sequential integer inserts.

  
UUID v7 embeds the current Unix timestamp in the first 48 bits. New rows always sort after existing ones (within the same millisecond, they sort randomly). This makes index inserts nearly sequential — much closer to BIGINT performance while keeping all the benefits of a distributed unique identifier.

  
// UUID v7 structure
018f4d1a-3b2c-7def-9abc-123456789abc
└──────────────┘ └──────────────────┘
    48-bit Unix        74 bits random
   timestamp (ms)     (version + variant bits)


  
crypto.randomUUID() vs uuid npm package

  
Use crypto.randomUUID() when you only need UUID v4 and target modern environments (all browsers, Node.js 19+). It requires zero dependencies and is the fastest option — implemented natively in C++ in V8.

  
Use the uuid npm package when you need other UUID versions (v1, v3, v5, v7), when you need to support older Node.js (v10-v18), or when you need deterministic UUIDs (v3/v5 use a namespace + name hash).

  
// crypto.randomUUID() — no install needed
const id = crypto.randomUUID();

// uuid npm — all versions
import { v4, v7 } from 'uuid'; // npm install uuid
const idV4 = v4();
const idV7 = v7(); // time-sortable, recommended for databases


  
ULID vs UUID — when to choose ULID

  
ULID (Universally Unique Lexicographically Sortable Identifier) is 26 characters in Crockford Base32. Unlike UUID, it sorts correctly as a plain string without any special parsing. This makes it useful when you want chronologically-ordered IDs that work in URL paths, file names, and string comparisons without needing a special UUID type in your database.

  
// ULID sorts chronologically as a string
const ids = ['01HZBQ7G4K...', '01HZBQ7G5M...', '01HZBQ7G7X...'];
ids.sort(); // ← already in creation order! UUID v4 sort is random.


  
Is Math.random() safe for UUID generation?

  
No. Math.random() is a pseudorandom number generator — it is deterministic and predictable if the seed is known. UUIDs generated with Math.random() can collide at much lower volumes than their 122-bit entropy would suggest. Always use crypto.getRandomValues() or crypto.randomUUID(), which use the operating system's CSPRNG (Cryptographically Secure Pseudorandom Number Generator).


  
Frequently Asked Questions

  

    

      What is the difference between UUID v4 and UUID v7 in JavaScript?
      

        
UUID v4 is purely random (122 bits of entropy) and has no time ordering. UUID v7 (RFC 9562, 2024) embeds the current Unix timestamp in milliseconds in the first 48 bits, making it time-sortable. For database primary keys, v7 is dramatically better because sequential inserts cause much less B-tree index fragmentation compared to v4's random inserts. The uuid npm package (v9+) supports both.

      

    

    

      Should I use crypto.randomUUID() or the uuid npm package?
      

        
Use crypto.randomUUID() for UUID v4 if you target Node.js 19+ or modern browsers — it requires no dependencies and is the fastest option. Use the uuid npm package if you need UUID v7 (recommended for databases), v1, v3, or v5, or if you need to support older Node.js versions (v10-v18).

      

    

    

      What is a ULID and when should I use it instead of UUID?
      

        
ULID is a 26-character Base32-encoded identifier. Its first 10 characters encode the millisecond timestamp and the remaining 16 are random. Unlike UUID, ULIDs sort lexicographically in creation order as plain strings — no parsing needed. Use ULID when you want URL-friendly IDs that sort chronologically and work in environments where the UUID type isn't supported.

      

    

    

      Is UUID generation safe to do client-side in the browser?
      

        
Yes. All modern browsers expose window.crypto.getRandomValues() which uses the OS's CSPRNG. crypto.randomUUID() uses this same source. Never use Math.random() for UUID generation — it is not cryptographically secure. The UUIDs generated by this tool have the same security properties as those generated server-side.
