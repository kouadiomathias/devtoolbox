What is a JWT (JSON Web Token)?

JWT is an open standard (RFC 7519) for compact, self-contained token-based authentication. A JWT consists of three Base64url-encoded parts separated by dots: the header (algorithm and type), the payload (claims like user ID, roles, expiration), and the signature (integrity proof). The payload is readable by anyone — it is encoded, not encrypted.

Security vulnerabilities this tool detects

Algorithm "none"

When a JWT has alg: none, it carries no signature at all. Some vulnerable servers accept tokens with this algorithm without verifying integrity — allowing any attacker to forge tokens with arbitrary claims (admin: true, user: any_id). This is one of the most critical JWT vulnerabilities. OWASP references it in JWT Security Best Practices.

RS256 / HS256 algorithm confusion

If a server normally validates tokens with RS256 (asymmetric — private key signs, public key verifies), an attacker may forge a token signed with HS256 using the server's public key as the HMAC secret. If the server's JWT library doesn't explicitly reject unexpected algorithm changes, it may verify this forged token using its own public key and accept it. This is a real-world attack documented in multiple CVEs.

Sensitive data in payload

JWT payloads are Base64url-encoded — anyone with the token can decode and read every claim. Storing passwords, credit card numbers, or other sensitive data in the payload is a critical security mistake. This tool scans for suspicious field names.

Standard claims (registered claims)

  • iss — Issuer: who issued this token
  • sub — Subject: identifier of the principal (usually user ID)
  • aud — Audience: who this token is intended for
  • exp — Expiration Time: Unix timestamp after which the token is invalid
  • nbf — Not Before: token not valid before this Unix timestamp
  • iat — Issued At: Unix timestamp of token creation
  • jti — JWT ID: unique identifier to prevent replay attacks

Frequently Asked Questions

What is the algorithm "none" vulnerability in JWT?

When a JWT uses alg: none, it carries no signature. Vulnerable servers accept tokens with this algorithm without verifying integrity — allowing attackers to forge tokens with arbitrary claims (making themselves admin, changing their user ID, etc.). Any JWT library should reject alg: none in production. This tool flags this as a Critical finding.

What is the RS256/HS256 algorithm confusion attack?

If a server uses RS256 but also accepts HS256, an attacker can sign a forged token with HS256, using the server's public key (which is often publicly available) as the HMAC secret. The server may then verify the forged token using its own public key and accept it. The fix: your JWT library must explicitly list allowed algorithms and reject tokens with unexpected algorithm changes.

Is my JWT sent to a server when I use this decoder?

No. All decoding, parsing, and analysis happens entirely in your browser via JavaScript. Your tokens never leave your machine. This is a key advantage over jwt.io, which sends the token to a remote server for signature verification. Never paste production JWT tokens into tools that require a server round-trip.

Can this tool verify the JWT signature?

No. Signature verification requires the secret key (for HMAC algorithms) or public key (for RSA/EC algorithms). You should never paste your secret key into any browser tool. This tool decodes and analyzes the header and payload, detects algorithm vulnerabilities, and displays the raw signature for inspection. For cryptographic signature verification, use your server-side JWT library.

Why should I never store passwords in a JWT payload?

JWT payloads are Base64url-encoded, not encrypted. Anyone who has the token — including the user's browser, any intermediate proxy, and any attacker who steals the token — can decode and read every claim in plain text. Store only non-sensitive identifiers and claims. If you need to transmit sensitive data in a token, use JWE (JSON Web Encryption) instead of JWS (JSON Web Signature).

What is the recommended token lifetime?

Access tokens should be short-lived: 15 minutes is a common recommendation. Refresh tokens can be longer (7–30 days) but must be stored securely (httpOnly cookie, not localStorage) and rotated on use. A stolen access token that expires in 15 minutes has a much smaller attack window than one valid for 24 hours.
