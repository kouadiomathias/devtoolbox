MySQL UUID Primary Keys: UUID_TO_BIN, the Swap Flag, and Why It Matters

On this page
  1. Two rows, wrong place
  2. Quick decision table
  3. Common MySQL UUID pitfalls
  4. FAQ
  5. References

A table using VARCHAR(36) DEFAULT (UUID()) PRIMARY KEY looked fine in every test — a handful of rows, instant inserts. Then two new rows landed in production, and instead of appearing at the end of the table, they showed up in the middle. InnoDB had to physically relocate existing rows to make room, because the new UUIDs sorted lower than rows already on disk. On a small table that's invisible. On a table with a billion rows, that's the clustered index rebalancing constantly, unable to keep up. Switching the column to BINARY(16) only fixed the storage footprint — the row still landed mid-table, because the raw bytes of a v1 UUID don't increase in a straight line. The part everyone misses is the swap_flag argument to UUID_TO_BIN(), which is what actually makes inserts sequential again.

This page generates a UUID in MySQL's own v1 layout, shows you the exact bytes UUID_TO_BIN() would store with and without that flag, and gives you SQL you can paste straight into a migration.

MySQL's built-in UUID() function only ever produces version 1 (timestamp + node) UUIDs — RFC 4122 style. There is no native v4 function; MySQL simply doesn't ship one.

Common MySQL UUID pitfalls

01Storing as BINARY(16) without the swap flag
High

A UUID v1's byte layout puts time_low — the fastest-changing part — first. Converting to binary without reordering still scatters new rows across the clustered index, because the leading bytes still don't increase monotonically over time.

Avoid
sql
CREATE TABLE users ( id BINARY(16) PRIMARY KEY DEFAULT (UUID_TO_BIN(UUID())), -- no swap_flag: still fragments name VARCHAR(100) NOT NULL );
Fix
sql
CREATE TABLE users ( id BINARY(16) PRIMARY KEY DEFAULT (UUID_TO_BIN(UUID(), 1)), -- swap_flag=1: time-high moves first, inserts stay sequential name VARCHAR(100) NOT NULL ); -- always read it back with the matching flag SELECT BIN_TO_UUID(id, 1) AS id, name FROM users;
02VARCHAR(36) instead of BINARY(16)
Medium

With the utf8mb4 charset, a VARCHAR(36) primary key index can carry a key length as high as 146 bytes (36 characters × 4 bytes/char, plus a length prefix). BINARY(16) is fixed at 16 bytes — that difference is duplicated across every secondary index on the table, since InnoDB stores the primary key in each one.

Fix
sql
-- prefer BINARY(16) over CHAR(36)/VARCHAR(36) whenever you control the schema id BINARY(16) PRIMARY KEY DEFAULT (UUID_TO_BIN(UUID(), 1))
03Needing UUID v4 as a MySQL default
Medium

MySQL only permits a small set of built-in, deterministic-enough functions in a DEFAULT expression. Custom stored functions and installed components — including any hand-written v4 generator — raise ER_DEFAULT_VAL_GENERATED_NAMED_FUNCTION_IS_NOT_ALLOWED when used as a column default.

Doesn't work
sql
-- fails: custom/component functions can't be a DEFAULT expression id BINARY(16) DEFAULT (UUID_TO_BIN(my_uuid_v4()))
Fix
sql
-- MySQL 8.0.30+: GIPK mode — let InnoDB manage a hidden auto_increment PK, -- and keep your app-generated v4 as a plain UNIQUE column SET sql_generate_invisible_primary_key = 1; CREATE TABLE users ( uuid BINARY(16) NOT NULL UNIQUE, -- v4, generated in application code name VARCHAR(100) NOT NULL ); -- INSERT INTO users (uuid, name) VALUES (UUID_TO_BIN(?), ?); -- ? = app-generated v4
04Treating UUID() as safe to expose publicly
High

A v1 UUID encodes the exact creation timestamp and a node identifier derived from the server's MAC address. Returning raw UUID() values in an API response leaks both to anyone who can parse a UUID.

Fix
sql
-- keep UUID() for internal, ordered primary keys only. -- for anything user-facing (API tokens, public IDs), generate a v4 -- in application code and store it separately if needed.

Frequently asked questions

Which UUID version does MySQL's UUID() produce?
Always version 1 — timestamp plus a MAC-derived node identifier, per RFC 4122. MySQL has no built-in function for version 4 (random) UUIDs.
What does the swap_flag argument to UUID_TO_BIN actually do?
It swaps the time-low and time-high hex groups of the UUID before converting to binary, so the most significant timestamp bits come first. That makes successive UUIDs compare as increasing values, which keeps InnoDB inserts at the end of the clustered index instead of scattering them.
Is a UUID primary key slower than AUTO_INCREMENT in MySQL?
Yes, if it's a plain v1 or v4 UUID stored without ordering — random insert positions cause page splits and rebalancing. With BINARY(16) and the swap flag, insert performance gets close to AUTO_INCREMENT, though the key is still 16 bytes versus 4 or 8 for an integer.
Can I use UUID() as a DEFAULT value in MySQL?
Yes — since MySQL 8.0.13, expression defaults like DEFAULT (UUID_TO_BIN(UUID(), 1)) are supported directly in CREATE TABLE, without needing a trigger. This works because UUID() and UUID_TO_BIN() are built-in functions; custom functions are not permitted the same way.
How do I get real UUID v4 values in MySQL?
Generate them in application code (every mainstream language has a secure v4 generator) and pass them in on insert. If you still want InnoDB to manage an efficient sequential primary key, enable GIPK mode and store the v4 as a plain UNIQUE BINARY(16) column instead of the primary key.
Is my UUID generated on a server somewhere?
No — everything on this page runs in your browser via JavaScript. The node bytes are randomized locally rather than reflecting any real hardware address.
