How to use UUID as a primary key in MySQL

MySQL does not have a native UUID data type, but you can store UUIDs in CHAR(36) or BINARY(16) columns. The UUID() function returns a 36‑character string (with hyphens).

1. Create a table with a UUID primary key (CHAR(36))

CREATE TABLE users (
    id CHAR(36) PRIMARY KEY DEFAULT (UUID()),
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

This uses MySQL 8.0.13+ expression default. For earlier versions, use a trigger.

2. Insert rows — the UUID is generated automatically

INSERT INTO users (name, email)
VALUES ('Alice', 'alice@example.com');
SELECT * FROM users;

3. Using a trigger for older MySQL versions

CREATE TRIGGER before_insert_users
BEFORE INSERT ON users
FOR EACH ROW
SET NEW.id = UUID();

4. Storing as BINARY(16) for performance

You can store UUIDs in a BINARY(16) column for a smaller footprint:

CREATE TABLE users_bin (
    id BINARY(16) PRIMARY KEY DEFAULT (UUID_TO_BIN(UUID())),
    name VARCHAR(100) NOT NULL
);

Why use UUID instead of AUTO_INCREMENT?

  • Distributed systems — UUIDs can be generated on multiple nodes without collisions.
  • Security — UUIDs don't expose the number of records.
  • Portability — UUIDs are easier to merge across databases.

Frequently Asked Questions

Which UUID version does UUID() produce?

MySQL's UUID() generates a UUID v1 (timestamp‑based) that may expose the MAC address. For random UUIDs (v4), use UUID_TO_BIN(UUID()) with a trigger or application code.

Is a UUID primary key slower than an auto‑increment in MySQL?

UUIDs are larger and may cause index fragmentation, especially if using random UUIDs. For InnoDB, consider using BINARY(16) and ordering by a timestamp column to reduce page splits.

Can I use UUID() as a default value in MySQL?

Yes, starting from MySQL 8.0.13, you can use DEFAULT (UUID()) for CHAR(36) columns. For earlier versions, a trigger is required.
