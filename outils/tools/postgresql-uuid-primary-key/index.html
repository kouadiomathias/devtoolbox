How to use UUID as a primary key in PostgreSQL

PostgreSQL supports the uuid data type natively. You can generate UUID values with the built‑in gen_random_uuid() function (available since PostgreSQL 13) or via the pgcrypto extension for older versions.

1. Enable the function (if needed)

For PostgreSQL 13+, gen_random_uuid() is available by default. For earlier versions, enable the pgcrypto extension:

CREATE EXTENSION IF NOT EXISTS pgcrypto;

After that, gen_random_uuid() will be available.

2. Create a table with a UUID primary key

CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

3. Insert rows — the UUID is generated automatically

INSERT INTO users (name, email)
VALUES ('Alice', 'alice@example.com')
RETURNING id;

Why use UUID instead of serial?

  • Distributed systems — each node can generate UUIDs independently without collisions.
  • Security — UUIDs don't reveal the number of records.
  • No sequential bottlenecks — unlike auto‑increment, UUIDs don't create a hot spot on the last page of the index.
  • Sorting — use UUID v7 if you need time‑ordered identifiers (see our UUID v7 generator).

Frequently Asked Questions

Which UUID version does gen_random_uuid() produce?

It generates UUID v4 (random). If you need time‑sortable UUIDs, use our UUID v7 generator and store the value manually.

Is a UUID primary key slower than a serial?

UUIDs are larger (16 bytes vs 4/8 bytes) and may lead to index fragmentation if you use random UUID v4. Using UUID v7 or a sequential UUID can mitigate this.

Can I use gen_random_uuid() in a default value?

Yes, as shown above. The function is stable and can be used directly in column default expressions.
