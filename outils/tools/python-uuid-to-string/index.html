Python UUID: str(), hex, int and Why Your Choice of Version Matters

Six months after launch, our users table had 40 million rows and every insert was taking twice as long as it did on day one. The table used uuid.uuid4() as its primary key — the "obviously correct" choice, since it meant no coordination needed between application servers. Nobody had asked what a fully random 128-bit value does to a B-tree index that expects roughly sequential inserts. The answer: constant page splits, terrible cache locality, and a primary key index that was larger than the table itself. We didn't fix it by abandoning UUIDs — we fixed it by understanding what each UUID version is actually for, and picking the right one for the job.

That's the gap this toolkit and this article are built to close. Python's uuid module gives you five ways to generate a UUID and at least seven ways to represent one as a string, and the standard library documentation doesn't tell you which combination to reach for. Below is a quick reference, followed by the failure modes that actually show up in production Python code.

Rule of thumb: v4 for general-purpose random IDs, v7 for anything that becomes a database primary key or needs rough chronological ordering, v5 for deterministic IDs derived from an existing name or external identifier, and v1 only when you specifically need embedded timestamp + node info and don't care about leaking them.

Common UUID pitfalls in Python

01UUIDv4 primary keys and index fragmentation
High

Every uuid.uuid4() value is 128 bits of random data. When it's the clustered/primary key, each insert lands at a random point in the B-tree instead of the end, forcing page splits and defeating the database's write cache. On tables with tens of millions of rows this measurably slows inserts and bloats the index.

Avoid
python
import uuid class User(Base): id = Column(Uuid, primary_key=True, default=uuid.uuid4) # random, fragments the index
Fix
python
# uuid.uuid7() ships in the stdlib from Python 3.14; use the `uuid7` PyPI # package on earlier versions. v7 embeds a timestamp so values are # roughly monotonic — inserts stay near the end of the index. class User(Base): id = Column(Uuid, primary_key=True, default=uuid.uuid7)
02UUIDv1 leaking MAC address and timestamp
High

UUIDv1 embeds the creation timestamp and, on many systems, a MAC-derived node identifier directly in the value. Anyone who can parse the UUID can extract exactly when it was created and infer information about the generating machine.

Avoid
python
session_id = str(uuid.uuid1()) # exposes creation time + node to anyone who decodes it
Fix
python
session_id = str(uuid.uuid4()) # or uuid7() if ordering is useful — neither leaks host info
03String comparison instead of UUID comparison
Medium

UUIDs can arrive as uppercase, lowercase, hyphenated, or unhyphenated hex. Comparing raw strings silently fails when casing or formatting differs, even though the underlying identifier is the same.

Avoid
python
if request_id == stored_id: # fails if one is uppercase and the other isn't ...
Fix
python
if uuid.UUID(request_id) == uuid.UUID(stored_id): # normalizes both before comparing ...
04Storing UUIDs as VARCHAR
Medium

A hyphenated UUID string is 36 bytes; the same value packed as raw bytes is 16. Using VARCHAR(36) instead of a native UUID column type more than doubles storage for that column and slows down index comparisons and joins.

Fix
python
# SQLAlchemy: use the native Uuid type, not String(36) from sqlalchemy import Uuid id = Column(Uuid, primary_key=True)

Frequently asked questions

What's the difference between uuid1, uuid4, and uuid7 in Python?
uuid1() encodes the current timestamp and a node identifier, so it's sortable but reveals when and (often) where it was created. uuid4() is fully random with no embedded information, at the cost of poor database index locality when used as a primary key. uuid7() combines a millisecond timestamp with random bits, giving you roughly sortable, index-friendly IDs without exposing a node identifier.
Is a Python UUID actually guaranteed to be unique?
Not mathematically guaranteed, but the collision probability for a random UUIDv4 is low enough to be treated as unique in practice: you'd need roughly a billion UUIDs generated per second for about 85 years before hitting a 50% chance of one collision. Deterministic versions (v3, v5) are only "unique" relative to their namespace and input name.
What does uuid.NAMESPACE_DNS actually do?
It's a fixed, standardized UUID that acts as a salt for uuid5(). Hashing the same name against the same namespace UUID always produces the same output UUID, which is what makes v5 deterministic. The standard library also ships NAMESPACE_URL, NAMESPACE_OID, and NAMESPACE_X500 for other identifier types.
How do I store a UUID efficiently in PostgreSQL or MySQL?
In PostgreSQL, use the native uuid column type, which stores it as 16 bytes and supports indexing directly. MySQL has no native UUID type before 8.0's UUID_TO_BIN()/BIN_TO_UUID() helpers, so store it as BINARY(16) and convert at the boundary rather than using CHAR(36).
Is uuid.uuid7() available in my Python version?
It's part of the standard library starting in Python 3.14. On earlier versions, the uuid7 or uuid6 packages on PyPI implement the same draft specification with an identical interface.
How do I convert a hex string without hyphens back into a UUID object?
uuid.UUID(hex=my_hex_string) accepts both hyphenated and non-hyphenated 32-character hex strings and normalizes them into a standard UUID object.
Is generating UUIDs safe across threads and multiprocessing?
Yes for uuid4 — it relies on the OS's cryptographically secure random source, which is thread- and process-safe. uuid1 is also safe but can theoretically repeat the same value if the system clock resolution is coarser than the generation rate on the same node; this is rarely a practical concern.
