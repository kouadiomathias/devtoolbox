Why URL encoding trips up developers

URL encoding (percent-encoding, defined in RFC 3986) converts characters that are not safe in a URL into a % followed by their two-digit hexadecimal UTF-8 byte value. A space becomes %20, an ampersand becomes %26, an equals sign becomes %3D.

The issue is not the encoding itself — it's doing it wrong or doing it twice. Double-encoding is the most common cause of broken API calls: encoding an already-encoded string produces %2520 (the % gets encoded as %25), which never decodes back to the intended value and causes 400 errors that take hours to diagnose.

Query string parsing pitfalls

A query string like ?name=John+Doe&role=admin&tags=css&tags=js has three subtleties most developers learn the hard way: spaces are encoded as + (form encoding) or %20 (RFC 3986), repeated keys (tags) represent an array, and special characters in values (&, =) must be encoded or they'll be misinterpreted as delimiters.

Frequently asked questions

What is the difference between encodeURIComponent and encodeURI in JavaScript?
encodeURIComponent encodes all characters except letters, digits, and the unreserved characters - _ . ! ~ * ' ( ). It is designed to encode the value of a single URL parameter. encodeURI preserves reserved characters that are part of the URL structure like : / ? # [ ] @ ! $ & ' ( ) * + , ; =. Use encodeURIComponent for parameter values and encodeURI for a complete URL you want to keep structurally intact.
What is double URL encoding and why is it a bug?
Double encoding happens when an already percent-encoded string gets encoded again. For example, encoding %20 (which represents a space) produces %2520, because % itself gets encoded as %25. The result is that when the URL is decoded once, the receiver gets %20 instead of a space — the intended value never arrives. Double encoding breaks API calls and form submissions in ways that are hard to spot visually.
Why do spaces sometimes appear as + instead of %20?
The + sign for spaces is a convention from the application/x-www-form-urlencoded format used by HTML forms. In standard URLs and REST APIs, %20 is the correct representation per RFC 3986. Both are widely accepted in query strings, but %20 is safer for APIs since some backends only decode one form. This tool's decoder handles both.
Is this tool secure for encoding sensitive data?
Yes. All encoding, decoding, and code generation happens entirely in JavaScript inside your browser. No text you enter is transmitted to any server. You can safely encode URLs containing API keys, tokens, or other sensitive parameters.
How do I encode a full URL without breaking its structure?
Use the Smart URL Encode tab. It parses your URL into components (scheme, host, path, query string), leaves the structural characters (://, /, ?, &, =) intact, and encodes only the values within each query parameter. This prevents double-encoding the separators while still properly encoding special characters in values.
