What is UUID v7 and why use it?

A UUID v7 (Universally Unique Identifier version 7) is a 128‑bit identifier designed to be time‑sortable. Unlike UUID v4 (random), it embeds a Unix timestamp in milliseconds in the first 48 bits. The remaining bits are filled with randomness, ensuring uniqueness even when many IDs are generated within the same millisecond.

This sorting property is critical for modern databases: an index on a UUID v7 column remains compact and fast, because new rows always land at the end of the index. With UUID v4, the random order causes fragmentation and slows down writes.

How UUID v7 works

The specification (RFC 9562) defines the following layout:

  • 48 bits : Unix timestamp in milliseconds (bits 0‑47).
  • 4 bits : version (fixed to 0111, i.e. 7).
  • 12 bits : random (bits 48‑59).
  • 2 bits : variant (10), indicating standard UUID format.
  • 62 bits : random (bits 62‑127).

Result: every UUID v7 contains the exact date and time of its creation, down to the millisecond, allowing you to sort identifiers chronologically while maintaining excellent uniqueness.

Frequently Asked Questions

What is the difference between UUID v4 and UUID v7?

UUID v4 is entirely random and cannot be sorted by time. UUID v7 starts with a timestamp, enabling natural chronological ordering and improving database index performance.

Is UUID v7 truly unique?

Yes. The 74‑bit random part (62 after the variant) provides enough combinations to avoid collisions, even under high‑volume generation. The collision probability is negligible.

Can I generate UUID v7 in Python?

Absolutely. The standard library uuid does not natively support UUID v7 (before Python 3.14), but you can use the uuid7 third‑party module or implement the logic yourself by combining time.time_ns() and random bytes.

What is the advantage over ULID?

ULID also offers time‑based sorting, but its representation is base32 (26 characters). UUID v7 remains a standard 36‑character UUID, fully compatible with all existing UUID libraries.
