    YAML, JSON, TOML — when to use which

    All three formats exist to solve the same problem: serializing structured data in a human-readable text format. But they were designed with different priorities, and choosing the wrong one causes real friction.

    FormatBest forNot great forUsed by
    JSONAPI payloads, data exchange, web storageLong human-edited config files (no comments)REST APIs, npm, package.json
    YAMLComplex configuration, multi-line strings, anchorsLarge data payloads (parsing is slow), beginner teamsKubernetes, GitHub Actions, Ansible, Docker Compose
    TOMLSimple config files, explicit typing, beginner-friendlyComplex nested structures, null valuesCargo.toml (Rust), pyproject.toml, Hugo, Go modules

    The YAML date problem — and how this tool handles it

    YAML's default schema automatically parses bare dates. A value like 2024-01-15 is parsed as a JavaScript Date object, not a string. When you then convert it to JSON, you get a UTC timestamp string ("2024-01-15T00:00:00.000Z") — not the string you put in. This trips up developers constantly when converting Kubernetes manifests or GitHub Actions workflows.

    Enable "Dates as strings" in the parse options to use JSON_SCHEMA instead of the default schema, which keeps date-like values as plain strings throughout the conversion.

    # YAML input
release_date: 2024-01-15
version: "2.1.0"

# ❌ Default behavior — JSON output
{ "release_date": "2024-01-15T00:00:00.000Z", "version": "2.1.0" }

# ✅ With "Dates as strings" — JSON output
{ "release_date": "2024-01-15", "version": "2.1.0" }

    YAML anchors and aliases — what they are and what happens to them

    YAML anchors let you define a value once and reference it multiple times, avoiding repetition. This is very common in CI/CD pipelines where you want to reuse environment definitions or step configurations.

    # YAML with anchors
defaults: &default-settings
  timeout: 30
  retries: 3
  log_level: info

service_a:
  <<: *default-settings
  port: 8080

service_b:
  <<: *default-settings
  port: 9090

    During conversion, anchors are resolved — the alias is replaced with the actual value. The "Resolve anchors" option makes this explicit in the YAML output (removes & and * references). JSON and TOML do not support anchors, so they always receive resolved values.

    TOML limitations — what cannot be converted

    TOML is less expressive than JSON and YAML in some areas. If your source data contains any of the following, conversion to TOML will fail:

    • null values — TOML has no concept of null. Convert to YAML instead.
    • Mixed-type arrays — TOML arrays must contain values of the same type. [1, "two", true] is invalid TOML.
    • Non-string keys — TOML keys must be strings (which JSON always uses, but some YAML documents may have integer keys).
    • Deeply nested anonymous arrays — TOML's array-of-tables syntax has limits on how structures can be nested.

    Semantic comparison — why string equality is not enough

    Two YAML files that represent the same configuration can look completely different as text: different key ordering, different indentation, different quote styles. A diff tool would show hundreds of changes. The semantic comparison in this tool normalises both documents — parses them, sorts all keys recursively, then compares the resulting structures. Two documents that are semantically identical produce the same normalised structure regardless of formatting.

    This is particularly useful when verifying that a configuration migration preserved all values, or confirming that two auto-generated files from different tools are equivalent.

    Frequently Asked Questions

    Why does YAML convert dates to unexpected values?

    By default, js-yaml parses bare dates like 2024-01-01 as JavaScript Date objects, which then serialize to UTC timestamp strings in JSON. Enable Dates as strings in the parse options. This uses JSON_SCHEMA instead of the full YAML schema, keeping date-like values as plain text throughout the conversion pipeline.

    What is a YAML anchor and alias?

    A YAML anchor (&name) marks a value so it can be reused elsewhere via an alias (*name). Anchors avoid repeating the same block across a config file. This converter resolves anchors during parsing — JSON and TOML outputs always contain the resolved values. In YAML output, enabling "Resolve anchors" replaces all aliases with their actual values.

    What is the difference between YAML and TOML?

    YAML is a flexible superset of JSON focused on human-readable data serialization, often used for complex configuration (Kubernetes, GitHub Actions, Ansible). TOML is simpler and more explicit, designed specifically for config files where clarity is more important than expressive power — it's used by Rust (Cargo.toml), Python (pyproject.toml), Hugo, and Go. TOML is less ambiguous than YAML but also less capable (no null, no mixed-type arrays).

    Can TOML represent all data that JSON can?

    No. TOML cannot represent null values — JSON null has no TOML equivalent. TOML also requires arrays to be homogeneous (all elements of the same type). If your JSON contains null or mixed-type arrays, conversion to TOML will fail. In those cases, convert to YAML instead, which supports null and mixed arrays natively.

    Is my data sent to a server?

    No. All parsing, conversion, validation, and comparison runs entirely in your browser using js-yaml and smol-toml. Your configuration files, secrets, and data never leave your machine. This is particularly important for files that may contain API keys, database credentials, or internal hostnames.

    Why does my YAML convert differently depending on the sort keys option?

    JSON objects and YAML mappings do not guarantee key order, so the order in the output depends on how the parser built the object. The "Sort keys" option in JSON output applies a recursive alphabetical sort to all object keys. This is useful for producing deterministic output for diffs and version control, but the sorted output is semantically identical to the unsorted one.
